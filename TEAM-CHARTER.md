# TEAM-CHARTER

Универсальный устав для проектов, где работает несколько параллельных Claude Code сессий (или людей) и нужна жёсткая дисциплина по git, контексту и документации.

Шаблон language-agnostic. Подставь свои значения вместо `<placeholder>` и удали разделы, которые не применимы.

---

## 1. Параллельная работа

### 1.1 Worktree-изоляция (обязательно)

Каждая параллельная сессия работает в собственном `git worktree` **внутри корня проекта**:

```
git worktree add .claude/worktrees/<task-name> -b <branch-name>
```

Worktrees снаружи проекта (`../<project>-<task>/`) запрещены: они выпадают из видимости, теряются при `git clean`, обходят `.gitignore`.

После мёрджа:
```
git worktree remove .claude/worktrees/<task-name>
```

**Почему:** две сессии в одной рабочей копии = `git checkout` одной стэшит изменения другой. Восстановление через `git stash pop` хрупкое.

### 1.2 Имена веток

Префикс по роли исполнителя: `<role>-<task-kebab>`, например `p1-text-rendering`, `backend-auth-refactor`. Префикс позволяет опознать владельца ветки если процесс крашнулся посреди задачи.

### 1.3 Резервирование задачи

Перед стартом — прочитать `STATUS-<role>.md` и `git branch`. Если в «In progress» уже есть задача — она занята.

Первым коммитом ветки записать в `STATUS-<role>.md`:
```
In progress: <task name>  branch: <branch-name>
Next step: <what to do first>  <file>:<line>
```

После мёрджа — очистить «In progress», перенести задачу в «Recent».

### 1.4 Никогда не оставлять worktree на `main` грязным

`main`-worktree существует только на момент атомарного мёрджа, после — `git worktree remove`. Грязный `main` блокирует все остальные сессии (`fatal: 'main' is already used by worktree`).

## 2. Распределение ролей

Делить проект **по доменам, а не по фичам** — снижает конфликты слияния.

| Роль | Зона ответственности | Пакеты/папки |
|---|---|---|
| `<role-1>` | <домен> | <модули> |
| `<role-2>` | <домен> | <модули> |
| … | | |
| **Bugfixer** | **Только баги.** Не пишет фичи. | весь репозиторий, read-only вне фиксов |

### Правило выделенного багфиксера

Остальные роли **не чинят баги при обнаружении**. Если нашёл баг во время работы над фичей:

1. Добавить строку в `BUGS.md` со статусом `OPEN` и следующим номером.
2. Опционально оставить `// BUG-NNN` на месте.
3. Продолжить текущую задачу — не переключать контекст.

Багфиксер берёт самые приоритетные `OPEN`-баги пачкой.

### Кросс-доменные задачи

- Owner публикует **типы/трейты со `<stub>`** в отдельном коммите.
- Consumer'ы пишут код против заглушки.
- Реальная реализация — drop-in замена.

## 3. Git-дисциплина

### 3.1 Запрещено

- Любой коммит напрямую в `main` (включая доки, «мелкие правки», координационные заметки).
- `git push --force` в `main`.
- Переписывание опубликованной истории.
- `git config` (никогда).
- Пропуск хуков (`--no-verify`, `--no-gpg-sign`).
- `git push` без явной просьбы пользователя.
- `git add -A` / `git add .` — только конкретные файлы по имени (защита от случайного `.env`).

### 3.2 Коммиты

- Один логический шаг = один коммит.
- Перед коммитом — минимум линтер/typecheck/build, желательно тесты.
- Сообщение: короткий subject (<80 символов) + пустая строка + body с **«почему»**, а не «что» (что и так в дифе).
- Trailer:
  ```
  Co-Authored-By: <Author> <email>
  ```

### 3.3 Мёрдж

```
git checkout main
git merge --no-ff <branch> -m "Merge <branch>: <reason>"
git branch -d <branch>
```

`--no-ff` обязателен — сохраняет «эта серия коммитов = одна задача» в `git log --graph`.

### 3.4 Защитные WIP-коммиты

Перед долгой паузой (отладка, прогон тестов, многофайловое редактирование) — `git commit -am "wip: <description>"` на своей ветке. Защищает от падений процесса и коллизий стэша. Перед мёрджем squash через `git rebase -i HEAD~N` — только пока ветку не подтянула другая сессия.

### 3.5 Если случайно оказался на чужой ветке

**Не** запускать `git restore .`. Сначала `git stash list` — там может оказаться чужая работа. Восстановить через `git stash pop`, потом вернуться на свою ветку.

## 4. Working boundary

**Писать код только внутри папки проекта.** Снаружи — `~/.bashrc`, `~/.config/*`, соседние проекты, ad-hoc папки — не трогать.

Если задача требует внешних изменений — описать, что нужно сделать пользователю, и ждать апрува.

Исключение: личная память агента в `~/.claude/projects/.../memory/` — живёт снаружи по дизайну.

## 5. Документация

### 5.1 Doc-комментарии обязательны

На **всех** публичных типах/функциях/полях — `///` (или эквивалент языка). Минимум: что значение представляет, какие единицы/координаты, что включает/исключает.

Параллельные сессии полагаются на это, чтобы понять семантику без чтения полной реализации.

### 5.2 SYMBOLS-индекс

Сгенерировать `SYMBOLS.md` — список всех публичных символов с `file:line` и первой строкой `///`-доки. Скрипт: `scripts/gen_symbols.<ext>`.

Использовать `grep SYMBOLS.md` вместо чтения исходников чтобы локализовать функцию. Регенерировать после переименования/перемещения публичных API — в том же коммите.

### 5.3 ADR на каждое решение

Каждое архитектурное решение — отдельный файл в `docs/decisions/NNNN-<title>.md` по шаблону `docs/decisions/TEMPLATE.md`. Индекс в `docs/decisions/README.md`.

Триггеры для ADR: новая зависимость, выбор API/протокола, отказ от очевидного решения в пользу неочевидного.

### 5.4 Что синхронно обновлять с кодом

В **том же коммите**, что и реализация:

- Roadmap / план — изменить статус ⬜ → 🟡 → ✅
- Описание подсистемы (`subsystems/<module>.md`)
- ADR при необходимости
- `SYMBOLS.md` если менялся публичный API

## 6. Контекст и токен-эффективность

### 6.1 Одна задача — одна сессия

Длинные сессии накапливают контекст — каждое сообщение дорожает. Использовать `/compact` если сессия разрослась, но задача ещё не завершена.

### 6.2 Не перечитывать после Edit/Write

Инструмент сообщает об ошибке, если что-то пошло не так. Корректность проверять через build/test, а не повторным `Read`.

### 6.3 Точные локации в задачах

Перед описанием бага или задачи — найти точное место. Включать `file:line` в формулировку:
```
<file>:<line> — <function>, <problem>
```

### 6.4 Dump-режимы вместо чтения исходников

5 строк dump-вывода часто заменяют чтение 3–4 файлов. Реализовать в инструменте отладочные команды (`--dump-<stage>`) и использовать их.

### 6.5 STATUS-<role>.md вместо большого плана

Полный план (`<project>-plan.md`) читать только если задача требует архитектурного контекста. Дневной статус — `STATUS-<role>.md` (~10 строк): «In progress», «Next», «Recent».

### 6.6 Grep вместо чтения

Целевой grep до открытия больших файлов:
```
grep "OPEN" BUGS.md
grep "<role>.*⬜\|<role>.*🟡" <project>-plan.md
```

### 6.7 Протокол старта сессии

Прочитать только: `STATUS-<role>.md` + `git branch`. Ничего больше.

## 7. Тесты и багтрекер

### 7.1 Tests-first для парсеров и алгоритмов

Сначала тест, потом код — для парсеров, рассчётов, алгоритмов с чёткой спецификацией.

### 7.2 Интеграционные тесты на реальных данных

Юнит-тесты на синтетике не заменяют тестов на реальных данных. Минимум один интеграционный тест на каждый компонент.

### 7.3 Единый трекер багов в репо

`BUGS.md` в корне, одна строка на баг:
```
BUG-018 | OPEN  | <description> | <file>:<line>
BUG-003 | FIXED 2026-05-10 | <description> | <file>:<line>
```

- Новый баг — следующий номер, статус `OPEN`.
- Исправлено — `OPEN` → `FIXED <date>`, **не удалять**.
- WONTFIX — остаётся в файле как есть.

### 7.4 Не подгонять тесты под движок

Тест-фикстуры — ground truth. Если тест падает, чинить движок, а не тест. Упрощение фикстуры ради прохождения теста = ложное прохождение.

Единственное основание править тест — баг в самом тесте.

## 8. Skills / автоматизация ритуалов

Повторяющиеся процедуры — в `.claude/skills/<name>/` как slash-команды:

| Skill | Назначение |
|---|---|
| `/<project>-task-start <name>` | Создать worktree, ветку, зарезервировать задачу |
| `/<project>-task-finish <name>` | Линт → тесты → merge --no-ff → worktree remove |
| `/<project>-add-<entity>` | Добавление сущности с обновлением всех связанных файлов |
| `/<project>-new-module` | Создание нового модуля/пакета |

Skill экономит токены и устраняет «забыл обновить файл X».

## 9. Коммуникация и стиль

### 9.1 Язык общения

Зафиксировать в `CLAUDE.md`: на каком языке отвечает агент, на каком пишутся коммиты.

### 9.2 Тон

- Технический, без эмодзи (если пользователь не использует их сам).
- Кратко и по делу. Никаких маркетинговых формулировок.
- Файлы — кликабельными ссылками: `[<file>](<path>)`.

### 9.3 Безопасность действий

Подтверждать перед выполнением:
- Деструктивные операции (`rm -rf`, `git reset --hard`, удаление веток).
- Hard-to-reverse (force-push, амэнд опубликованных коммитов, изменение CI/CD).
- Действия видимые другим (push, PR, сообщения в Slack/email).

Одобрение пользователем одного `git push` ≠ одобрение всех будущих push'ей.

## 10. Зависимости

### 10.1 Категории

- **Постоянные** — обоснованы навсегда (стандартные библиотеки экосистемы).
- **Временные** — пока нет своей реализации, с критерием замены.

### 10.2 Обоснование в коммите

Каждая новая зависимость требует в теле коммита:

> **Why this dependency:** <категория>, <якорь-интерфейс если временная>, <критерий замены>

## 11. Чек-лист для нового проекта

При копировании этого устава в новый репозиторий:

- [ ] Создать `CLAUDE.md` с проектным контекстом и ссылкой на этот устав
- [ ] Создать `STATUS-<role>.md` по числу ролей
- [ ] Создать пустой `BUGS.md` с заголовком и форматом
- [ ] Создать `docs/decisions/README.md` + `TEMPLATE.md`
- [ ] Создать `.claude/worktrees/.gitkeep` и добавить `.claude/worktrees/*/` в `.gitignore`
- [ ] Настроить скрипт `scripts/gen_symbols.<ext>` под язык проекта
- [ ] Добавить dump-режимы в основной CLI/бинарь
- [ ] Подставить `<role>`, `<project>`, `<lang>` плейсхолдеры в этом файле
- [ ] Удалить разделы, которые не применимы (например, §10 для проектов без внешних зависимостей)

## 12. Когда устав окупается

Этот устав имеет смысл, если выполнено **хотя бы одно**:

- ≥2 параллельных Claude Code сессий или живых разработчиков
- Проект живёт >3 месяцев и `git log` уже не помнишь наизусть
- Подсистем ≥3 с разными владельцами
- Контекст одной задачи не помещается в одну сессию агента

На соло-проекте из 5 файлов на 2 недели — это лишние церемонии. Не применять без причины.
