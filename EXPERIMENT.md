## Что уже сделано (7+ коммитов, см. git log)

1. **OpenGL (femtovg/glutin) удалён полностью** (−4961 строк). Единственный
   рендер — wgpu `Renderer` (WGSL), `default = ["backend-wgpu"]`.
2. **BUG-274 диагностирован до корня**: wgpu/DX12 на Intel Iris Plus берёт
   ~2.3 мс CPU за закрытие КАЖДОГО render pass (вся стоимость в `drop(pass)`),
   не зависит от площади кадра (проверено `LUMEN_WINDOW=512x378` vs `2048x1512`),
   плюс разовый скачок памяти +500 МБ. Это фикс. оверхед пасса, не clear.
3. **Skip-identical-frame** в `Renderer::render`: тотальный хэш кадра +
   `content_generation` (бампается register_image/снапшотами/шрифтами/resize).
   Идентичный кадр не рисуется вовсе. Выкл: `LUMEN_NO_FRAME_SKIP=1`.
   Проверка: 5 с движения мыши над статикой = 31 skip / 2 рендера.
4. **patch_scroll_layer**: скролл overflow-контейнера правит DL in-place
   (PushScrollLayer + DrawScrollbar thumbs) вместо полной пересборки
   paint_ordered. 12×: 1.54 → 0.13 мс/тик. Эквивалентность пересборке
   закреплена 5 тестами `patch_scroll_layer_*` + бенч (ignored).
5. **Безаллокационный HashFmt** в hash_commands / diff_display_lists / TileGrid.
6. **Пофазная диагностика кадра** (см. env-переменные ниже).
7. Слияние соседних Draw-батчей уровня 0 — испробовано и **откачено**:
   в реальных планах соседних level-0 батчей не бывает (между ними composite).
8. **Ярус 1, срез 1 — кэш шрифтовых метрик + мемоизация резолва** (2026-07-09):
   `FaceMetrics` (owned: units_per_em, ascent/descent, `OwnedCmap`, advances)
   строится один раз при загрузке face; codepoint-cascade и advance больше
   не парсят шрифты. `Font::parse` остался только на промахе глиф-атласа и
   для variation axes — лениво через per-frame memo (`LazyParsedFaces`).
   Плюс мемоизация `resolve_face_id` (хэш families+weight+style → face_id,
   сброс в `set_font_provider`) — раньше КАЖДЫЙ DrawText каждого кадра гонял
   `to_lowercase` + `FontProvider::pick_face` (2 Vec-аллокации + матчинг).
   Замер (1000000-final.html, dev-release): CPU к t=5s **3218.8 → 2359–2547 мс
   (−21%)**; холодный кадр faces 201.6 → 172.9 мс. **Вскрытая правда: посылка
   очереди «Font::parse 2–4 мс тёплый» была неверна** — тёплый парс был <1 мс;
   тёплый остаток фазы faces (~1.4 мс) — это skip-frame `hash_display_list` +
   image-pre-pass, а холодные 172 мс — `fs::read`+WOFF-декод lazy-загрузки
   face-ов внутри первого кадра (новые пункты очереди).
9. **Ярус 1, срез 2 — быстрый box-фильтр + параллельный image pre-pass**
   (2026-07-09, 2ff7183e). **Вскрытая правда №2 (faces-sub тайминг): посылка
   «холодные 172 мс фазы faces = fs::read+WOFF face-ов» была неверна** —
   шрифтовые prefetch/resolve ~0.02 мс (прогреты preload_curated_fallbacks);
   реальный тормоз — ПОСЛЕДОВАТЕЛЬНЫЕ CPU-ресайзы картинок в
   `ensure_image_gpu_key` (~219 мс), из них 147 мс — один `resize_area_avg`
   1024²→1000². Сделано: (1) `resize_area_avg` переписан — наивный 2D
   box-фильтр O(N·k²)+f64+roundf → сепарабельный двухпроходный O(N·k),
   предвычисленные веса по осям, векторизуемые zip-циклы, `+0.5` вместо
   `.round()`; изолированно 131.8 → 15.6 мс (8.4×), бенч `area_avg_bench`;
   (2) `prefetch_image_resizes_parallel` — resize/RGBA/ICC кадра в
   scoped-потоках, текстуры на UI-потоке; (3) `prefetch_faces_parallel` —
   та же схема для face-ов (защита для холодных @font-face, здесь no-op);
   (4) диагностика `faces-sub`/`imgjob` под `LUMEN_FRAME_LOG=2`.
   Замер: CPU к t=5s **2390.6 → 2078.1 мс (−13%)**; холодный imgpre
   155.5 → 24.2 мс (кадр 1), 63.4 → 32.5 мс (кадр 2). Тёплый остаток фазы
   faces теперь = только `hash_display_list` (1.2–2.5 мс/кадр).
10. **Ярус 0 — авто-проба бэкенда** (`crates/engine/paint/src/backend_probe.rs`):
   при старте для кандидатов Vulkan → GL → DX12 рисуется пробный кадр (clear
   в характерный цвет) в поверхность окна; два сигнала — texture readback
   (COPY_SRC + map) и **захват реальной презентации** через
   `PrintWindow(PW_CLIENTONLY|PW_RENDERFULLCONTENT)` — именно он ловит BUG-275
   (белое Vulkan-окно при «исправном» рендере). Кандидат принят, если
   презентация совпала с пробным цветом. Стоимость ~0.5 с на старте, окупается
   до t=5s. Выкл: `LUMEN_NO_BACKEND_PROBE=1`; `WGPU_BACKEND=...` главнее пробы.
   Замер (1000000-final.html, dev-release): idle-10s CPU 765.6 → **250 мс (3×)**,
   CPU к t=5s 4469 → 2219 мс, скачок памяти +185 МБ → нет (−28 МБ),
   private к t=15s 993 → 720 МБ. **Попутная находка: BUG-275 в этот вечер НЕ
   воспроизводится** — форс-Vulkan рисует корректно (транзиентный драйверный
   глюк); проба как раз и защищает от его возврата.
11. **Ярус 1, срез 3 — структурный хэш display-list'а** (2026-07-09):
    `hash_command_into` фолдит команду по сырым байтам полей (тег варианта —
    `mem::discriminant`), вместо прогона `{cmd:?}` через `HashFmt`. Стоимость
    была не в хэшировании, а в `Debug` для `f32`: алгоритм кратчайшего
    представления на каждый флоат, тысячи флоатов на кадр. Быстрые ветки —
    `FillRect`, `FillRoundedRect`, `DrawBorder`, `DrawOutline`, `DrawText`,
    `PushClipRect`, `PushOpacity`, `PushTransform`; остальные варианты падают в
    прежний Debug-fallback. Безопасность по построению: горячие ветки разбирают
    все поля без `..` (новое поле = ошибка компиляции), новый вариант попадает в
    fallback и хэшируется корректно с первого дня. Битовый фолд флоатов строже
    текстового — ошибка только в сторону лишней перерисовки, не устаревшего
    кадра. Тесты `structural_hash_is_stable_and_collision_free_on_corpus` и
    `structural_hash_refines_debug_hash` (второй закрепляет инвариант «всё, что
    различал Debug, различает и структурный» — то есть ложных skip нет). Тем же
    фолдом пошли `hash_commands` (кэш поддеревьев) и `hash_one_command`
    (`diff_display_lists`, `TileGrid`). Попутно в `hash_display_list` добавлены
    длины полос: раньше `content=[a,b], overlay=[]` и `content=[a], overlay=[b]`
    хэшировались одинаково.
    Замер: бенч `hash_display_list_bench` (1060 команд × 200) —
    **1.085 → 0.133 мс/кадр (8.1×)**; `faces-sub: hash` по кадрам
    3.49 → 0.71 / 1.43 → 0.63 / 1.30 → 0.40 мс.
    **Вскрытая правда №3: на idle-CPU это НЕ влияет.** `t=5s` 2093.8 → 1859.4 мс
    выглядело как −11%, но три прогона ОДНОГО И ТОГО ЖЕ после-патчевого бинарника
    дали 1859.4 / 1984.4 / 2140.6 мс — разброс 281 мс, и «до»-цифра лежит внутри
    него. Причина: за 15 с простоя рисуется **4 кадра**, ни одного
    `skip (identical frame)` — без ввода шелл вообще не зовёт `render`. Значит
    суммарная экономия за прогон = 6.40 → 1.83 мс, то есть 4.6 мс при CPU ~2000 мс
    (0.2%), и она на порядок меньше шума. Посылка «тёплый остаток faces =
    1.2–2.5 мс/кадр» верна ПРО КАДР, но кадров в простое почти нет: ярус-0 и
    skip-identical-frame уже свели их к четырём. Реальная цена хэша — на кадрах
    **от ввода** (по п.3: 5 с движения мыши = 31 skip / 2 рендера), где каждый
    skip платит за хэш перед решением: 1.3 → 0.4 мс на событие. Патч оставлен:
    строго быстрее, покрыт тестами, нулевой риск — но idle-CPU он не двигает.
    **Методический вывод: одиночный `measure_idle` не отличает сигнал от шума.**
    Порог значимости на этой машине — не меньше 300 мс по `t=5s`; всё, что ниже,
    требует 3+ прогонов на конфигурацию. Раннер `scripts/exp/run_hash_experiment.ps1`
    собирает полный цикл ДО/ПОСЛЕ в `.tmp/exp-hash/` (по файлу на шаг).

12. **Диагностика encode (`LUMEN_FRAME_LOG=3`), 2026-07-09 — вскрытая правда №4.**
    Добавлены: `alloc:` (создано текстур за кадр, время в `create_texture`+view+bg,
    hit/miss пула), гистограмма длительностей элементов плана, топ-12 самых
    дорогих с их `drop(pass)`. Три прогона, все совпадают.
    - **Аллокация ни при чём:** `created 14 tex in 0.04ms`. Пул render-target
      текстур (`pool hit 0 miss 1`) чинить бессмысленно — экономить нечего.
    - **Распределение плоское:** кадр 3 — `top 12 items = 44.8ms of 250.0ms`
      (18%), остальные 246 элементов по 0.2–5 мс. «Единиц дорогих пассов» нет.
    - **Фиксированной цены пасса не существует.** Кадр 2 и кадр 4 — один и тот
      же план (128 элементов, 869 ops, 2 слоя): encode **116.10 мс против
      3.88 мс**, 30×. Значит холодная цена — это разовый прогрев (похоже на
      ленивое создание `VkRenderPass`/framebuffer в wgpu-hal под каждую новую
      комбинацию attachment'а), гаснущий к четвёртому кадру. Заодно: бэкенд
      сейчас **Vulkan** (проба принимает его), а «~2.3 мс за закрытие каждого
      render pass» в BUG-274 мерили на DX12 — формулировку надо переписать.
    - **Следствие: пасс-схлопывание (RenderTaskGraph) НЕ окупится.** На тёплом
      кадре 128 пассов = 3.88 мс encode; схлопывание до 2–5 пассов выиграет
      максимум эти 3.88 мс. Строка очереди «это добивает остаток BUG-274 на
      любом бэкенде» — неверна.
    - **Следствие 2: `t=5s cpu ≈ 2000 мс` — это СТАРТ, а не простой.** Слагаемые:
      проба бэкенда 435 мс, холодный encode 116+250 = 366 мс, `collect`
      77.7+15.1 = 93 мс, adblock 107923 правил, imgpre ~22 мс, парсинг/layout.
      Простой = 4 кадра за 15 с, ни одного `skip (identical frame)`.
      Метрика «idle-CPU за 10 с» больше ничего не измеряет — она измеряет хвост
      старта. Для целей 100–1000× нужны другие метрики: время до первого кадра
      и время тёплого кадра под вводом (скролл/анимация).

13. **Стенд тёплого кадра + сравнение с Chrome, 2026-07-09 — вскрытая правда №5,
    самая важная.** Добавлен `crates/shell/src/bench_frames.rs`
    (`LUMEN_BENCH=hover|scroll:N:W`) и счётчики `FRAMES_RENDERED`/`FRAMES_SKIPPED`,
    которыми отчёт проверяет сам себя (первая версия стенда молча мерила скип-путь
    вместо перерисовки и показывала одинаковые медианы — счётчик это ловит).
    - **hover** (`rendered 0 skipped 600`): медиана 0.033–0.047 мс. Это цена
      `hash_display_list` на событие ввода. Тем самым п.11 наконец получил
      измеримый эффект: ~0.25 мс на кадр, ~15 мс CPU/с при движении мыши.
    - **scroll** (`rendered 600 skipped 0`): медиана **46.6 мс** (три прогона:
      46.54 / 46.71 / 46.65 — разброс 0.2%, не шум). Скролл = 21 fps.
    - **Пофазно:** `total 46.68 | faces 0.37 collect 0.64 prep 0.40 acquire 36.18
      encode 7.42 submit 1.67`. **`acquire` — 36 из 46 мс.** Present-режим
      подтверждён Immediate, значит это не ожидание vsync, а блокировка на
      `get_current_texture()`, пока GPU дорисовывает предыдущий кадр.
      **Кадр упирается в GPU: CPU ~10 мс, GPU ~46 мс.**
      Гипотеза «backdrop-кэш инвалидируется scroll_y» — ОПРОВЕРГНУТА: `bdrop 5x0.5ms`.
    - **Chrome на той же странице:** медиана **16.70 мс**, p95 17.0, max 17.2,
      кадров >20 мс — ноль, на вьюпорте 1357×881 (в 1.6× большей площади).
      Chrome упёрт в vsync, поэтому его истинная цена кадра ≤16.7 мс и точно не
      измеряется; нижняя граница нашего отставания при равной площади —
      **не менее 4.5×**. Страница НЕ патологическая: виноват движок.
    - **Причина структурная.** Chrome при скролле не перерастеризует страницу:
      контент лежит в растровых тайлах, скролл = сдвиг offset при композиции +
      дорисовка въехавшего края. Мы каждый кадр заново гоняем 80 draw-пассов и
      30 полноэкранных фильтр-пассов. Это ровно WebRender picture caching из
      «Карты заимствований» — единственный пункт очереди, ни разу не тронутый.
    - **Следствие: ярус 2 (рендер вон из UI-потока) для этой нагрузки бесполезен.**
      Время кадра = `max(CPU, GPU)`, а не сумма. Сейчас `max(10, 46) = 46`;
      после выноса в поток `max(10, 46) = 46`. Параллелизм окупится только после
      того, как GPU-часть подешевеет настолько, что CPU снова станет узким местом.
    - **Следствие: п.12 («пасс-схлопывание не окупится») сужается.** Тот вывод
      сделан из тёплого `encode` = 3.88 мс, то есть из **CPU-цены записи** пассов.
      GPU-цена их исполнения не измерена до сих пор. Полноэкранные offscreen-слои
      под каждый фильтр — это fillrate, и они вполне могут стоить тех самых 36 мс.
      Отменяется не приём, а обоснование его отмены.
    - **Следствие: цель «100–1000×» требует честной арифметики.** От 46.6 мс до
      бюджета vsync — 2.8×. До Chrome-подобного композиторного скролла (~1 мс) —
      порядка 40×. Тысячекратного ускорения на этой метрике не существует.

## Замеры бэкендов (idle-CPU за 10 с, graphic_tests/1000000-final.html, dev-release, Intel Iris Plus)

| Бэкенд | idle 10 с | Прогретый кадр | Картинка |
|---|---|---|---|
| femtovg/OpenGL (эталон main) | 375 мс | ед. мс | ок |
| wgpu **DX12** (текущий дефолт) | ~1500–2500 мс | 450–950 мс (wall) | ок |
| wgpu **Vulkan** | 203 мс | **7 мс** | **БЕЛОЕ ОКНО — BUG-275** |
| wgpu **GL** (`WGPU_BACKEND=gl`) | **156 мс — лучший** | 104–122 мс (wall, блокирующий GL) | ок |

Выбор бэкенда: **ярус-0 авто-проба** (Vulkan → GL → DX12, принимается первый,
чей пробный кадр реально виден в DWM-захвате). Статическая цепочка
DX12 → Vulkan → GL осталась резервом (`LUMEN_NO_BACKEND_PROBE=1` / все
кандидаты отклонены). Вопрос «GL дефолтом» снят: проба сама берёт Vulkan,
когда он презентует честно (сейчас — да), и GL, когда Vulkan белеет.

проблемам; у каждого приёма — конкретные имена типов для поиска в исходниках.

### Проблема «~270 render pass'ов на кадр» (корень BUG-274) → 2–5 пассов

- **WebRender**: display list → **RenderTaskGraph**; топологическая сортировка
  (`RenderTaskGraphBuilder`, `render_on: PassId`) сливает все независимые
  offscreen-задачи в ОДИН пасс — число пассов ≈ глубине вложенности эффектов
  (типично 2–5), не числу эффектов. Offscreen-задачи размером с **bbox**
  (клип по viewport, кламп blur ≤300px), пакуются `GuillotineAllocator`-атласом
  в общие таргеты ~2048×2048 из **пула** (`return_render_target_to_pool`).
  Плюс opaque pass front-to-back с z-буфером + alpha pass back-to-front;
  цель «≤100 draw calls на кадр».
- **GPUI (Zed)**: вообще без offscreen-слоёв на эффект — фиксированные
  примитивы (shadows → quads → paths → glyphs → sprites → images), каждый тип =
  один instanced draw call. Скругления/рамки — SDF в шейдере квада; box-shadow —
  замкнутая формула Эвана Уоллеса (erf-аппроксимация гаусса) прямо в шейдере,
  никаких blur-пассов; один общий MSAA-таргет на все пути кадра. Показательно:
  Windows-бэкенд Zed — **DX11, не DX12** («сложность не окупается») — наш
  wgpu-GL-фолбэк идейно то же самое.
- **WebRender, opacity**: простая `opacity` — «opacity collapse» в модуляцию
  alpha вершинных данных, слоя нет вовсе; маски — только сегменты с углами
  (`BrushSegment`) в общий alpha-атлас с кэшем между кадрами (`RenderTaskCache`);
  mix-blend читает backdrop из тайла кэша, а не копирует кадр.
- **re_renderer (rerun)**: фиксированные draw phases, фаза ≈ один пасс,
  drawable ≈ один draw call — число пассов константно и не зависит от сцены.

### Проблема «скролл/GIF перерисовывают весь кадр» (ярус 1)

- **WebRender picture caching**: контент растеризуется в крупные тайлы
  (~2048×512) в координатах **scroll root** (`TileCacheInstance`) — скролл =
  сдвиг offset'а при композиции, рисуются только въехавшие тайлы. Инвалидация:
  per-tile списки зависимостей (примитивы/клипы/image keys/opacity bindings) +
  интернинг примитивов → сравнение по id кадр-к-кадру → dirty rect + scissor
  внутри тайла. GIF в углу инвалидирует один тайл. Итог Mozilla: скролл
  16.4 Вт → 9.4 Вт.
- **smithay `OutputDamageTracker`** — каноничный damage-алгоритм: элементы с
  commit-счётчиками; damage = изменившиеся + появившиеся/сдвинувшиеся (старый
  И новый rect); **накопление damage по возрасту буфера** (при double/triple
  buffering перерисовывается union damage последних age-1 кадров); при
  фрагментации регион упрощается до extents.
- **alacritty**: построчный damage (`LineDamageBounds`), damage двух кадров,
  старый курсор/выделение добавляются в damage. **iced-урок**: их GPU-путь до
  сих пор без damage, но софтверный `iced_tiny_skia` диффит примитивы — диффинг
  display-list'а работает без реактивной системы свойств (наш
  `diff_display_lists` уже есть — его надо ПОДКЛЮЧИТЬ к рендеру).
- **Slint**: `PropertyTracker` → `DirtyRegion` (до 3 прямоугольников, старая +
  новая геометрия), история damage на 3 буфера.

### Проблема «Font::parse всех face каждый кадр» (ярус 1)

- **cosmic-text `FontSystem`** — эталон: fontdb парсит метаданные ОДИН раз при
  загрузке, данные в `Arc`; кэш `Font`-объектов по font_id (внутри распарсенный
  Face), кэши font-matching и codepoint→face (fallback). Поверх —
  `ShapeRunCache` (кэш шейпинга целых run'ов, Bevy получил 82→90–100 fps) и
  `SwashCache` (растр глифа по ключу face_id+glyph+size+subpixel).
- **GPUI/egui/glyphon**: глиф-атлас персистентный, alpha-only, тонирование
  цветом в шейдере; glyphon разделяет `prepare()` (CPU) и `render()` (один
  instanced draw в чужом пассе — middleware-паттерн wgpu).

### Проблема «idle ≠ 0» (BUG-271/274)

- **alacritty / niri / Slint / egui reactive mode** — общий принцип: нет
  таймера кадров вообще; redraw только от источников изменений (ввод, сеть,
  анимации, GIF-таймер); статичная страница = ноль wakeup'ов. Continuous-режим
  существует только как debug-флаг. Наш skip-identical-frame — полумера:
  кадр всё ещё СТРОИТСЯ и хэшируется; цель — не строить.

### Рендер вне UI-потока (ярус 2)

- **WebRender**: 3 потока — scene builder (тяжёлый CPU: picture/spatial/clip
  tree, интернинг) / render backend (culling, task graph, батчи) / renderer
  (единственный трогает GPU). Скролл и анимируемые свойства — property
  bindings в render backend, минуя пересборку сцены.
- **Bevy pipelined rendering**: фазы Extract → Prepare → Queue → Render; Extract —
  единственная точка синхронизации (копирование в render-мир), render-поток
  рисует кадр N, пока main считает N+1. **GPUI**: пул instance-буферов с triple
  buffering и асинхронным возвратом — нет CPU↔GPU stall'ов.

### Ярус 3 (Vello) — статус

Vello classic: клипы/слои/блэнды — команды PTCL внутри compute-конвейера
(вложенные эффекты не порождают пассов), но на конец 2025 — **alpha**:
worst-case GPU-память, conflation-артефакты, фильтры не доделаны. Зреет ветка
sparse strips: `vello_cpu` (быстрейший CPU-рендерер в Rust) и `vello_hybrid`
(CPU-геометрия + лёгкий GPU-финал, до WebGL2). Servo взял Vello для canvas.
Вердикт: **следить, не брать**; наш порядок ярусов подтверждается.

### Порядок внедрения (как шла Mozilla)

1. Граф задач + bbox-слои в атласе из пула → схлопывает 270 пассов;
2. тайловый кэш на scroll root → скролл почти бесплатен;
3. per-tile/damage инвалидация → GIF стоит свой прямоугольник;
4. вынос рендера в поток (extract-паттерн Bevy) — последним, когда кадр уже дёшев.

|---|---|

| `LUMEN_NO_FRAME_SKIP=1` | выключить skip-identical-frame |

| `LUMEN_PRESENT=mailbox\|immediate` | present mode (дефолт Fifo) |

## Грабли (уже наступали — не повторять)

- **«Белая страница + таб-бар» = страница НЕ ЗАГРУЗИЛАСЬ** (os error 2 при
  относительном пути из PowerShell), а не баг рендера. Проверяй stderr на
  «Ошибка загрузки» и заголовок окна.
- **Vulkan-окно белое целиком** (и таб-бар тоже) — это BUG-275, рендер при
  этом «исправен» по всем логам. Оба способа захвата показывают белое.
- graphic_tests/run.py калиброван под femtovg-пиксели — на этой ветке его
  пороги не показательны; визуальную проверку делать скриншотом + глазами
  (Read PNG) против DX12-эталона.
- `LUMEN_FRAME_LOG` читается один раз за процесс (OnceLock) — менять уровень
  между кадрами нельзя.
- Правки делать по пути worktree (`.claude/worktrees/exp-wgpu-only/...`),
  не по main-root — иначе уедут в чужое дерево.
- **`git stash push` без парного `pop` = замер baseline вместо патча.** Симптом:
  тесты проходят, но новых тестов НЕТ в списке (`running 9 tests` вместо 11), а
  `--ignored` бенч даёт `0 tests`. Проверять `git stash list` до паники.
  `run_hash_experiment.ps1` теперь отказывается стэшить, если под `crates/` и
  `subsystems/` нет изменений.
- **В PowerShell `git stash pop stash@{0}` не работает** — `@{0}` парсится как
  литерал хэш-таблицы, git получает мусор (`unknown switch 'e'`). Писать
  `git stash pop` (берёт верхний) или `git stash pop 'stash@{0}'`.
- **`$ErrorActionPreference = "Stop"` + cargo = скрипт падает на первой же
  строке прогресса.** Cargo пишет `Checking …` в stderr, `*>&1` превращает это в
  `ErrorRecord`. Успех определять по `$LASTEXITCODE`, префенс держать `Continue`.
- **`rust-lld: undefined symbol: anon.<hash>.llvm.<N>`** при линковке
  интеграционных тестов (`headless_tests`, `snapshot_tests`,
  `texture_pool_integration`) — протухший инкрементальный кэш, не ошибка в коде
  (клиппи проходит, ссылки идут из `std::sync::mpmc`, `core::slice::sort`).
  Лечится `cargo clean -p lumen-paint`, в упор — `CARGO_INCREMENTAL=0`. Юнит-тесты
  крейта гонять с `--lib`, чтобы вообще не линковать эти бинарники.
- **Один `measure_idle` ничего не доказывает.** Разброс `t=5s` между прогонами
  одного и того же бинарника — до 281 мс (замерено: 1859.4 / 1984.4 / 2140.6).
  Любой эффект меньше ~300 мс требует 3+ прогонов на конфигурацию, иначе шум
  выдаст себя за −11%.

### 2. Задача: offscreen-слои эффектов по bbox вместо full-frame

**Обоснование (п.14, замерено, не предположено).** Время кадра линейно по
площади: A1/A2 = 3.56× при 4× уменьшении площади. Стресс-страница стоит 46.6 мс
при контенте высотой 711 px — дороги 30 фильтр-пассов, каждый рисуется в
текстуру размером с окно. При bbox 200×150 против 1024×720 площадь падает в 24×.
Обычная страница (`samples/page.html`) уже стоит 0.274 мс — там ловить нечего.

**Где сидит проблема.** `renderer.rs:6346` —
`self.ensure_layer_textures(max_level, surface_w, surface_h)`: слои
аллоцируются по размеру поверхности, по одному на уровень вложенности
(`layer_textures[level - 1]`), и все эффекты этого уровня пишут в него
последовательно. Blur H/V пассы блитят полноэкранные quad-ы.

**Что уже есть.** bbox лежит прямо в display list:
`DisplayCommand::PushFilter { filters, bounds: Option<Rect> }` и
`PushBackdropFilter { filters, bounds: Rect }`. Их никто не читает при
конфигурировании пассов.

**Рекомендуемый путь — scissor, а не пересборка аллокации.** Не менять размер
текстур (аллокация стоит 0.04 мс на кадр, п.12 — она не при чём; зато смена
размеров ломает framebuffer-кэш wgpu-hal и вернёт холодные 116 мс, п.12).
Вместо этого ограничить **закрашиваемую площадь**:

1. `RenderPlanItem::Draw` с `target_level > 0` → `pass.set_scissor_rect(bbox)`.
   Осторожно: `LoadOp::Clear` чистит всю текстуру независимо от scissor. Либо
   принять эту цену (3 уровня × 0.74 Mpx — мелочь против 30 блюров), либо
   перейти на `LoadOp::Load` + скиссорный clear-quad.
2. `FilterComposite` / `BackdropFilterComposite`: quad блита и оба прохода
   гауссова размытия ограничить bbox — и по вершинам, и `set_scissor_rect`.
   **Здесь лежат искомые 24×.**
3. bbox для `PushOpacity` в display list отсутствует — считать объединение
   прямоугольников команд внутри слоя на фазе `collect` и класть в план.

**Проверка.** Обязателен замер ДО/ПОСЛЕ через `run_warm_frame_bench.ps1`:
A1 (`1000000-final.html` @ 1024×720) сейчас 46.6 мс медианы. Ожидание — единицы
миллисекунд. Если не уехало — гипотеза №7 мертва, писать сюда почему, и не
продолжать. Плюс: `run_gpu_bound_probe.ps1` не должен испортить прогон C
(обычная страница, 0.274 мс).

**Визуальная проверка обязательна.** Scissor легко обрезает тень/размытие,
которые по спецификации выходят за bbox элемента (`blur` расширяет границы на
~3σ). `PushFilter.bounds` может уже это учитывать — проверить; если нет,
расширять bbox на радиус размытия. `graphic_tests/run.py` на этой ветке
некалиброван (см. «Грабли») — сверять скриншотом глазами против текущего
Vulkan-эталона, особенно `30-css-filter.html`, `31-clip-path.html`,
`1000000-final.html`.

### 3. Что НЕ делать (закрыто замерами)

- **Пул render-target текстур** — аллокация 0.04 мс/кадр (п.12). Нечего экономить.
- **Ярус 2, рендер в отдельный поток** — время кадра = `max(CPU, GPU)` = 46 мс
  при CPU 10 мс (п.13). Ноль выигрыша, пока GPU не подешевеет.
- **Picture caching / тайловый кэш скролла как первый шаг** — экономит на
  перерисовке при скролле, а дорог сам кадр (п.14). Делать ПОСЛЕ bbox-слоёв.
- **Смена оконной библиотеки или графического API** — CPU простаивает 36 мс из
  46 в `acquire`. Менять нечего (п.13).
- **Оптимизации под метрику idle-CPU** — она меряет хвост старта (п.13).

### 4. Цель ветки — переформулировать

Шапка требует «100 и 1000 раз». От 46.6 мс до бюджета vsync — 2.8×; до
композиторного скролла уровня Chrome — порядка 40×. На метрике времени кадра
тысячекратного ускорения не существует. Честная цель: **держать 60 fps на
`1000000-final.html`**, то есть тёплый кадр < 16.7 мс. Обсудить с пользователем.

  на перенос в main» — завести при первом кандидате).
- Каждая вскрытая правда — новым пунктом в «Что уже сделано», даже (особенно)
  если она хоронит собственную работу. Их уже шесть; они и есть главный
  результат ветки.
