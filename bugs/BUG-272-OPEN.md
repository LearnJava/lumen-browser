# BUG-272 — ~1 ГБ RAM на lenta.ru при одной загруженной картинке (Edge целиком ~530 МБ)

**Статус:** OPEN
**Компонент:** shell / js / layout (потребитель не установлен — нужен heap-профиль)
**Найден:** 2026-07-07, сравнительный замер Lumen vs Edge на lenta.ru

## Симптом

https://lenta.ru в release-сборке: **1371 МБ** Working Set / **1124 МБ** private, стабильно (не растёт после загрузки). При этом:

- загружена всего **одна** картинка 280×186 (трекеры порезал easylist, остальные — lazy вне вьюпорта) → image-кэши тут ни при чём;
- baseline `samples/page.html`: 328 МБ WS / 138 МБ private → страница добавляет ~1 ГБ private;
- Edge (изолированный профиль) на полной lenta.ru с рекламой и всеми картинками: ~530 МБ WS суммарно на 8–9 процессов, ~290 МБ private.

Страница по логу: 1913 DOM-узлов, 933 CSS-правила, 1874 paint-команды, 121 preload-хинт, 7 webfonts. У пользователя в диспетчере наблюдалось и 2.4 ГБ (вероятно, другая страница/несколько навигаций — кэши чистятся только при навигации).

## Что уже известно (статический аудит 2026-07-07)

Для image-heavy сайтов найдены многокопийные хранилища пикселей (в этом баге НЕ главный потребитель, но чинить в рамках memory-diet):

- `femtovg_backend.rs:401` `raw_images: HashMap<String, Image>` — глубокая копия декодированных пикселей (`Image.data: Vec<u8>`, `image.clone()` в register_image:3469), без eviction до навигации;
- `femtovg_backend.rs:2344` — на каждый отображаемый размер создаётся GPU-текстура `"src@WxH"`, никогда не вытесняется; строка 2349 — временный полный clone при каждом ресайзе;
- шелл `image_cache.rs:81` — вторая CPU-копия в `Arc<Slot>`, без лимита; GIF декодируется целиком (все кадры RGBA);
- decode-кэш движка — LRU 256 МБ (сам по себе ок);
- `font_registry.rs:24` `bytes_store: HashMap<PathBuf, Vec<u8>>` — webfonts навсегда, без лимита, `get` возвращает `.cloned()`;
- `canvas2d.rs:32` — thread-local контексты canvas не освобождаются при удалении узла из DOM.

## Задача

1. Снять heap-профиль на lenta.ru (например, DHAT/heaptrack-аналог под Windows, или временная инструментализация аллокатора) и назвать точного потребителя ~1 ГБ.
2. Починить главного потребителя; многокопийные image-кэши — отдельными срезами по мере подтверждения профилем.

## Ожидаемое поведение

Страница класса lenta.ru (~2000 DOM-узлов, без тяжёлых картинок) — порядка 200–400 МБ, не 1.4 ГБ.

## Связанные

- BUG-271 (busy-loop CPU на той же странице) — возможный источник части аллокаций.
