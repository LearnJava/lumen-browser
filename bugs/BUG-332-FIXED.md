# BUG-332 — `scripts/scroll_perf.py` не парсит текущий формат `[frame] paint`

**Renumbered 2026-07-23** from BUG-327 — коллизия номера: BUG-327 уже занят
другим багом (`Node.prototype.hasChildNodes()`, см. `bugs/BUG-327-FIXED.md`),
влитым независимо той же датой (два разных `bugs/BUG-327-*.md` — `-OPEN` и
`-FIXED` — не конфликтуют путём, поэтому оба тихо смержились в `main`; та же
методология, что уже применялась к BUG-315/316, см. `CLAUDE.md` →
memory `project_git_pull_bug_renumber_collision`). Также пропущены 328 и 331 —
на момент переименования заняты незакоммиченным (P2, WPT) файлами.

**Статус:** FIXED 2026-07-23
**Компонент:** tooling (`scripts/scroll_perf.py`)
**Найден:** 2026-07-21, при верификации BUG-273 срез 3 (P1)

## Симптом

`python scripts/scroll_perf.py <page>` падает с `ValueError: min() iterable
argument is empty` на строке `print(f'display list: {min(cmds)}–{max(cmds)}
команд')` — паника происходит **после** печати числа кадров, но до печати
всей остальной статистики (avg/p50/max по `paint`/`content`/`flush`).
Воспроизводится и на `LUMEN_BACKEND=femtovg`, и на дефолтном (wgpu) бэкенде.

## Корень

`PAINT_RE` (`scripts/scroll_perf.py:44-47`) ожидает, что строка лога
заканчивается сразу после `swap X.XXms)`:

```
r'...flush\s+([\d.]+)ms, swap\s+([\d.]+)ms\)'
```

Но текущий формат `[frame] paint …` (femtovg-бэкенд, см.
`lumen_paint::frame_log_level`) несёт дополнительное поле `culled N/M leaf`
**перед** закрывающей скобкой:

```
[frame] paint  35.69ms  (content   4.24ms / 0 cmds, overlay 11.68ms / 13 cmds,
                flush  19.68ms, swap   0.08ms, culled 0/13 leaf) [thr main]
```

`re.search` не находит совпадение (регекс требует `)` сразу за `ms` от
`swap`), поэтому список `paints` остаётся пустым независимо от бэкенда —
`cmds`/`content`/`flush` статистика теряется целиком, а `min(cmds)` на
пустом списке роняет скрипт. `TOTAL_RE` (строка `[frame] total …`) не
пострадал — общая статистика по кадрам (`avg`/`p50`/`max`/эффективный FPS)
всё ещё доступна, но только руками (`grep '\[frame\] total' <лог> | ...`),
не через штатный вывод скрипта.

Отдельно замечено: `[frame] paint` вообще не эмитится на wgpu-бэкенде (эта
диагностика — femtovg-специфичная инструментация), только `[frame] total`
— на wgpu paint/content/flush-разбивка скрипту в принципе недоступна, даже
после починки регекса.

## Куда чинить

`PAINT_RE` должен либо не якориться на закрывающую скобку (обрезать по
`swap\s+([\d.]+)ms` без `\)`), либо явно матчить опциональный хвост
`(, culled \d+/\d+ leaf)?\)`. Второй момент (paint-уровень лога недоступен
на wgpu) стоит либо задокументировать как известное ограничение скрипта,
либо завести отдельно как feature-запрос на эквивалентную wgpu-инструментацию.

## Фикс

`PAINT_RE` (`scripts/scroll_perf.py:44-47`) получил некапчурящий опциональный
хвост `(?:, culled \d+/\d+ leaf)?` перед закрывающей `\)` — матчит и старый
формат (без `culled`), и текущий femtovg-формат с ним, не меняя индексы
существующих 7 капчур-групп (`paints.append(tuple(...))`/`p[N]`-индексация
ниже осталась рабочей без изменений). Докстринг файла (строки 19-21)
обновлён под текущий формат лога. Второй момент из «Куда чинить» (paint-уровень
недоступен на wgpu) остаётся задокументированным ограничением, отдельно не
заводился — не движковый баг, а отсутствие инструментации на этом бэкенде.
