# BUG-271 — постоянный busy-loop ~1.6 ядра CPU после загрузки JS-тяжёлой страницы

**Статус:** FIXED 2026-07-07 (P3)
**Компонент:** shell (event loop) + js (таймер-шим)
**Найден:** 2026-07-07, сравнительный замер Lumen vs Edge на lenta.ru

## Симптом

После полной загрузки https://lenta.ru процесс `lumen.exe` непрерывно потреблял ~160% одного ядра (≈40% CPU на 4-ядерной машине), без взаимодействия с окном, бесконечно. Burn включался через ~25–30 с после старта. Контроль `samples/page.html`: 0 CPU в простое. Edge на той же полной странице: ~0.03 ядра.

## Диагностика

Пер-источниковые счётчики redraw + пер-поточный CPU показали: **кадры не рисуются вообще** (0 RedrawRequested/с), но event loop крутится на **~22 800 итераций/с**, а горят потоки `main` (0.7 ядра) и `lumen-js` (0.94 ядра) — каждая итерация `about_to_wait` делает несколько блокирующих eval-round-trip'ов в JS-поток (`tick_timers`, `pump_websockets`, `pump_sse`, `flush_canvas_updates`).

## Корень

**Протухший `ControlFlow::WaitUntil`.** `about_to_wait` ставил `WaitUntil(deadline)` по `take_timer_wakeup()`, но никогда не сбрасывал его обратно. Когда последний JS-таймер отработал (`take_timer_wakeup()` → `None`), в winit оставался дедлайн в прошлом — эквивалент `Poll`: цикл просыпался немедленно каждую итерацию, навсегда. Отложенный старт burn'а = момент, когда догорела последняя таймер-цепочка страницы.

## Фикс (main.rs + dom.rs)

1. **Сброс в `Wait`:** `about_to_wait` теперь всегда выставляет control flow явно: `WaitUntil(min(timer_deadline, raf_deadline))` при наличии дедлайнов, иначе `ControlFlow::Wait`. lenta.ru: 22 800 → 2 пробуждения/с, 1.6 ядра → 0.002 ядра в простое.
2. **rAF отвязан от полной перерисовки:** убран безусловный `has_raf_pending → request_redraw` из `RedrawRequested` (гнал full display-list rebuild + GPU paint 60 fps для любого «пустого» rAF-цикла — 0.95 ядра на минимальной странице). Вместо него rAF-насос в `about_to_wait`: батчи исполняются по `WaitUntil`-таймеру (общий vsync-гейт `last_raf_batch_ms`), а перерисовка запрашивается только если колбэк изменил DOM (`take_dom_dirty` → `relayout()` + `request_redraw`). Чистый rAF-цикл: 0.95 → 0.04 ядра; rAF-анимации сохраняют 60 fps (проверено визуально).
3. **Спековый клампинг таймеров (HTML LS §8.6):** таймер-шим ведёт «timer nesting level»; вложенность >5 клампит timeout/interval к ≥4 мс. Раньше `setTimeout(fn,0)`-цепочки и `setInterval(fn,0)` будили цикл с максимальной частотой.
