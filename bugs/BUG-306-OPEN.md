# BUG-306: github.com раздувает процесс до ~2.9 ГБ одной вкладкой

**Статус:** OPEN
**Дата:** 2026-07-17
**Компонент:** js/dom (подозрение) — атрибуция не сделана
**Найден:** живой перф-аудит `/lumen-perf-audit` (docs/perf/runs/2026-07-17-live.json)

## Симптом

В живом окне (вкладка на сайт, кумулятивная RAM процесса, WinAPI WorkingSet):

```
lenta   ram=610 МБ   (6 вкладок открыто)
github  ram=2904 МБ  (+~2.3 ГБ на одну вкладку github.com; ready=45с)
```

Воспроизводится стабильно (в прогоне без --maximized: 613 → 2101 МБ, +1.5 ГБ;
следом stackoverflow добавил ещё +1.26 ГБ до 3.36 ГБ). После такой вкладки
сессия вскоре умирает — см. [BUG-307](BUG-307-OPEN.md).

## Контекст

- Известные memory-срезы BUG-272 касались lenta.ru (кэши изображений/строк);
  здесь другой профиль: гигабайты появляются на JS-тяжёлом сайте — вероятно
  V8-хип / удержание DOM/строк шимом, а не кэш картинок.
- `module error: Automatic publicPath is not supported` — бандл github
  частично падает (см. BUG-303), но до этого успевает съесть гигабайты.

## Диагностика

`LUMEN_MEM_REPORT=1` на живом окне с одной вкладкой github.com; разбивка
Rust-heap / V8 / GPU (см. reference-инструментарий BUG-272). Логи прогона:
`.tmp/perf-audit/20260717-*/live.stderr.0.log`.

## Повторный прогон 2026-07-20

Воспроизводится идентично после S12b-18 (V8-миграция продолжается) и после
BUG-297/316/308 фиксов: github.com HUNG (`Wait error: automation command
timed out`) в обоих повторах живого аудита (`docs/perf/runs/2026-07-20-live.json`,
коммит движка `a014c9e6`). RAM не удалось измерить в этот раз — вкладка
зависает раньше, чем скрипт успевает опросить WorkingSet, до самого зависания
успевает вырасти сравнимо (наблюдалось визуально в gdigrab-снимках соседнего
эксперимента). Статус не меняется, не регрессия.
