# BUG-286 — скролл: `content-visibility:auto` relevance-триггер стопорил UI-поток на ~1с

**Статус:** OPEN (частичная митигация влита — см. «Фикс» ниже; корневая стоимость каскада не устранена)
**Компонент:** shell (`crates/shell/src/main.rs`, `maybe_expand_cv_relevant`)
**Найден:** 2026-07-15, ручной тест колесом мыши на https://ria.ru (в ходе расследования BUG-284)

## Симптом

Первый скролл колесом на тяжёлой реальной странице «подвисает» примерно на
секунду, дальнейший скролл идёт гладко. `LUMEN_FRAME_LOG=1` на ria.ru поймал
это прямо посреди прокрутки:

```
[engine] relayout 1041.99ms
```

## Корень

`maybe_expand_cv_relevant` (`crates/shell/src/main.rs`, вызывается на каждом
кадре `RedrawRequested`, Step 1.6 «Update the rendering») — единственный из
scroll/relayout-триггеров, звавший **напрямую** синхронный `self.relayout()`
вместо `self.relayout_raf_dirty()`, которым уже пользуются rAF DOM-dirty flush
и zoom-дебаунс. Как только скроллом в расширенный viewport (`viewport +
CV_SLACK_FACTOR`) попадает пропущенный `content-visibility:auto`-узел, эта
функция раскрывает его (`cv_relevant.extend`) и синхронно гонит **полный**
relayout — пересчёт CSS-каскада (`compute_style`) для всех styled-узлов
документа на UI-потоке. На ria.ru (3093 правила, 1141 styled-узел, до фикса
BUG-284 — 2.4с/relayout, после — ~1.0–1.1с) это блокирует событийный цикл
ровно на то время, пока идёт `compute_style`.

Совпадает с «первым» скроллом чаще всего потому, что до первого скролл-тика
весь под-fold контент под `content-visibility:auto` остаётся «skipped» —
раскрывается он только когда впервые попадает в viewport+запас.

## Фикс (влит)

`maybe_expand_cv_relevant`: `self.relayout()` → `self.relayout_raf_dirty()`.
Под `LUMEN_ENGINE_THREAD=1` (не дефолт) это уводит relayout на движковый
поток (fire-and-forget, latest-wins — контент выкладывается на один из
следующих кадров вместо блокировки текущего). На дефолтной сборке (флаг
выключен) `relayout_raf_dirty()` падает обратно на тот же
incremental-затем-full путь — **поведение и стоимость на UI-потоке не
меняются**, зафиксирован только пропущенный кейс маршрутизации (согласованность
с остальными scroll/relayout-триггерами).

## Остаток (не устранён этой задачей)

Настоящий источник ~1с — сам каскад (`compute_style` на ~1141 узел), не
маршрутизация. `try_relayout_raf_incremental()` тоже не спасает много — он
всё равно пересчитывает полный каскад, пропускает только геометрию
неизменившихся поддеревьев. Дальнейшее ускорение требует либо ускорения
самого `compute_style`/`matches_complex` (см. `docs/plan/security-performance.md`
§14.2 п.1–4 — BTreeSet→Vec dedup уже проверен и опровергнут, реального
источника не нашли ad-hoc-профилированием), либо перевода `LUMEN_ENGINE_THREAD`
в дефолт (отдельное архитектурное решение, не в скоупе этой задачи), либо
подключения `tracy` к `lumen-layout` для сэмплирующего профилирования (§14.3,
п.7 бэклога — не проверено, подключён ли он уже).

## Воспроизведение

Нужна реальная тяжёлая страница с `content-visibility:auto` под-fold
контентом и большим числом CSS-правил (например, `@media`-тяжёлый сайт вроде
ria.ru/lenta.ru — синтетические `samples/`/`graphic_tests/` страницы не
используют `content-visibility`, репродукции на них не будет):

```
cargo run --release -p lumen-shell -- https://ria.ru
# один тик колеса вниз после первого непустого кадра
LUMEN_FRAME_LOG=1 cargo run --release -p lumen-shell -- https://ria.ru
```
