# BUG-243

**Статус:** OPEN
**Компонент:** js (JS DOM API)
**Симптом:** страницы, строящие SVG на клиенте через `document.createElementNS`, рендерятся пустыми (виден только HTML/CSS-каркас, само SVG-дерево не появляется).
**Репро-страница:** `docs/roadmap-svg-cleaves.html` (и `docs/roadmap-*.html` вообще), любой data-viz/иконки/графики на динамическом SVG.

---

## Симптом

Запуск:
```
target/debug/lumen.exe docs/roadmap-svg-cleaves.html
```
лог:
```
WebCodecs API init failed: Exception generated by QuickJS
SVG DOM API init failed: Exception generated by QuickJS
script error: JS runtime error: not a function
```
HTML/CSS-обвязка (шапка, навигация, поиск, чекбоксы, селект, футер, текст-статистика
из JS) рисуется **пиксель-в-пиксель как в Edge**. Пустует ровно `<main><svg id="g">` —
а в нём весь смысл страницы. Сравнение скриншотов: `.tmp/edge_roadmap.png` (полное дерево)
vs `.tmp/lumen_roadmap.png` (чёрный провал на месте дерева).

## Что НЕ сломано (проверено)

- **SVG layout+paint работают полностью.** Статический inline-`<svg>` с `rect/circle/path/text/viewBox`
  рисуется корректно (probe `.tmp/svg_static.html` → `.tmp/lumen_svg_static.png`: синий rect,
  красный circle, зелёный path, текст — всё на месте).
- **HTML DOM API работает:** `createElement`, `getElementById`, `textContent`, `appendChild`
  для HTML-узлов — нативные, доходят до арены (на странице это видно: `<span id="stat">`
  заполнен из JS).

## Корень проблемы (доказано probe-страницей)

Прогон `.tmp/probe.html` на свежем v0.5.0:
```
typeof Element     = undefined      ← нет глобального класса Element
typeof Node        = undefined      ← нет глобального класса Node
typeof EventTarget = function
document.createElementNS(...)  →  TypeError: not a function
```

Две связанные дыры в `crates/js/`:

### 1. `document.createElementNS` вообще не определён на настоящем `document`
В `crates/js/src/dom.rs` (объект `document`, ~строка 5529) есть `createElement`, но нет
`createElementNS`. Первый же вызов в хелпере страницы
`function s(tag,a,txt){const e=document.createElementNS(NS,tag);…}` бросает
`TypeError: not a function` → функция `render()` падает → `<svg>` пуст. Это и есть лог
`script error: JS runtime error: not a function`.

### 2. SVG DOM-иерархия (`svg.rs`) не устанавливается целиком
`crates/js/src/svg.rs:246` делает `class SVGElement extends Element`, но глобального `Element`
(и `Node`) в базовом шиме нет — DOM в `dom.rs` построен на прототипах/нативных байндингах,
**не на классах**. `extends undefined` бросает на этапе инициализации → весь
`install_svg_bindings` падает (`SVG DOM API init failed`), ни один `window.SVG*Element`,
ни патч `createElementNS` (тот, что внутри блока, `svg.rs:895`) не устанавливаются.

### Почему даже патч из svg.rs не помог бы (важно!)
Патч `createElementNS` в `svg.rs:899-915` возвращает `new Ctor()` — **чистый JS-объект без
нативного `__nid__`**. А нативный `appendChild` (`dom.rs:5547`) делает
`if (c && c.__nid__ !== undefined) _lumen_append_child(...)` — объект без `__nid__`
**молча игнорируется**, в арену не попадает, layout/paint его не видят. То есть текущая
типизированная SVG-иерархия архитектурно оторвана от нативного дерева и рендерить не может
в принципе.

## Ключ к простому фиксу

`layout` детектит SVG-узел **по имени тега, а не по namespace** —
`crates/engine/layout/src/box_tree.rs:521` (`is_svg_root`), прямой комментарий в коде:
«all elements are created with `Namespace::Html`. We detect SVG elements by local name».
Значит **нативный узел с local-именем `circle`/`rect`/`path`/… уже распознаётся как SVG**.
Достаточно, чтобы `createElementNS` создавал НАТИВНЫЙ узел (с `__nid__`), а не JS-объект.

### Доказательство фикса (end-to-end)
Probe `.tmp/svg_dyn.html`: на уровне страницы переопределил
`document.createElementNS = (ns,tag) => document.createElement(String(tag).toLowerCase())`,
затем построил SVG-дерево через `appendChild` → **отрисовалось идентично статическому**
(`.tmp/lumen_svg_dyn.png`: rect+circle+path+text на месте). Подход подтверждён.

## План фикса (минимальный, по приоритету)

1. **`crates/js/src/dom.rs` — добавить `createElementNS` в объект `document`** (рядом с
   `createElement`, ~5529). Для SVG namespace (`http://www.w3.org/2000/svg`) и вообще для
   любого — создавать нативный узел и возвращать `_lumen_make_element(nid)`:
   ```js
   createElementNS: function(ns, qualifiedName) {
       var local = String(qualifiedName).replace(/^[^:]+:/, '');
       // SVG case-sensitive: НЕ опускать регистр для camelCase-тегов
       // (linearGradient/clipPath). Для рендеринга текущей страницы хватит как есть.
       var nid = _lumen_create_element(local);   // см. примечание о namespace ниже
       if (nid < 0) throw new DOMException('DOM node limit exceeded', 'QuotaExceededError');
       return _lumen_make_element(nid);
   },
   ```
   Узким местом будет нативный `_lumen_create_element` (`dom.rs:716`): он жёстко вешает
   `QualName::html(tag.to_ascii_lowercase())`. Два варианта:
   - **(A) быстрый, достаточный для страницы:** оставить html-namespace — layout всё равно
     детектит по имени тега, рендер заработает. Регистр: `to_ascii_lowercase()` сломает
     детект case-sensitive SVG-тегов (`linearGradient`), но на roadmap-странице таких нет
     (только `g/circle/path/text/tspan`).
   - **(B) корректный:** добавить нативный байндинг `_lumen_create_element_ns(ns, local)`,
     создающий `QualName { namespace: Namespace::Svg, local }` (без lowercase), и научить
     `is_svg_*` в `box_tree.rs` смотреть на `namespace == Svg` ИЛИ имя тега. `Namespace::Svg`
     уже есть (`crates/engine/dom/src/lib.rs:112`). Это попутно закроет TODO в комментарии
     `box_tree.rs:519`.

   Рекомендация: сделать (A) в первом коммите (страница оживает), (B) — отдельным шагом
   как корректную foreign-content поддержку.

2. **`crates/js/src/svg.rs` — перестать падать в `install_svg_bindings`.** Сейчас весь блок
   рушится на `class SVGElement extends Element`. Варианты:
   - объявить базовый глобальный `class Element {}` (и `Node`) до svg-инициализации
     (в `dom.rs`/раннем bootstrap), тогда `extends` пройдёт и `instanceof SVGElement` заработает; ЛИБО
   - убрать зависимость от `Element` (`class SVGElement { … }` без наследования) — дешевле,
     но теряется `instanceof Element`.
   - **Важно:** патч `createElementNS` внутри `svg.rs:897` сейчас гейтится на
     `typeof document.createElementNS === 'function'`. После шага 1 он станет `true` и
     **перезапишет** нативный `createElementNS` сломанной pure-JS-версией (`new Ctor()`).
     Этот патч надо либо удалить, либо переписать так, чтобы он **декорировал нативный
     элемент** (вызвать оригинальный нативный `createElementNS`, потом навесить prototype
     с `getBBox`/типами), а не возвращать оторванный объект.

3. **Регресс-тест.** Добавить юнит/интеграционный тест: построить SVG через
   `document.createElementNS` + `appendChild`, проверить, что узлы попали в нативную арену
   (`document.getElementById('g').children.length > 0` или dump-layout содержит svg-боксы).
   Опционально — графический smoke на `.tmp/svg_dyn.html`-подобной странице.

## Вторично (отдельные баги, не блокируют этот фикс)

- **WebCodecs API init failed** — та же болезнь (`class X extends <отсутствующий базовый>`),
  скорее всего тот же недостающий `Element`/`EventTarget`-класс. Если решать через общий
  базовый `Element`/`Node` (шаг 2, вариант 1), отвалится попутно. Иначе — отдельный BUG.
- Отсутствие глобальных `Element`/`Node` как классов — системная дыра: любой сторонний
  шим/страница с `class X extends Element` или `instanceof Node` сломается. Стоит решить
  централизованно.

## Стратегический вывод

Это **не дефект рендеринга/CSS** — layout и paint в порядке (SVG и весь HTML/CSS-каркас
рисуются верно). Это **пробел в полноте JS DOM API**: class-ориентированные интерфейсы
(`Element`/`Node`/`SVGElement…`) и `createElementNS`. Под него попадает целый класс реальных
страниц (динамический SVG: графики, иконки, деревья — включая собственные roadmap-страницы
проекта). Graphic-тесты его не ловят: они синтетические и статические, JS-строящие страницы
не гоняются. Желательно завести smoke-слой «открыть реальный JS-driven файл, проверить,
что контент появился».
