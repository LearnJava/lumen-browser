# BUG-307: UI-поток «не отвечает» на тяжёлых вкладках; сессия умирает необратимо

**Статус:** OPEN
**Дата:** 2026-07-17
**Компонент:** shell (main thread) / ADR-016 (границы off-thread не покрывают этот путь)
**Найден:** живой перф-аудит `/lumen-perf-audit`; «приложение не отвечает» видел пользователь вживую

## Симптом

Метрика IsHungAppWindow (та же эвристика, по которой Windows пишет
«Не отвечает») в живом прогоне 14 вкладок (docs/perf/runs/2026-07-17-live.json):

```
lenta  hung=2.5с
ria    hung=39.0с   (окно висело всё время обработки после загрузки)
rbc    hung=47.5с   ready=23.4с
stackoverflow / habr — окно зависло НЕОБРАТИМО (даже navigate/new_tab
  отваливаются по «automation command timed out») → авторестарт харнесса
```

В прогоне без рестартов (2026-07-17, первый live-прогон): после
github (2.1 ГБ) + stackoverflow (3.4 ГБ) окно умерло навсегда — habr, mdn,
rbc не загрузились вовсе (TIMEOUT 240с каждый), хотя поодиночке эти сайты
грузятся за секунды. Для пользователя это «браузер завис, надо убивать
процесс» — худший класс UX-дефекта.

## Наблюдения

- Корреляция с RAM: необратимое зависание наступает после вкладок-гигантов
  (BUG-306); после авторестарта окна те же сайты грузятся нормально
  (crates 1.7с, docs-rs 1.1с, mdn 3.2с).
- Обратимые зависания (39–48 с на ria/rbc) — похоже на длинные синхронные
  участки на main thread (paint/layout/JS-дренаж), несмотря на влитые
  ADR-016 M0–M4 (off-thread layout/render существуют — проверить, какие
  пути всё ещё блокируют event loop).

## Ожидание

UI-поток не блокируется дольше сотен миллисекунд независимо от тяжести
вкладки; деградация допустима (janky-кадры), смерть сессии — нет.
Возможно нужен watchdog + принудительная выгрузка вкладки-агрессора.

## Диагностика

`LUMEN_FRAME_LOG=1` + Tracy на живом окне github→stackoverflow;
`scripts/perf_audit.py --only github --only stackoverflow --only habr`
воспроизводит за ~5 минут (колонка «не отвечает, с» в сводке).
