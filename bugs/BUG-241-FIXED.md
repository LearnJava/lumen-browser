# BUG-241

**Статус:** FIXED 2026-06-23
**Компонент:** layout
**Файл:** `crates/engine/layout/src/box_tree.rs`

## Описание

Вложенный column-flex item схлопывался по высоте контента вместо растяжения на
cross-size своей ячейки. Видно на TEST-90 (AVIF-страница): `.cell-item` (column-flex,
`flex:1` внутри row-flex `.cell`, которая сама `flex:1` внутри внешнего column-flex
`.__f`) рендерились короткими полосками по высоте текста-метки, а не заполняли ряды
на всю высоту.

## Реальная причина

`.__f` (column-flex) раскладывает каждую `.cell` дважды: сначала предварительным
indefinite-проходом (cross-size ряда ещё не известен), затем настоящим проходом с
`.cell`, растянутой на выросшую высоту. В `lay_out_flex` повторная раскладка
вложенного column-flex item-а после indefinite-прохода **не** была обусловлена
*definite* cross-size контейнера: при `explicit_cross = None` эффективный cross
падал на `line_cross` (собственная высота самой высокой ячейки строки), поэтому
"stretch" был no-op относительно текущей высоты item-а. Но re-layout всё равно
записывал резолвнутую px `style.height` обратно на item, безвозвратно затирая его
состояние `height: auto`. Последующий проход, у которого cross-size уже definite,
видел `is.height.is_some()` и пропускал настоящий stretch → вложенная ячейка
схлопывалась до content-высоты.

## Фикс

Добавлен гейт `&& explicit_cross.is_some()` в условие `relayout_column_flex`:
повторная раскладка column-flex item-а (и затирание `height: auto`) выполняется
только когда контейнер имеет определённый cross-size, т.е. когда stretch реально
что-то меняет.

Регресс-тест `flex_nested_stretch_after_indefinite_pass_fills_row` фиксирует
сценарий: вложенный `#col` (column-flex) внутри row-flex `#cell` внутри внешнего
column `#outer` (height:300px) — `#col` должен заполнить ряд (300px), а не
схлопнуться до content-высоты.

Найдено при разборе TEST-90 (ветка `p3-bug209`): схлопнутые ячейки — отдельный
дефект, не связанный с самим BUG-209 (отсутствующий AVIF-декодер).
