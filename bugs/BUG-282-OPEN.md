# BUG-282

**Статус:** OPEN (DEBTOR — KNOWN_DEBTORS 10.16%)
**Компонент:** layout / css-parser
**Тест:** TEST-142 (diff 10.16%)

## Описание

`@color-profile` + `color(--name c1 c2 c3)`: swatches отличаются от Edge-эталона.

## Расследование (2026-07-15)

Скриншот Lumen (`142-color-profile-lumen-cropped.png`) показывает ровно
задуманные цвета: чистые красный/зелёный/синий/серый/чёрный swatch-боксы плюс
полупрозрачный красный (`color(--swop5c 1 0 0 / 0.5)`). Реализация трактует
каналы `color(--name ...)` как уже sRGB (реальная ICC-трансформация отложена,
см. `predefined_to_srgb_linear`-соседняя ветка в `style.rs`).

Скриншот Edge (`142-color-profile-edge.png`) — **полностью пустой** (белый
фон `.__f`, ни одного swatch-а). Edge (актуальная стабильная сборка на момент
теста) не поддерживает CSS Color L5 `@color-profile`/`color(--dashed-ident>
...)` вообще: весь `color()` со custom-ident-профилем невалиден → каждое
`background` объявление отбрасывается → элементы остаются с UA-default
(прозрачный) фоном.

Diff-регион (`x:5–960 y:5–168`) — ровно область первых двух рядов swatch-ов,
подтверждает: расхождение целиком в тех элементах, что используют
`color(--name ...)`, никакой другой геометрии/цвета не задето.

## Почему не P3-фикс

Дефекта движка нет: Lumen рендерит спек-корректный (для реализованного среза)
результат. Совпасть с Edge означало бы либо (а) не реализовывать
`@color-profile`/`color()`-custom-ident вовсе, либо (б) воспроизвести
отсутствие поддержки — оба варианта отключают рабочую фичу. Тот же класс, что
TEST-71/BUG-199 (`@starting-style`) и TEST-77/BUG-126 (`inset-area`): Lumen
спек-корректнее reference-браузера.

## Резолюция

- TEST-142 → `KNOWN_DEBTORS` в `graphic_tests/run.py` (baseline 10.16%, anchor
  BUG-282).
- BUG-282 остаётся OPEN — закроется, когда Edge (или согласованный
  reference-браузер) добавит поддержку `@color-profile`, либо когда Lumen
  реализует реальную ICC-трансформацию (тогда потребуется новый эталон и,
  возможно, новый тест).
