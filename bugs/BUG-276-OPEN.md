# BUG-276 — wgpu/Vulkan на Linux: blur-тяжёлые страницы GPU-bound до клина свопчейна

**Статус:** OPEN · **Компонент:** paint (wgpu renderer) · **Найден:** 2026-07-12,
Linux-валидация ветки `p1-wgpu-cross-platform` (CachyOS, KDE/Wayland, Intel iGPU, Mesa Vulkan)

## Симптом

На страницах с большим числом полноэкранных фильтр-пассов
(`samples/bench-static-scroll.html` — 40 секций, по gaussian blur на секцию)
скролл под `LUMEN_BACKEND=wgpu` фактически стоит: CPU-часть кадра ~44 мс, но
GPU-исполнение занимает секунды, очередь present забивается, и каждый ~4-й
`get_current_texture` упирается в `SurfaceError::Timeout` (~7–10 с),
т.е. эффективные ~0.5 кадра/с. Периодичность «4 кадра ок → стойка» = глубина
свопчейна. Видимость окна и present mode (Fifo/Mailbox/Immediate) на картину
не влияют; на лёгких страницах (`samples/page.html`, 0.9 мс/кадр,
20 скроллов/с) проблемы нет. KWin делит тот же iGPU — его композитинг тоже
голодает, поэтому релизы буферов запаздывают.

Замер: `python3 scripts/bench_scroll.py --page samples/bench-static-scroll.html
--backend wgpu` (медиана 44 мс / p95 7300 мс / SurfaceLost×21 за прогон);
для сравнения femtovg на той же странице — стабильные ~1050 мс/кадр (CPU-
композитинг), Chromium — 60 fps при 37% CPU.

## Причина (уровень архитектуры)

Каждый фильтр рисуется в offscreen-текстуру размером с всю поверхность, без
ограничения по bbox контента и без отсечения слоёв вне вьюпорта — на
1024×900 это ~40 полноэкранных blur-пассов на кадр на iGPU. На ветке
`p1-exp-wgpu-only` это уже решено: bbox-scissor фильтр-пассов (3.2–4×),
viewport-cull невидимых слоёв (ещё 1.27×), скролл-композитор полоса+blit
(6–8×) — см. EXPERIMENT.md п.16–19. Перенос этих срезов в main отдельными
чистыми ветками — рекомендуемый фикс.

## Сопутствующее (исправлено в p1-wgpu-cross-platform)

- Shell не выполнял контракт `RenderBackend`: после `SurfaceLost` не звал
  `resize` и не повторял кадр — одна потеря поверхности навсегда убивала
  рендер. Исправлено (реконфигурация свопчейна по текущему размеру окна).
- `surface_error_to_render_error` теперь печатает исходный вариант
  (`Lost`/`Outdated`/`Timeout`) в stderr — иначе они неразличимы.
