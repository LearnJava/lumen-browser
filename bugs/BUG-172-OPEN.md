# BUG-172

**Статус:** OPEN
**Компонент:** shell (streaming / page pipeline)
**Приоритет:** низкий (лишняя сетевая работа, не визуальный дефект)

## Корень

Картинки `<img>` на странице со streaming-загрузкой скачиваются и декодируются
ДВАЖДЫ — двумя независимыми код-путями, которые не знают друг о друге:

1. **PH1-2c, прогрессивно во время streaming** (коммит 26d88c19):
   `spawn_stream_image_loads` (`crates/shell/src/main.rs:6051`) на каждом
   промежуточном кадре `paint_partial_dom` находит `<img>` через
   `lumen_layout::collect_image_requests` и спавнит фоновые потоки fetch+decode,
   присылая `LoadEvent::ImageDecoded`. Дедуп внутри пути — через
   `self.stream_images_requested` (`main.rs:6058`).

2. **Финальный полный pipeline** (мой параллельный fetch, коммит f9ee8b46):
   `fetch_and_decode_images` (`main.rs:2833`) в `parse_and_layout` по
   `LoadEvent::LoadDone` снова берёт `collect_image_requests` и качает+декодит
   ВСЕ не-lazy картинки через `parallel_map`.

`stream_images_requested` живёт в `Lumen` (UI-объект) и недоступен из
`parse_and_layout` (свободная функция, не метод). Поэтому финальный проход
повторяет всю работу streaming-пути.

Дублирование сетевой работы существовало частично и до PH1-2c: финальный
`parse_and_layout` всегда пере-фетчил все картинки. PH1-2c добавил второй
прогрессивный проход, сделав дубль полным. Это **не поломка** (renderer
регистрирует по ключу `src` идемпотентно, картинки корректны), но лишний
трафик и CPU на декодирование, особенно заметно на тяжёлых страницах с
десятками картинок.

## Что надо

Один fetch+decode на каждую картинку за навигацию. Финальный проход должен
переиспользовать уже загруженное streaming-путём, а не качать заново.

## Как реализовать (варианты)

- **A. Общий кэш декодированных картинок (предпочтительно).**
  Завести потокобезопасный `HashMap<src, Arc<Image>>` (per-load), который
  заполняют ОБА пути. `fetch_and_decode_images` сначала смотрит в кэш —
  при попадании пропускает сеть+декод. Это же закрывает этап 1 BUG-171
  (префетч-кэш подресурсов вне UI) — стоит делать заодно.

- **B. Прокинуть `stream_images_requested` + уже декодированные картинки в
  `parse_and_layout`.** Передавать множество уже-загруженных `src` и их
  `Image`, чтобы финальный проход их не трогал. Уже, чем A, но менее общо.

- **C. Не делать полный image-pass в финале, если streaming уже отработал.**
  Рискованно: финальный layout всё равно должен иметь intrinsic-размеры для
  всех картинок (в т.ч. тех, что streaming не успел догрузить) — нужна аккуратная
  сверка, какие `src` ещё не пришли. A безопаснее.

## Затронутые файлы

- `crates/shell/src/main.rs`:
  - `fetch_and_decode_images` :2833 (чтение из общего кэша),
  - `spawn_stream_image_loads` :6051 + `stream_images_requested` :4692 (запись в кэш),
  - `parse_and_layout` :3129 (проброс кэша) / обработчик `ImageDecoded` ~6459.

## Критерий приёмки

- Каждая не-lazy картинка качается и декодируется один раз за навигацию
  (проверить по логам `Загружена картинка: <src>` — без повторов одного src).
- Без визуальных регрессий: картинки на месте, intrinsic-размеры применены,
  `cargo test -p lumen-shell` зелёный.

## Связано

- Возникло из совмещения PH1-2c (db43759e) и параллельного fetch (f9ee8b46).
- Вариант A пересекается с этапом 1 BUG-171 (префетч-кэш подресурсов вне
  UI-потока) — логично решать вместе.
