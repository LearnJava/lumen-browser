# BUG-171

**Статус:** OPEN
**Компонент:** shell (event loop, threading)
**Приоритет:** высокий (визуальный фриз окна на всё время загрузки тяжёлой страницы)

## Корень

Тяжёлая часть конвейера страницы выполняется СИНХРОННО на UI-потоке (потоке
winit event loop). В обработчике `LoadEvent::LoadDone` (`crates/shell/src/main.rs:6369`)
вызывается `render_bytes` → `parse_and_layout` (`main.rs:3129`), которая на
одном потоке делает: fetch внешних скриптов → исполнение QuickJS → fetch+декод
картинок → fetch CSS → fetch+декод `@font-face` → layout. Пока это крутится,
event loop не обрабатывает `RedrawRequested` и ввод → **окно замораживается**
на всё время (после параллелизации fetch — ~4.5 с, до неё было ~7.5 с на lenta.ru).

Отдельный симптом: клик по adblock-галочке активного таба
(`main.rs:7657`: `tab.adblock = !tab.adblock` → `set_global_adblock_enabled` →
полный `reload()` `main.rs:5687`) запускает весь синхронный конвейер на
UI-потоке → «галочка очень долго думает».

Начальная сетевая загрузка ГЛАВНОГО HTML уже вынесена в поток
(`start_streaming_load`, `main.rs:~5860`, шлёт `LoadEvent::HtmlChunk`/`LoadDone`).
Но финальный `parse_and_layout` по `LoadDone` снова падает на UI-поток — там и
живёт фриз.

## Что надо

Окно должно оставаться отзывчивым (перерисовываться, показывать спиннер/прогресс,
реагировать на ввод) во время загрузки. Клик по adblock не должен морозить UI.

## Главное ограничение

**QuickJS-контекст (`rquickjs`) НЕ `Send`** и привязан к потоку: `run_scripts_with_dom`
создаёт рантайм и `js_ctx` остаётся живым после загрузки для обработчиков событий
(`addEventListener`). Поэтому `parse_and_layout` целиком нельзя просто перенести на
worker-поток и вернуть `ParsedPage` (он содержит `js_ctx`). Это определяет
поэтапную стратегию.

## Как реализовать — поэтапно

### Этап 1 (дёшево, снимает большую часть фриза): префетч-кэш подресурсов вне UI

Сеть — основной источник простоя UI-потока. Вынести именно её, оставив
JS+layout на UI:

1. Завести потокобезопасный кэш `HashMap<Url, Arc<Vec<u8>>>` (Mutex/`dashmap`),
   живущий на время загрузки страницы.
2. Preload-сканер уже работает в потоке (`feed_preload_and_emit`, `main.rs:2632`,
   спавнит загрузку CSS). Расширить его: качать в фоне ВСЕ найденные подресурсы
   (скрипты, шрифты, картинки, CSS) через `parallel_map`, складывать байты в кэш.
3. `fetch_image_bytes` / `fetch_subresource`-вызовы внутри `parse_and_layout`
   сначала смотрят в кэш — при попадании возвращают мгновенно. UI-поток тратит
   время только на CPU (parse + JS + layout), сеть уже прогрета и идёт
   параллельно стримингу HTML.
4. Пока главный HTML стримится и подресурсы качаются — UI-поток свободен,
   рисовать спиннер/прогрессив (инфраструктура промежуточных кадров PH1-2 есть).

Эффект: убирает ~3–4 с сетевого ожидания с UI-потока. Остаётся CPU-время
(JS ~1.9 с + layout ~0.3 с), но окно уже не «висит» всю загрузку.

### Этап 2 (архитектурно, Phase 4 / ADR-006): весь пайплайн вне UI

Запускать весь конвейер страницы (включая QuickJS) на выделенном per-tab
потоке, а на UI-поток слать готовый display list + снапшот документа.
Это направление `BrowserSession` (ADR-006, `core/src/ext.rs:1514`). QuickJS
живёт на своём потоке, обработчики событий гоняются через сообщения. Крупная
задача, не входит в быстрый фикс.

### Adblock-галочка (точечный быстрый фикс, можно отдельным коммитом)

Сейчас toggle делает синхронный `reload()` на UI-потоке (`main.rs:7657→5687`).
Минимум: запускать перезагрузку через тот же асинхронный путь, что и обычная
навигация (`start_streaming_load`), а не синхронный `reload()`. Тогда галочка
переключается мгновенно, а страница перерисовывается по приходу `LoadDone`.
В идеале — не делать полный reload вообще, а только переинсталлировать фильтр и
перекрасить (если контент не зависит от заблокированных подресурсов).

## Затронутые файлы

- `crates/shell/src/main.rs`:
  - обработчик `LoadEvent::LoadDone` :6369 (источник фриза),
  - `parse_and_layout` :3129 и его fetch-вызовы (чтение из префетч-кэша),
  - `feed_preload_and_emit` :2632 (расширить до всех типов подресурсов),
  - `enum LoadEvent` :100 (возможно +`SubresourcePrefetched`/прогресс),
  - adblock toggle :7657, `reload()` :5687 (сделать асинхронным),
  - `start_streaming_load` ~5860.

## Критерий приёмки

- Во время загрузки тяжёлой страницы окно перерисовывается (спиннер/прогрессив),
  не «висит».
- Клик по adblock-галочке возвращает управление мгновенно; перерисовка
  происходит по завершении фоновой перезагрузки.
- Без регрессий: `cargo test -p lumen-shell` зелёный; порядок исполнения
  скриптов и каскад CSS не нарушены (кэш отдаёт те же байты).
