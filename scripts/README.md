# Оркестратор задач Lumen

Автоматический запуск Claude Code сессий для разработчиков P1–P5.
Каждая задача выполняется в отдельной сессии с чистым контекстом.
После завершения задачи автоматически запускается следующая.

Ключевые возможности:

- **Параллельный запуск** до пяти сессий, каждая в своём окне.
- **Восстановление после краша** через `claude --resume <session_id>` — диалог продолжается с места остановки.
- **Fallback на Haiku при rate limit** — переключение на `claude-haiku-4-5` без паузы 5 минут.
- **Мягкая остановка** через стоп-файл — текущая задача доработает до конца.

## Требования

- Python 3.10+
- Claude Code CLI (`claude`) в PATH
- Файлы `STATUS-P1.md` ... `STATUS-P5.md` в корне проекта (нужен только тот, который запускаешь)

## Запуск

```bash
# Один разработчик (работает в текущем окне)
python scripts/orchestrator.py P1

# Два разработчика (каждый в отдельном окне)
python scripts/orchestrator.py P1 P2

# Все четверо
python scripts/orchestrator.py P1 P2 P3 P4

# С лимитом — максимум 3 задачи на каждого, потом стоп
python scripts/orchestrator.py P1 --max-tasks 3

# Все, но не больше 5 задач
python scripts/orchestrator.py P1 P2 P3 P4 --max-tasks 5
```

При запуске нескольких разработчиков каждый открывается в отдельном окне
с заголовком «Lumen P1», «Lumen P2» и т.д. Вывод Claude виден в каждом окне
в реальном времени.

## Статус

```bash
# Проверить кто работает, кто остановлен
python scripts/orchestrator.py --status
```

Пример вывода:

```
Статус разработчиков:
--------------------------------------------------
  P1: работает | задача #3 (обн. 2026-05-22 14:45:18)
  P2: работает | задача #1 (обн. 2026-05-22 14:32:01)
  P3: остановлен | выполнено задач: 5 (обн. 2026-05-22 14:20:00)
  P4: не запущен
--------------------------------------------------
```

## Остановка

```bash
# Остановить P1 после текущей задачи (мягкая остановка)
python scripts/orchestrator.py --stop P1

# Остановить нескольких
python scripts/orchestrator.py --stop P1 P3

# Остановить всех
python scripts/orchestrator.py --stop-all

# Немедленная остановка — Ctrl+C в окне
```

### Как работает мягкая остановка

1. `--stop P1` создаёт файл `scripts/.stop-P1`
2. Цикл проверяет этот файл **между задачами**
3. Текущая задача всегда доработает до конца
4. После завершения цикл увидит стоп-файл, удалит его и остановится

## Жизненный цикл задачи

```
orchestrator.py запускается
    │
    ├─► Проверка .session-P1.json → есть → возобновление через `claude --resume`
    │
    ├─► Проверка .stop-P1 → найден? → СТОП
    │
    ├─► Проверка STATUS-P1.md → нет задач? → СТОП
    │
    ├─► .jobstatus-P1 ← «работает | задача #N»
    │
    ├─► .session-P1.json ← {task_number, started}
    │
    ├─► Запуск: claude -p "..." --dangerously-skip-permissions
    │       (если включён fallback — добавляется `--model claude-haiku-4-5`)
    │       session_id захватывается из первого события → дописывается в .session-P1.json
    │       Claude читает STATUS-P1.md
    │       Берёт задачу (In progress или первую из Next)
    │       Работает...
    │       Вызывает /lumen-task-finish
    │       Сессия завершается
    │
    ├─► Проверка кода возврата
    │       0          → задача выполнена → .session-P1.json удалён
    │       rate limit → переключение на Haiku ИЛИ пауза 5 мин (см. ниже)
    │       auth 403   → пауза 60 сек, повтор
    │       прочее ≠0  → пауза 30 сек, повтор
    │
    └─► Следующая итерация цикла
            │
            └─► .jobstatus-P1 ← «остановлен» (при выходе из цикла)
```

## Обработка ошибок и rate limit

Оркестратор различает три класса сбоев и реагирует на каждый по-своему.

### Rate limit основной модели → fallback на Haiku

По умолчанию `claude` запускается без `--model` — CLI берёт настроенную модель (обычно Sonnet/Opus). При **первом** детекте rate limit оркестратор:

1. **Не ставит паузу 5 минут.**
2. Запоминает резервную модель `claude-haiku-4-5` для всех последующих вызовов **этого разработчика** в рамках текущего процесса оркестратора.
3. Печатает в лог заметный баннер:

```
============================================================
  RATE LIMIT основной модели
  Причина: задача #3
  Переключаюсь на резервную модель: claude-haiku-4-5
  Следующий вызов claude пойдёт через Haiku БЕЗ паузы.
  Сбросить fallback можно только перезапуском оркестратора.
============================================================
```

4. В `--status` строка разработчика становится `P1: fallback model | claude-haiku-4-5`.
5. На следующей итерации перед запуском claude печатается `Модель: claude-haiku-4-5`.

Если **и Haiku** упирается в лимит — срабатывает стандартная пауза 5 минут (`wait_for_rate_limit`). Квоты у Haiku обычно гораздо щедрее, такой каскад редок.

Сбросить fallback в рамках работающего оркестратора нельзя — только перезапуском процесса.

### Auth error (403)

Пауза 60 секунд, состояние сессии удаляется, попытка повторяется без `--resume`. Обычно связано с проблемами авторизации Claude CLI — за минуту чаще всего восстанавливается само.

### Прочие ненулевые коды возврата

Пауза 30 секунд, состояние сессии удаляется, попытка повторяется. Задача в счётчике не засчитывается.

## Восстановление после краша

При старте каждой задачи оркестратор пишет файл состояния:

```json
// scripts/.session-P1.json
{
  "developer": "P1",
  "task_number": 3,
  "started": "2026-05-28T14:32:01",
  "session_id": "abc123..."
}
```

`session_id` захватывается из первого `stream-json` события и дописывается в файл сразу. При нормальном завершении задачи файл удаляется.

Если оркестратор упал (Ctrl+C, закрытие терминала, отключение питания):

- При следующем запуске оркестратор найдёт `.session-P1.json`.
- Если есть `session_id` → запустит `claude --resume <id>` с recovery-промптом. Claude увидит полную историю диалога и состояние git, продолжит с места остановки.
- Если `session_id` нет (сессия не успела стартовать) → сбросит файл, начнёт задачу заново.

Возобновление учитывает текущий fallback — если до краша оркестратор уже переключился на Haiku, `--resume` пойдёт через Haiku.

## Служебные файлы

| Файл | Назначение |
|---|---|
| `scripts/.stop-P1` ... `scripts/.stop-P5` | Стоп-сигнал (создаётся `--stop`, удаляется оркестратором) |
| `scripts/.jobstatus-P1` ... `scripts/.jobstatus-P5` | Текущий статус (обновляется автоматически) |
| `scripts/.session-P1.json` ... `scripts/.session-P5.json` | Состояние активной задачи (`task_number`, `started`, `session_id`) для восстановления после краша |

Все служебные файлы в `.gitignore`, в репо не попадают.

## Частые вопросы

**Что если Claude упадёт посередине задачи?**
Зависит от того, что именно упало:
- **Сам процесс claude** (ненулевой код возврата) → оркестратор ждёт 30 сек и запустит новую сессию. Новая сессия прочитает STATUS-файл и продолжит незавершённую задачу (поле "In progress").
- **Сам оркестратор** (Ctrl+C, закрытие окна, отключение питания) → при следующем запуске поднимет `.session-PN.json`, вызовет `claude --resume <session_id>` и продолжит **тот же** диалог с места остановки. См. раздел «Восстановление после краша».

**Что если упёрся в rate limit основной модели?**
Оркестратор автоматически переключится на `claude-haiku-4-5` для этого разработчика и продолжит без паузы. См. раздел «Обработка ошибок и rate limit». Сброс fallback — перезапуском оркестратора.

**Что если STATUS-файл пустой?**
Цикл завершится с сообщением "Нет задач".

**Можно ли запустить два экземпляра для одного разработчика?**
Нет, будут конфликты — оба возьмут одну задачу. Один разработчик = одно окно.

**Как посмотреть что делает разработчик прямо сейчас?**
`python scripts/orchestrator.py --status` — сводка по всем.
Или открыть окно нужного разработчика — вывод Claude виден в реальном времени.
