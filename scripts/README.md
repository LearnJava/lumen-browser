# Оркестратор задач Lumen

Автоматический запуск Claude Code сессий для разработчиков P1–P5.
Каждая задача выполняется в отдельной сессии с чистым контекстом.
После завершения задачи автоматически запускается следующая.

Ключевые возможности:

- **Интерактивный мастер** — запуск без аргументов (`python scripts/orchestrator.py`) проводит по всем параметрам пошагово и печатает эквивалентную CLI-команду.
- **Параллельный запуск** до пяти сессий, каждая в своём окне.
- **Восстановление после краша** через `claude --resume <session_id>` — диалог продолжается с места остановки. Сбрасывается флагом `--new`.
- **Выбор модели на старте** — `--model haiku|sonnet|opus|fable|<custom>` или `LUMEN_MODEL=<id>`.
- **Fallback на резервную модель при rate limit** — интерактивный выбор (`haiku` / `sonnet` / `opus` / `fable` / custom) или unattended-режим через `--fallback-model` / `LUMEN_FALLBACK_MODEL`.
- **Делегирование задач кодерам Kilo Gateway** через `--coders assist|solo|team` — Claude-водитель пишет код руками Step 3.7 Flash (`assist`); оркестратор сам ведёт агент-петлю поверх двух чередующихся кодеров с визуальной приёмкой nano-omni (`solo`); полный цикл «Claude-брифы → кодеры → Claude-ревью и merge» (`team`). См. раздел «Делегирование кодерам Kilo».
- **Мягкая остановка** через стоп-файл — текущая задача доработает до конца.

## Требования

- Python 3.10+
- Claude Code CLI (`claude`) в PATH
- Файлы `STATUS-P1.md` ... `STATUS-P5.md` в корне проекта (нужен только тот, который запускаешь)
- **Только для `--coders`:** ключ `KILO_API_KEY` (в env или `.tmp/kilo.env`). SDK не нужен — клиент работает на stdlib `urllib`.

## Запуск

### Интерактивный мастер (без аргументов)

Самый простой способ — запустить **без единого аргумента**: оркестратор проведёт по всем параметрам пошагово.

```bash
python scripts/orchestrator.py
```

Мастер спрашивает по шагам:

1. **Что сделать** — запустить / статус / остановить / остановить всех / выход.
2. **Каких разработчиков** — `P1`–`P5` (можно `1 3 4` или `P1 P3`), с подсказкой роли каждого.
3. **Режим выполнения** — Claude (обычный) / Кодеры `assist` / Кодеры `solo` / Кодеры `team` (для кодеров напоминает про ключ и приватность).
4. **Стартовая модель** и **резервная при rate limit** — меню `haiku`/`sonnet`/`opus`/`fable`/вручную (пропускается для `solo` — там Claude не используется).
5. **Лимит задач** — `0` = без лимита (для P5 и `solo` рекомендует `1`).
6. **Старт с чистого листа** (`--new`).

В конце печатается сводка и **эквивалентная CLI-команда** (чтобы в следующий раз можно было запустить тем же набором флагов напрямую), затем — подтверждение запуска. `Ctrl+C` на любом шаге безопасно отменяет — ничего не запускается.

Всё, что делает мастер, доступно и флагами напрямую — см. ниже.

### Прямой запуск флагами

```bash
# Один разработчик (работает в текущем окне)
python scripts/orchestrator.py P1

# Два разработчика (каждый в отдельном окне)
python scripts/orchestrator.py P1 P2

# Все четверо
python scripts/orchestrator.py P1 P2 P3 P4

# С лимитом — максимум 3 задачи на каждого, потом стоп
python scripts/orchestrator.py P1 --max-tasks 3

# Все, но не больше 5 задач
python scripts/orchestrator.py P1 P2 P3 P4 --max-tasks 5

# Unattended-режим: заранее заданная резервная модель (без интерактивного prompt)
python scripts/orchestrator.py P1 --fallback-model haiku

# То же самое через env (удобно для CI / автозапуска по cron)
LUMEN_FALLBACK_MODEL=haiku python scripts/orchestrator.py P1 P2

# Принудительный старт с чистого листа: удалить .session-PN.json и не возобновлять
python scripts/orchestrator.py P1 --new
python scripts/orchestrator.py P1 P2 P3 P4 --new   # пробрасывается в каждое окно

# Стартовать сразу на конкретной модели (alias или полный ID)
python scripts/orchestrator.py P1 --model haiku
LUMEN_MODEL=haiku python scripts/orchestrator.py P1 P2
```

**Алиасы моделей** (можно указать вместо полного ID в `--model`, `--fallback-model` и env-переменных):

| Alias    | Полный model ID       |
|----------|-----------------------|
| `haiku`  | `claude-haiku-4-5`    |
| `sonnet` | `claude-sonnet-4-6`   |
| `opus`   | `claude-opus-4-8`     |
| `fable`  | `claude-fable-5`      |

Алиасы разворачиваются при разборе командной строки. В логах и в вызовах `claude --model <id>` всегда идёт полный ID — чтобы было видно, какая модель реально запущена. Любое другое значение (не из таблицы) передаётся в `claude` как есть.

При запуске нескольких разработчиков каждый открывается в отдельном окне
с заголовком «Lumen P1», «Lumen P2» и т.д. Вывод Claude виден в каждом окне
в реальном времени.

**Кроссплатформенность окон.** На Windows окна — это `cmd`. На Linux/BSD с
графической сессией скрипт подбирает первый установленный эмулятор терминала
(konsole → gnome-terminal → alacritty → kitty → xfce4-terminal → foot →
xterm → x-terminal-emulator). Если графической сессии нет (`DISPLAY` и
`WAYLAND_DISPLAY` не заданы), но есть `tmux`, каждый разработчик уходит в
detached-сессию `lumen-<dev>` (подключение — `tmux attach -t lumen-P1`). Если
нет ни того, ни другого — процессы запускаются в фоне текущего терминала (их
вывод перемешивается; для изоляции поставьте `tmux`). Зависшие дочерние
процессы `claude`/`cargo` добиваются после сессии на всех платформах (на
Linux — через `/proc`, на macOS/BSD — через `ps`).

## Сводка флагов CLI

| Флаг / env                       | Назначение                                                                                       |
|----------------------------------|--------------------------------------------------------------------------------------------------|
| `DEV ...` (позиционный)          | Разработчики для запуска: `P1` … `P5`. Один — в текущем окне; несколько — каждый в своём.        |
| `--max-tasks N`                  | Максимум задач на разработчика (`0` = без ограничения).                                          |
| `--model MODEL`                  | Стартовая модель для `claude --model <id>`. Принимает alias или полный ID.                       |
| `LUMEN_MODEL` (env)              | То же, что `--model`. Приоритет ниже CLI.                                                        |
| `--fallback-model MODEL`         | Резервная модель при rate limit (отключает интерактивный prompt). Alias или полный ID.           |
| `LUMEN_FALLBACK_MODEL` (env)     | То же, что `--fallback-model`. Приоритет ниже CLI.                                               |
| `--new`                          | Удалить `.session-PN.json` и стартовать с нуля, не возобновляя через `claude --resume`.          |
| `--coders assist\|solo\|team`    | Делегировать задачи кодерам Kilo. `assist` — Claude-водитель пишет код через `.tmp/kilo_client.py`; `solo` — оркестратор сам агент поверх кодеров без Claude; `team` — цикл «Claude-брифы → кодеры → Claude-ревью и merge» (`--max-tasks` считает циклы). См. раздел «Делегирование кодерам Kilo». |
| `--prep-model MODEL`             | Только `team`: модель Claude-бригадира (фаза 1, брифы). По умолчанию `fable`. Alias или полный ID. |
| `--review-model MODEL`           | Только `team`: модель Claude-ревьюера (фаза 3, ревью и merge). По умолчанию `--model`/дефолт CLI; haiku не рекомендуется. |
| `--stop DEV [DEV ...]`           | Мягкая остановка указанных разработчиков (доработают текущую задачу).                            |
| `--stop-all`                     | Мягкая остановка всех разработчиков.                                                             |
| `--status`                       | Печать статуса всех разработчиков и выход.                                                       |

Все «model»-флаги пробрасываются в дочерние окна при параллельном запуске. Подробности — в разделах ниже.

## Статус

```

Пример вывода:

  P2: работает | задача #1 (обн. 2026-05-22 14:32:01)
  P3: остановлен | выполнено задач: 5 (обн. 2026-05-22 14:20:00)
  P4: не запущен

```

## Остановка

```bash
# Остановить P1 после текущей задачи (мягкая остановка)
python scripts/orchestrator.py --stop P1

# Остановить нескольких
python scripts/orchestrator.py --stop P1 P3

# Остановить всех
python scripts/orchestrator.py --stop-all

# Немедленная остановка — Ctrl+C в окне
```

### Как работает мягкая остановка

1. `--stop P1` создаёт файл `scripts/.stop-P1`
2. Цикл проверяет этот файл **между задачами**
3. Текущая задача всегда доработает до конца
4. После завершения цикл увидит стоп-файл, удалит его и остановится

## Жизненный цикл задачи

```
orchestrator.py запускается
    │
    ├─► --new задан? → удалить .session-P1.json (без возобновления)
    │
    ├─► Иначе: проверка .session-P1.json → есть → возобновление через `claude --resume`
    │       (модель: --model если задана, иначе дефолт CLI; fallback ещё не активен)
    │
    ├─► Проверка .stop-P1 → найден? → СТОП
    │
    ├─► Проверка STATUS-P1.md → нет задач? → СТОП
    │
    ├─► .jobstatus-P1 ← «работает | задача #N»
    │
    ├─► .session-P1.json ← {task_number, started}   (session_id дописывается позже)
    │
    ├─► Запуск: claude -p "..." --dangerously-skip-permissions [--model <id>]
    │       --model добавляется, если задана --model (initial) или активен fallback
    │       приоритет: fallback_model > initial_model
    │       session_id захватывается из первого stream-json события → дописывается в .session-P1.json
    │       Claude читает STATUS-P1.md
    │       Берёт задачу (In progress или первую из Next)
    │       Работает...
    │       Вызывает /lumen-task-finish
    │       Сессия завершается
    │
    ├─► Проверка кода возврата
    │       0          → задача выполнена → .session-P1.json удалён
    │       auth 403   → проба сети: обрыв → ждать восстановления; сеть есть → пауза 60 сек
    │       rate limit → выбор резервной модели ИЛИ пауза 5 мин (см. ниже)
    │       прочее ≠0  → пауза 30 сек, повтор
    │
    └─► Следующая итерация цикла
            │
            └─► .jobstatus-P1 ← «остановлен» (при выходе из цикла)
```

> В режиме `--coders solo` шаг «Запуск claude» заменяется на `run_coder_solo` (своя
> worktree-петля поверх кодера Kilo; восстановление после краша — через собственный
> `.coder-session-PN.json`, а не `.session-PN.json`/`--resume`). Все остальные шаги
> цикла — стоп-файл, `--max-tasks`, проверка `STATUS-PN.md` через `has_tasks` — работают
> одинаково. Подробнее — раздел «Делегирование кодерам Kilo».

## Выбор модели на старте

По умолчанию `claude` запускается без `--model` — CLI берёт настроенную модель (обычно Sonnet/Opus). Чтобы стартовать сразу на конкретной модели, не дожидаясь rate limit:

```bash
# CLI-аргумент (alias или полный ID)
python scripts/orchestrator.py P1 --model haiku
python scripts/orchestrator.py P1 --model claude-haiku-4-5   # эквивалентно

# env-переменная (удобно для нескольких окон сразу)
LUMEN_MODEL=haiku python scripts/orchestrator.py P1 P2 P3 P4
```

Доступные алиасы: `haiku` / `sonnet` / `opus` (см. таблицу в разделе «Запуск»). Алиас разворачивается при разборе CLI, и дальше по всему коду — а главное в логах — фигурирует только полный model ID, чтобы вы видели, что реально запущено.

Заданная модель используется во **всех** вызовах `claude` (включая возобновление через `--resume`), пока не сработает fallback на резервную модель при rate limit (см. ниже). После активации fallback он переопределяет `--model` до конца жизни процесса оркестратора.

Приоритет источников: `--model` → `LUMEN_MODEL` → дефолт CLI.

`--model` и `--fallback-model` независимы и могут комбинироваться:

```bash
# Стартуем на Sonnet; если упрёмся в лимит — без вопросов уйдём на Haiku
python scripts/orchestrator.py P1 --model sonnet --fallback-model haiku
```

## Обработка ошибок и rate limit

Оркестратор различает три класса сбоев и реагирует на каждый по-своему.

### Rate limit основной модели → выбор резервной модели

По умолчанию `claude` запускается без `--model` — CLI берёт настроенную модель (обычно Sonnet/Opus). При **первом** детекте rate limit оркестратор:

1. **Не ставит паузу 5 минут.**
2. Показывает интерактивное меню выбора резервной модели (если она не задана заранее):

```
============================================================
[P1] Выберите резервную модель для переключения:
  1) haiku    — самая быстрая, отдельные щедрые лимиты (рекомендуется)
  2) sonnet   — баланс скорости и качества
  3) opus     — мощная, обычно общие лимиты с Sonnet
  4) fable    — новейшая, максимальное качество
  5) Ввести имя модели вручную (alias или полное claude-*)
============================================================
  (для unattended-режима задайте --fallback-model или LUMEN_FALLBACK_MODEL)
[P1] Выбор [1 = haiku]:
```

   - Пустой ввод (Enter) → первый вариант (`haiku` → `claude-haiku-4-5`).
   - Можно ввести alias (`haiku`/`sonnet`/`opus`) или полный ID напрямую вместо номера.
   - При Ctrl+C / EOF — также берётся первый вариант, чтобы не блокировать цикл.

После выбора алиас разворачивается в полный model ID, и баннер в логе показывает уже полный ID — видно, что реально запустилось.

3. Запоминает выбор для всех последующих вызовов **этого разработчика** в рамках текущего процесса оркестратора.
4. Печатает в лог заметный баннер:

```
============================================================
  RATE LIMIT основной модели
  Причина: задача #3
  Переключаюсь на резервную модель: claude-haiku-4-5
  Следующий вызов claude пойдёт через claude-haiku-4-5 БЕЗ паузы.
  Сбросить fallback можно только перезапуском оркестратора.
============================================================
```

5. В `--status` строка разработчика становится `P1: fallback model | claude-haiku-4-5`.
6. На следующей итерации перед запуском claude печатается `Модель: claude-haiku-4-5`.

Если **и резервная модель** упирается в лимит — срабатывает стандартная пауза 5 минут (`wait_for_rate_limit`).

Сбросить fallback в рамках работающего оркестратора нельзя — только перезапуском процесса.

#### Unattended-режим (без интерактивного prompt)

Для автозапуска (cron, CI, длительные ночные сессии) задайте резервную модель заранее — тогда интерактивное меню пропускается:

```bash
# Через CLI-аргумент (приоритет выше). Можно alias или полный ID.
python scripts/orchestrator.py P1 --fallback-model haiku

# Через env-переменную (наследуется в дочерние окна при параллельном запуске)
LUMEN_FALLBACK_MODEL=haiku python scripts/orchestrator.py P1 P2 P3 P4
```

При параллельном запуске нескольких разработчиков `--fallback-model` пробрасывается в каждое окно с уже развёрнутым полным ID.

Приоритет источников: `--fallback-model` → `LUMEN_FALLBACK_MODEL` → интерактивный prompt.

### Auth error (403)

403 приходит в двух случаях: реальная проблема авторизации Claude CLI **или обрыв сети** (без сети запрос не проходит аутентификацию). Оркестратор различает их TCP-пробой `api.anthropic.com:443`:

- **сети нет** — ждёт её восстановления (проба раз в 30 сек), после чего сразу возобновляет сессию через `--resume`;
- **сеть есть** — пауза 60 секунд (реальный auth error за минуту чаще всего восстанавливается сам), затем возобновление через `--resume`.

Проверка 403 имеет приоритет над флагом rate limit: терминальный `result` с `api_error_status: 403` достовернее текстовых срабатываний, накопленных за сессию. Причина ошибки берётся структурно из JSON-событий CLI; текстовые регэкспы применяются только к не-JSON строкам вывода (служебное `rate_limit_event` со `status: "allowed"` — не rate limit).

### Прочие ненулевые коды возврата

Пауза 30 секунд, состояние сессии удаляется, попытка повторяется. Задача в счётчике не засчитывается.

## Делегирование кодерам Kilo (`--coders`)

Оркестратор умеет отдавать задачи бесплатным моделям через **Kilo Gateway** (OpenAI-совместимый, `https://api.kilo.ai/api/gateway`). Состав команды выбран бенчем 2026-07-02 (`.tmp/kilo_bench.py`, результаты `.tmp/kilo_bench_results.json`):

| Роль | Модель | id |
|---|---|---|
| Кодер №1 (нечётные задачи) | Step 3.7 Flash — быстрые «руки»: полный модуль с одного патча | `stepfun/step-3.7-flash:free` |
| Кодер №2 (чётные задачи) | Nemotron 3 Ultra — качество кода, честный фидбэк-цикл | `nvidia/nemotron-3-ultra-550b-a55b:free` |
| Визуальная приёмка | Nemotron 3 Nano Omni — «глаза», мультимодальная (точно описывает скриншоты; для правок кода непригодна) | `nvidia/nemotron-3-nano-omni-30b-a3b-reasoning:free` |

Ранее интегрированная Laguna M.1 (poolside) дисквалифицирована тем же бенчем — пустые ответы на спек-задачи, «чинит» провалы удалением тестов, нестабильный эндпоинт; её интеграция удалена.

Включается флагом:

```bash
python scripts/orchestrator.py P4 --coders assist              # Claude-водитель + кодер step37
python scripts/orchestrator.py P4 --coders solo --max-tasks 2  # без Claude: задача 1 → step37, задача 2 → nemotron
python scripts/orchestrator.py P1 --coders team --max-tasks 1  # цикл: Claude-брифы → кодеры → Claude-ревью и merge
python scripts/orchestrator.py P1 P4 --coders assist           # флаг идёт в каждое окно
```

Значения: `assist` (Claude остаётся водителем), `solo` (оркестратор сам агент поверх кодеров) и `team` (трёхфазный цикл с Claude-бригадиром и Claude-ревьюером). Без флага поведение не меняется — это полностью опциональный слой.

### Требования и ключ

- Ключ `KILO_API_KEY` (JWT, берётся на app.kilo.ai → API Keys). Читается из переменной окружения, а если её нет — из `.tmp/kilo.env` (gitignored), формат `export KILO_API_KEY=...`.
- **SDK не нужен**: клиент `KiloClient` работает на stdlib `urllib` — та же форма запроса, что у `.tmp/kilo_client.py`.
- **Сеть.** Оркестратор живёт в реальном окне терминала, поэтому сетевые вызовы работают (в отличие от песочницы Claude Code, где сеть только в foreground).
- **reasoning придушен.** Всем моделям шлётся `reasoning: {"effort": "low"}` — иначе reasoning-модели жгут токены на раздумья. Модель, не знающая параметр, получает повтор без него (детект по 400).
- **Троттлинг free-tier общий на ключ**: 403/429 обрабатываются прогрессивным бэкоффом (до 8 попыток, пауза до 180 с). Параллельные окна с `--coders` душат друг друга — запускай по одному (оркестратор печатает предупреждение).

> ⚠️ **Приватность.** Free-эндпоинты Kilo/NVIDIA — trial, запросы **логируются** на их стороне. В режиме `solo` наружу уходят содержимое исходников (команда `READ`) и скриншоты страниц (визуальная приёмка). Режимы включаются только явным флагом; при старте `solo` печатается баннер-предупреждение. Не гоняй на чувствительном коде.

### Режим `assist` — Step 3.7 Flash под управлением Claude

Claude остаётся полноценным водителем: читает `STATUS-PN.md`, правит файлы, гоняет `cargo`/`git`, вызывает `/lumen-task-finish`. Меняется только одно — к его промпту (и стартовому, и recovery после ошибки) приклеивается инструкция `CODERS_ASSIST_NOTE`:

> код НЕ пиши сам — собери точное ТЗ в messages-JSON (файл в `.tmp/`) и вызови `source .tmp/kilo.env && python .tmp/kilo_client.py <messages.json> <ответ.txt>`; ревью чужим взглядом — тот же вызов с `KILO_MODEL='nvidia/nemotron-3-ultra-550b-a55b:free'`; получив код, САМ проверь его (cargo check/clippy/test `-p <crate>`), при ошибках отправь на доработку; затем интегрируй по правилам Lumen и закоммить.

Что это значит на практике:

- **Правила разработчика соблюдаются полностью** — их держит Claude (ему автозагружается `CLAUDE.md`, он умеет `/lumen-task-finish`: clippy → test → merge `--no-ff` → доксинк → удаление ветки).
- **Claude-сессия и её лимиты всё ещё нужны.** Кодер здесь — «пишущая машинка для кода», а не замена Claude.
- **Экономии токенов Claude практически нет**: дорогая часть — контекст (CLAUDE.md, чтение исходников, вывод cargo, агентный цикл) — не уменьшается, плюс добавляются ТЗ, чтение кода кодера обратно в контекст и раунд-трипы исправлений. Assist покупает делегирование «печати», а не лимиты.

### Режим `solo` — два независимых кодера + визуальная приёмка

Здесь Claude **не участвует вообще** — это и есть реальная экономия лимитов Claude. Оркестратор сам ведёт многоходовый диалог с кодером (`run_coder_solo`, клиент `KiloClient`, in-process). **Кодеры чередуются по задачам**: нечётная задача цикла — Step 3.7 Flash, чётная — Nemotron 3 Ultra. Каждый работает в своём worktree над своей задачей — два независимых кодера, не мешающих друг другу.

**Изоляция в worktree.** Для каждой задачи создаётся отдельный `git worktree` от `main`:
- путь `.claude/worktrees/<dev>-<кодер>-<HHMMSS>` (напр. `p4-step37-140202`);
- ветка `p<N>-<кодер>-t<task>-<HHMMSS>` (напр. `p4-step37-t1-140202`);
- все правки/чтения/сборки идут туда, корень репозитория не трогается — конфликта с параллельными сессиями нет.

**Текстовый tool-протокол.** У кодера нет инструментов, поэтому он «вызывает» их текстом — по одной команде на строку, с начала строки:

| Команда | Действие |
|---|---|
| `READ <путь>` | прислать содержимое файла (путь относительно корня worktree) |
| `LIST <glob>` | список файлов по маске (напр. `crates/**/*.rs`), до 200 строк |
| `WRITE <путь>` | СЛЕДУЮЩИМ идёт один ```‑блок с **полным** новым содержимым файла |
| `BASH <команда>` | выполнить команду (см. whitelist ниже) |
| `DONE <текст коммита>` | задача готова — прогнать ворота и закоммитить |

В одном ответе допускается несколько команд. После `WRITE` оркестратор применяет файл. Системный промпт требует **перед `WRITE` всегда делать `READ`** изменяемого файла — `WRITE` перезаписывает файл целиком, частичный текст уничтожит остальное.

**Границы безопасности:**
- `WRITE`/`READ` ограничены корнем worktree (`_safe_path`) — выйти за пределы (`../…`) нельзя.
- `WRITE` в `graphic_tests/` запрещён: тестовые страницы неприкосновенны (бенч показал, что free-модели «чинят» провалы удалением/правкой тестов). Системный промпт дополнительно запрещает удалять и ослаблять существующие тесты.
- `BASH` пропускает только чтение и сборку: `cargo check|test|clippy|build|fmt`, `grep`, `rg`, `ls`, `git status|diff|log|branch`. Чёрный список режет `rm`/`del`/редиректы `>`/`push`/`reset --hard`/`checkout`/`--force`/`rebase`/`clean -`.

**Ворота перед коммитом (`_run_solo_gates`).** На `DONE` оркестратор не верит модели на слово, а сам прогоняет по каждому затронутому crate (имя берётся из `[package] name` ближайшего `Cargo.toml`):

```
cargo check  -p <crate>
cargo clippy -p <crate> --all-targets -- -D warnings
cargo test   -p <crate>
```

Любой ненулевой код → `DONE` отклоняется, вывод ошибок/варнингов возвращается кодеру на доработку. Если crate определить не удалось (правился не-crate файл) — прогоняется общий `cargo check`. Это полный набор правил Lumen «перед коммитом», а не просто «компилируется».

**Визуальная приёмка (nano-omni).** После зелёных ворот и коммита оркестратор снимает headless CPU-скриншоты (`lumen --screenshot`) страниц `samples/page.html` и `graphic_tests/1000000-final.html` — из worktree кодера и из main (эталон кэшируется на процесс) — и отправляет пары изображений nano-omni с просьбой описать различия и дать вердикт: `PASS` (различий нет/косметика), `WARN <причина>` (подозрительно), `FAIL <причина>` (явная поломка). Короткий вердикт идёт в worklog, полный ответ — в `scripts/vision-reports/<ветка>.md`. Вердикт **совещательный**: коммит не откатывается, финальное решение принимает Claude-ревью. Если задача не тронула код движка — `N/A`; если скриншоты или модель недоступны — `SKIP (...)`.

**Следы для ревью.** На успехе `solo` оставляет два следа для Claude-разработчика:
- строка-указатель в `STATUS-PN.md` помечается `<указатель> | <ветка> | <маркёр>` (маркёры: `Step37`, `Nemotron3U`);
- в `scripts/coders-worklog.md` дописывается строка: время / dev / кодер / задача / указатель / ветка / worktree / commit / vision-вердикт.

Обычная Claude-сессия (запуск без `--coders solo`) **перед своей задачей обязана** разобрать эти наработки (инструкция `coders_review_note` в промпте): ревью диффа ветки, ворота, `merge --no-ff` в main, доксинк, чистка worklog/STATUS/vision-отчёта — или мотивированный отказ от влития. Это и есть полный цикл: **кодер пишет → nano-omni смотрит → Claude проверяет, одобряет и добавляет в проект**.

**Параметры цикла:**
- лимит итераций `CODER_MAX_ITERS = 24`; при исчерпании — останов, worktree с незавершённой работой остаётся (в логе — команда удаления);
- `temperature=0.2`, `max_tokens=16000`, `reasoning effort=low`; докачка длинных ответов по `finish_reason == "length"` — кодерам до 4 кусков, nano-omni до 2 (склонен зацикливаться на continuation);
- сбой сети/API на итерации → пауза 30 с и повтор; 5 сбоев подряд → останов с сохранением сессии (резюмируемо);
- `solo` **не** ведёт `.session-PN.json` (это claude-механизм). У него **своё** восстановление после краша через `.coder-session-PN.json` — см. ниже;
- стоп-файл и `--max-tasks` действуют как обычно (проверяются между задачами). Наличие задач берётся из `STATUS-PN.md` тем же `has_tasks`.

**Рекомендуемый режим.** Для полного автомата используй `team` (ниже) — он сам готовит брифы и сам вливает. `solo` оставлен для случая «просто набросай черновики, разберу потом».

### Режим `team` — Claude-бригадир → кодеры → Claude-ревью

Полный конвейер одного цикла (`--coders team`), закрывающий главный провал `solo` (прогон 2026-07-07: кодеры получали сырой указатель на большую roadmap-задачу и тонули в разведке — 24 итерации grep без единого WRITE):

1. **Фаза 1 — брифы.** Запускается Claude-сессия «бригадир» (`team_prep_prompt`): читает `STATUS-PN.md`, берёт две первые непомеченные задачи, для крупных выделяет один мелкий срез (XS/S), пишет по самодостаточному брифу в `.tmp/briefs/<dev>-task-a.md`/`-b.md` (точные файлы/строки, сигнатуры, выдержки кода, что не трогать, критерии готовности) и манифест `.tmp/briefs/<dev>-manifest.json`. Код не меняет. Сессия завершается.
2. **Фаза 2 — кодеры.** Оркестратор раздаёт брифы: задача A — Step 3.7 Flash, задача B — Nemotron 3 Ultra (последовательно — троттлинг ключа общий). Механика как в `solo` (worktree, протокол, ворота, vision), но стартовое сообщение кодера — целиком бриф, и **DONE без единого WRITE отклоняется**.
3. **Фаза 3 — ревью.** Запускается Claude-сессия «ревьюер» (`team_review_prompt`): для каждой успешной ветки — ревью диффа против брифа + ворота + вердикт vision; одобрено → `merge --no-ff` в main, доксинк, чистка STATUS/worklog/vision-отчёта, удаление ветки и worktree; отклонено → ветка сносится без влития, указатель возвращается в STATUS, причина — в итоге сессии. Провальные пустые ветки просто сносятся. В конце `git push origin main` и уборка брифов. Сессия завершается.
4. Все сессии закрыты — цикл повторяется, пока есть задачи. **`--max-tasks` считает циклы** (1 цикл = до 2 задач): `--coders team --max-tasks 1` — один полный заход.

**Модели фаз.** Claude-фазы настраиваются раздельно: `--prep-model` — бригадир (по умолчанию **fable**: качество брифов — главный фактор успеха слабых кодеров), `--review-model` — ревьюер (по умолчанию `--model` / дефолт CLI). Слабые модели (haiku) для ревью **не рекомендуются** — ревью самый ответственный этап конвейера: именно ревьюер решает, что попадёт в main. Активный fallback при rate limit переопределяет обе. Пример: `python scripts/orchestrator.py P1 --coders team --review-model opus --max-tasks 1`.

Экономия токенов Claude против обычного режима: фазы 1 и 3 — короткие сессии (брифы и ревью), вся генерация кода и раунды починки ошибок компиляции уходят на бесплатных кодеров. Расплата — качество кодеров: часть веток будет отклоняться на ревью.

### Восстановление solo-сессии после краша

У `solo` нет `claude --resume`: API Kilo (OpenAI-совместимый chat completions) **stateless** — серверного состояния диалога, которое можно «возобновить по id», не существует. Единственный поддерживаемый способ продолжить разговор — **переслать весь массив `messages` заново**. Поэтому оркестратор делает это сам, на стороне клиента:

- После **каждой итерации** состояние пишется в `scripts/.coder-session-PN.json`: `messages` (весь диалог READ/WRITE/BASH), путь к worktree, ветка, список записанных файлов, номер задачи и итерации, **кодер задачи**. **Ключ `KILO_API_KEY` в файл НЕ пишется** — он перечитывается из env/`.tmp/kilo.env` при каждом запуске (это секрет, не часть состояния). Файл в `.gitignore`.
- Worktree фиксируется в файле **сразу при создании** — чтобы краш даже на первой итерации был восстановим.
- При следующем запуске `solo`, если файл найден и его worktree цел, оркестратор **переигрывает сохранённый `messages` в том же worktree тем же кодером** и продолжает с места обрыва (новый worktree не создаётся, чередование кодеров игнорируется). В лог идёт `Возобновляю прерванную solo-сессию: задача #N, итер. K`.
- Если worktree пропал (удалён вручную) — состояние сбрасывается, задача стартует заново со свежим worktree.
- Файл удаляется: на **успехе** (ворота зелёные + коммит), при **исчерпании 24 итераций** (терминальная неудача — чтобы повторный запуск не возобновлял заведомо застрявший диалог, а начал новую попытку), и по флагу **`--new`** (`--coders solo --new` принудительно стартует с чистого листа).

Что переживает краш в любом случае: файлы, уже записанные через `WRITE`, лежат на диске в worktree; закоммиченное — в git. С персистом дополнительно сохраняется **контекст диалога** — кодер не теряет, что уже прочитал и что чинил.

> Отличие от Claude-режимов: там состояние — это `session_id` в `.session-PN.json`, а историю хранит сам Claude Code. У `solo` истории на сервере нет, поэтому весь диалог хранится локально в `.coder-session-PN.json` и переигрывается при возобновлении.

### Сравнение режимов

| | без `--coders` | `--coders assist` | `--coders solo` | `--coders team` |
|---|---|---|---|---|
| Кто водитель | Claude | Claude | оркестратор | оркестратор (Claude в фазах 1 и 3) |
| Кто пишет код | Claude | Step 3.7 Flash | step37 / nemotron (чередуются) | step37 (задача A) + nemotron (задача B) по брифам Claude |
| Claude-сессия нужна | да | да | **нет** (ревью — потом) | да, две короткие на цикл (брифы + ревью) |
| Экономия токенов Claude | — | ≈ ноль | максимальная | высокая (генерация и починка ошибок — у кодеров) |
| Правила разработчика (clippy/test) | да | да (через Claude) | да (ворота `_run_solo_gates`) | да (ворота + Claude-ревью) |
| Визуальная приёмка | графические тесты | графические тесты | скриншоты → nano-omni (вердикт в worklog) | скриншоты → nano-omni + вердикт учитывается на ревью |
| Финиш (merge/доки/STATUS) | авто (`/lumen-task-finish`) | авто (Claude) | **Claude-сессия позже** (`coders_review_note`) | авто в фазе 3 (Claude-ревьюер) |
| Восстановление после краша | `--resume` (`.session-PN.json`) | `--resume` (`.session-PN.json`) | переигрывание `messages` (`.coder-session-PN.json`) | фазы 1/3 — `--resume`; фаза 2 — `.coder-session-PN.json` |
| Worktree-изоляция | да (запускается отдельно) | да | да (создаётся сам) | да (создаётся сам) |
| Уход кода наружу (Kilo/NVIDIA) | нет | ТЗ + ревью кода | исходники (`READ`) + скриншоты | брифы + исходники (`READ`) + скриншоты |

## Восстановление после краша

При старте каждой задачи оркестратор пишет файл состояния:

```json
// scripts/.session-P1.json
{
  "developer": "P1",
  "task_number": 3,
  "started": "2026-05-28T14:32:01",
  "session_id": "abc123..."
}
```

`session_id` захватывается из первого `stream-json` события и дописывается в файл сразу. При нормальном завершении задачи файл удаляется.

Если оркестратор упал (Ctrl+C, закрытие терминала, отключение питания):

- При следующем запуске оркестратор найдёт `.session-P1.json`.
- Если есть `session_id` → запустит `claude --resume <id>` с recovery-промптом. Claude увидит полную историю диалога и состояние git, продолжит с места остановки.
- Если `session_id` нет (сессия не успела стартовать) → сбросит файл, начнёт задачу заново.

Возобновление идёт через модель, заданную текущим запуском (`--model` / `LUMEN_MODEL`), либо через дефолт CLI, если она не указана. Решение о fallback (если оно было принято в упавшей сессии) **не сохраняется** между процессами оркестратора — после рестарта fallback считается неактивным. Если упавшая сессия уже работала через Haiku и хотите продолжить там же — задайте явно `--model haiku` при перезапуске.

### Принудительный старт с чистого листа (`--new`)

Если прошлая сессия зависла, её контекст больше не релевантен, или просто нужно начать новую задачу без истории диалога — запустите с флагом `--new`:

```bash
python scripts/orchestrator.py P1 --new
python scripts/orchestrator.py P1 P2 P3 P4 --new   # пробрасывается в каждое окно
```

Что делает `--new`:

1. До старта основного цикла удаляет `scripts/.session-PN.json`, если он есть.
2. Пропускает попытку `claude --resume <session_id>` — сразу стартует свежий диалог.
3. В лог пишет одну из двух строк:
   - `Флаг --new: удаляю сохранённое состояние сессии, не возобновляю.`
   - `Флаг --new: сохранённого состояния нет, стартую с нуля.`

Дальше всё как обычно: оркестратор читает `STATUS-PN.md`, берёт первый указатель-задачу (для P5 — рекуррентную секцию), запускает `claude` без `--resume`. Файл `.session-PN.json` создаётся заново при старте новой задачи.

## Служебные файлы

| Файл | Назначение |
|---|---|
| `scripts/.stop-P1` ... `scripts/.stop-P5` | Стоп-сигнал (создаётся `--stop`, удаляется оркестратором) |
| `scripts/.jobstatus-P1` ... `scripts/.jobstatus-P5` | Текущий статус (обновляется автоматически) |
| `scripts/.session-P1.json` ... `scripts/.session-P5.json` | Состояние активной задачи (`task_number`, `started`, `session_id`) для восстановления после краша (Claude-режимы) |
| `scripts/.coder-session-P1.json` ... `scripts/.coder-session-P5.json` | Состояние solo-сессии кодера (`messages`, worktree, ветка, `written`, `iter`, кодер) для возобновления после краша. **Без ключа.** Только при `--coders solo` |
| `scripts/coders-worklog.md` | Наработки кодеров, ожидающие ревью и влития (одна строка = одна задача). Чистится Claude после merge |
| `scripts/vision-reports/<ветка>.md` | Полный отчёт визуальной приёмки nano-omni по ветке кодера. Чистится Claude после merge |
| `scripts/coder-debug-P1.log` ... | Debug-лог обмена с Kilo (тайминги, троттлинг). Транзиентный |

Все служебные файлы в `.gitignore`, в репо не попадают.

## Частые вопросы

**Что если Claude упадёт посередине задачи?**
Зависит от того, что именно упало:
- **Сам процесс claude** (ненулевой код возврата) → оркестратор ждёт 30 сек и запустит новую сессию. Новая сессия прочитает STATUS-файл и продолжит незавершённую задачу (её ветка `p<N>-…` уже существует).
- **Сам оркестратор** (Ctrl+C, закрытие окна, отключение питания) → при следующем запуске поднимет `.session-PN.json`, вызовет `claude --resume <session_id>` и продолжит **тот же** диалог с места остановки. См. раздел «Восстановление после краша».

**Что если упёрся в rate limit основной модели?**
Оркестратор покажет интерактивное меню выбора резервной модели (`haiku` / `sonnet` / `opus` / `fable` / custom). По умолчанию — `haiku` (`claude-haiku-4-5`). После выбора разработчик продолжит без паузы. Для unattended-режима без меню задайте `--fallback-model haiku` или `LUMEN_FALLBACK_MODEL=haiku`. См. раздел «Rate limit основной модели». Сброс fallback — перезапуском оркестратора.

**Можно ли сразу стартовать на Haiku, не дожидаясь rate limit?**
Да: `--model haiku` (или `LUMEN_MODEL=haiku`). См. раздел «Выбор модели на старте».

**Что делать, если прошлая сессия зависла или её контекст больше не нужен?**
Запустить с `--new` — сохранённое состояние сессии удалится, оркестратор стартует с нуля. См. «Принудительный старт с чистого листа».

**Можно ли отдать задачу не Claude, а бесплатным моделям?**
Да: `--coders assist` (Claude-водитель + кодер Step 3.7 Flash) или `--coders solo` (без Claude вообще: step37/nemotron чередуются по задачам, визуальная приёмка nano-omni). См. раздел «Делегирование кодерам Kilo». Нужен `KILO_API_KEY` (env или `.tmp/kilo.env`).

**Экономит ли `--coders` лимиты Claude?**
`assist` — практически нет (контекст тот же, плюс ТЗ + ревью кода кодера). Реальная экономия только у `solo` — там Claude не вызывается вообще. Расплата: `solo` не доводит задачу по правилам (ревью/merge/доки/STATUS делает Claude-сессия потом). Рекомендуемый гибрид — `solo` как генератор черновиков + финальный проход Claude, который вливает проверенное.

**Куда `solo` девает результат?**
В отдельный worktree `.claude/worktrees/<dev>-<кодер>-<HHMMSS>` (ветка `p<N>-<кодер>-*`), коммитит туда на зелёных воротах (check + clippy `-D warnings` + test), после чего nano-omni даёт вердикт по скриншотам. Ветка/worktree остаются для ревью; следы — в `scripts/coders-worklog.md` и в маркёре строки STATUS. Корень репозитория и чужие сессии не затрагиваются.

**Уйдёт ли мой код наружу?**
В `assist` наружу идёт ТЗ и фрагменты, которые Claude шлёт в `.tmp/kilo_client.py`. В `solo` команда `READ` отправляет содержимое читаемых файлов, а визуальная приёмка — скриншоты страниц. Free-эндпоинты Kilo/NVIDIA — trial, запросы логируются — не используй на чувствительном коде.

**А если solo-сессия прервалась посередине?**
Восстановится сама: оркестратор персистит диалог в `scripts/.coder-session-PN.json` после каждой итерации и при следующем запуске переигрывает его в том же worktree тем же кодером (у Kilo нет серверного `--resume`, API stateless — продолжение возможно только пересылкой всего диалога). Ключ туда не пишется. Принудительно с чистого листа — `--coders solo --new`. Детали — раздел «Восстановление solo-сессии после краша».

**Что если STATUS-файл пустой?**
Цикл завершится с сообщением "Нет задач".

**Можно ли запустить два экземпляра для одного разработчика?**
Нет, будут конфликты — оба возьмут одну задачу. Один разработчик = одно окно.

**Как посмотреть что делает разработчик прямо сейчас?**
`python scripts/orchestrator.py --status` — сводка по всем.
Или открыть окно нужного разработчика — вывод Claude виден в реальном времени.
