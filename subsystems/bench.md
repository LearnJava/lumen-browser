# lumen-bench ✅ (baseline pipeline measurements)

- **Done:** Standalone binary crate with no third-party dependencies. Runs `decode → parse_html → parse_css → layout → paint::build_display_list` on `samples/page.html` + `samples/page.css` + bundled Inter; prints min / median / mean / p95 / max per phase and TOTAL. `LUMEN_BENCH_ITERS` env var overrides iteration count (default 100). 10 warm-up iterations before measurements. `std::hint::black_box` wraps pipeline results — protection against dead-code elimination in LTO release builds. Run: `cargo run -p lumen-bench --release`.
- **CI gate (9G.3):** `--ci` flag runs 3 pipeline iterations on `samples/heavy.html`, asserts mean total < 200 ms and peak RSS < 512 MB, exits 1 on failure. Implemented in `bench/src/ci_gate.rs` + `bench/src/util.rs`. 6 unit tests. Invocation: `cargo run -p lumen-bench --release -- --ci`.
- **Cold/warm start bench (PERF-4):** `--startup` flag (`bench/src/startup.rs`) is *process-level*, not in-process — it spawns the already-built shell binary headlessly (`--screenshot <tmp> <url>`) and measures wall-clock per spawn, so it captures the OS loader cost of the ~70 MB (statically-linked V8) binary plus V8 platform/isolate init and DOM-shim install, which an in-process bench cannot. cold = first spawn (cold loader cache), warm = steady state of the rest, `cold − warm_median` = loader/init warm-up cost. Flags `--iters N` / `--url <page>` / `--exe <path>` / `--json <file>`; binary is located via `--exe` > `$LUMEN_EXE` > `target/{dev-release,release,debug}/lumen[.exe]`. Journal `docs/perf/startup.md` + committed runs `docs/perf/startup-runs/*.json`. 7 unit tests (arg parsing, cold/warm split, percentiles, JSON round-trip). Invocation: `cargo run -p lumen-bench --release -- --startup`. Limitation: the GUI "window on screen" variant is not deterministically measurable headless — the process start to first frame is taken as proxy; a regression threshold is deferred to PERF-7.
- **Why:** before this task the plan's performance targets (cold start <300 ms, RAM <100 MB for empty tab) were slogans without a baseline. Now regressions as functionality grows can be tracked — each phase must stay within its budget.
- **Baseline (dev profile, samples/page.html 667 B HTML + 300 B CSS, 49 DOM nodes, 7 CSS rules, 18 paint commands, on x86_64 CachyOS):** decode ~2 µs, parse_html ~19 µs, parse_css ~13 µs, layout ~48 µs (dominant), paint ~1 µs, TOTAL ~85 µs. Release build should show similar or better numbers (dev already uses opt-level=3 for deps).
