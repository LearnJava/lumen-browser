- **Done (HTTP/3 slice 100 ‚Äî QUIC connection reuse across requests, [P1] P3-h3, 2026-07-07):** New `h3::client_pool` module ‚Äî `H3ConnectionPool` keyed by `origin_key(host, port)`, storing `RequestDriver<UdpDatagram>` entries. - **Done (HTTP/3 slice 83 ‚Äî real Mozilla root store adapter, [P1] P3-h3, 2026-07-06):** New `h3::mozilla_roots` module ‚Äî the supplier every trust-anchor slice deferred. Slices 75‚Äì76 built `x509_trust_anchor::verify_trust_anchor` and wired it into `ConnectDriver`, but both stated the real anchor list "is the caller's to wire in, in the same later slice" ‚Äî until now the connect loop's trust store had only ever been fed *synthetic* anchors by unit tests. **`mozilla_trust_anchors() -> Vec<OwnedTrustAnchor>`** turns the compiled-in `webpki_roots::TLS_SERVER_ROOTS` ‚Äî the *same* Mozilla list `HttpClient`/`dot` already load into their `rustls::RootCertStore` for the HTTP/1.1 / HTTP/2 path (`crates/network/src/lib.rs:1193`, `crates/network/src/dot.rs:262`) ‚Äî into the `Vec<OwnedTrustAnchor>` shape `ConnectDriver::new` accepts, so both transports terminate their chains at one identical anchor set. **The encoding fixup (the slice's whole subtlety):** `rustls_pki_types::TrustAnchor` stores **both** `subject` and `subject_public_key_info` as bare *values* ‚Äî the contents of their `SEQUENCE` **without** the outer tag/length (a `subject` begins at the first RDN `SET` `0x31`, an SPKI at its inner `AlgorithmIdentifier` `0x30`), `webpki`'s decomposed-storage convention. Every consumer in this sprint wants the *whole* `SEQUENCE` span: `x509_name_chain::certificate_names` returns a certificate's `issuer` as the full outer `Name` `SEQUENCE`, and `x509_spki::parse_subject_public_key_info` strips an outer `0x30` first. So **both** fields are re-wrapped in a `SEQUENCE` header (`wrap_in_sequence`, `0x30 ‚Äñ DER-len ‚Äñ value`) before becoming an `OwnedTrustAnchor`. A root imported without this fixup would match no certificate's `issuer` and its key would never decode ‚Äî a store that silently trusts *nothing*, a failure that *looks* like it verified. An adversarial unit test (`every_public_key_is_a_wrapped_parsable_spki`) caught an earlier draft that wrapped only the subject: it asserts every real root's re-wrapped SPKI decodes via `x509_spki`, which failed hard (`Malformed("SPKI algorithm is not a SEQUENCE")`) when the SPKI was passed verbatim. Per-anchor `nameConstraints` (a third `TrustAnchor` field) are dropped ‚Äî `OwnedTrustAnchor` does not model them yet, no less safe than the existing `rustls` path which delegates that to `webpki`. **Invariants:** pure allocation over compiled-in bytes ‚Äî no clock, no I/O, no fallible parse (source roots are valid DER; re-wrapping a `SEQUENCE` cannot fail); holds no state. No new deps (`webpki-roots` was already a `lumen-network` dependency; this slice is its first h3 consumer). No panics/unwraps in production. +6 unit tests (store non-empty and same length as upstream; every subject is `0x30 ‚Äñ len ‚Äñ raw-value`; every SPKI re-wrapped and parses; `wrap_in_sequence` short- and long-length forms; DER length encoder short/long forms). **Remaining for P3-h3:** the `h3_do_request` dispatch alongside the H1/H2 paths (the connect loop can now be handed `mozilla_trust_anchors()` as its production trust store).
- **Done (HTTP/3 slice 82 ‚Äî X.509 unrecognized-critical-extension rejection, module + connect-loop wiring, [P1] P3-h3, 2026-07-06):** New `h3::x509_critical_ext` module ‚Äî the relying-party leg of RFC 5280 path validation (¬ß4.2, ¬ß6.1.4(n)), the tenth and final certificate gate in the connect loop. Where the eight interpreting walks each *read* one extension, this polices the *set* of extensions each certificate marks `critical` against the fixed set the validator actually processes: `subjectAltName` (2.5.29.17, `x509_hostname`), `keyUsage` (2.5.29.15, `x509_key_usage`), `basicConstraints` (2.5.29.19, `x509_basic_constraints`), and `extendedKeyUsage` (2.5.29.37, `x509_ext_key_usage`). RFC 5280 ¬ß4.2 makes the `critical` flag the issuer's demand that a relying party understand the extension ‚Äî a validator that silently ignored a critical extension it does not process (say a critical `nameConstraints` (2.5.29.30) scoping an intermediate to one domain) would defeat the issuer's intent and let the intermediate mint certificates for any name, so failing closed is the only safe behaviour and every browser does it. **`certificate_has_unrecognized_critical_extension(cert_der) -> Result<bool, CritExtError>`** navigates the `TBSCertificate` by field order to the `[3] EXPLICIT` extensions and, for each `Extension`, reads its `extnID` and the `critical` BOOLEAN (present-and-non-zero = critical, per X.690 ¬ß11.1/¬ß11.5; a DER-illegal present-FALSE encoding is read tolerantly as non-critical); the extension *value* is never decoded. **`verify_no_unknown_critical_extensions(chain: &[&[u8]]) -> Result<(), CritExtWalkError>`** applies the check to *every* certificate in the server's `certificate_list` (RFC 8446 ¬ß4.4.2), leaf and issuers alike (¬ß6.1.4(n) processes each certificate in the path). `CritExtWalkError { EmptyChain, Certificate { index, error }, UnrecognizedCritical { index } }` (Display + Error). **Wiring:** `ConnectDriver::poll` calls it last, after possession, identity, time, all four chain walks, the trust-anchor termination, and the `extendedKeyUsage` leg hold, wrapped as `ConnectError::CriticalExtension`; a certificate marking an unrecognized extension critical abandons the connection **before** the client Finished is flushed. A *non-critical* unrecognized extension is permitted (¬ß4.2). **Invariants:** pure DER parsing over borrowed bytes ‚Äî no clock, no I/O; reuses the sibling `Der` reader shape specialised to this error type. No panics/unwraps in production, no new deps. +15 module unit tests (no-extensions cert; all four recognized criticals pass; non-critical unrecognized passes; critical unrecognized flagged, including after recognized criticals; present-FALSE critical treated non-critical; malformed rejections; the chain walk accepts a recognized-critical chain and a no-extensions chain, rejects an unrecognized critical on the leaf and on an intermediate, rejects an empty chain, reports a malformed cert, error Display) + 2 connect-loop tests (a non-critical unrecognized extension completes the handshake; a critical one aborts with `ConnectError::CriticalExtension(UnrecognizedCritical { index: 0 })` before the client Finished flushes). **Remaining for P3-h3:** the production Mozilla root store and the `h3_do_request` dispatch alongside the H1/H2 paths.
