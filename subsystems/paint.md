- **Offscreen layer pool ‚Äî femtovg (BUG-272 slice 1, 2026-07-07; slice 2, 2026-07-08):** every Push{ClipRoundedRect,ClipPath,Opacity,Filter,Mask,BackdropFilter,BlendMode} draws its subtree into a full-framebuffer offscreen texture. These now come from `FemtovgBackend::layer_pool` (`acquire_layer`/`release_layer`): a layer released on Pop is reused by the next Push in the same frame ‚Äî femtovg executes its command queue strictly in submission order, so overwriting a released layer's pixels is safe where deleting its ImageId mid-frame is not (that is what the `*_pending_delete` after-flush queues are for). Slice 2 added the blend-mode (PA-3) offscreen `src_image_id`: it is only ever used as a render target and read back via `screenshot()`, never GPU-sampled with `Paint::image`, so the pool's `FLIP_Y` sampler flag is a no-op for it ‚Äî same reasoning as the `acquire_layer()` blur destinations in `composite_filter_layer`. **Invariant: layers acquired from the pool must be cleared (`clear_rect` transparent) before drawing, and one-off images with different creation mechanisms (colour-matrix re-upload via `create_image` from CPU pixels, blend-mode CPU-composited result image, filtered backdrop CPU upload) must NOT be released into the pool.** Before pooling, ~150 layer commands per frame held ~750 MB of GPU memory simultaneously (freed only after flush), which the driver retained forever ‚Äî lenta.ru sat at 1.37 GB WS. Peak layer memory is now bounded by nesting depth (cap 8; resize retires the pool via the pending-delete queue). Slice-2 measurement (synthetic 120-element `mix-blend-mode:multiply` page): 1230 MB WS / 976 MB private ‚Üí 637 MB WS / 447 MB private at first paint. **New observation during slice-2 verification: on this same synthetic page, WS/private keep climbing over ~20s of idle (637‚Üí896 MB WS) even with `ControlFlow::Wait`/`WaitUntil` (no continuous Poll loop) ‚Äî cause not yet diagnosed. Likely related to BUG-273 (each blend/filter Pop does flush‚ÜíglReadPixels‚ÜíCPU-blend‚Üíre-upload; the re-uploaded result image is still a one-off, not pooled) if some periodic timer is triggering redraws on this idle page; not observed on lenta.ru (no mix-blend-mode use). Left as an open follow-up, not blocking this slice.**

## Planned: RenderBackend abstraction (ADR-010, Phase 2‚Äì3)

See [ADR-010](../docs/decisions/ADR-010-render-backend-abstraction.md) for full rationale.

### Goal

Replace the single concrete `Renderer` (wgpu) with a `RenderBackend` trait so that
femtovg, vello, wgpu, and the existing cpu_raster are interchangeable at compile time
and testable side-by-side at runtime.

### Stable contract ‚Äî `RenderBackend` trait

```rust
// crates/engine/paint/src/backend.rs
pub trait RenderBackend: Send {
    fn render(&mut self, content: &[DisplayCommand], overlay: &[DisplayCommand],
              scroll_y: f32, scroll_x: f32) -> Result<(), RenderError>;
    fn resize(&mut self, width: u32, height: u32);
    fn set_scale_factor(&mut self, scale: f64);
    fn register_image(&mut self, src: String, image: &Image) -> Result<(), String>;
    fn clear_images(&mut self);
    fn set_font_provider(&mut self, provider: Option<Arc<dyn FontProvider>>);
    // Extended API (RB-4, Phase 1):
    fn viewport_size(&self) -> Size { Size { width: 1024.0, height: 720.0 } }
    fn scale_factor(&self) -> f64 { 1.0 }
    fn preload_curated_fallbacks(&mut self) {}
    fn on_layer_memory_pressure(&mut self, _level: MemoryPressureLevel) {}
    fn screenshot_rgba(&mut self) -> Option<Vec<u8>> { None }  // headless only
}
```

`DisplayCommand` does not change. Layout and shell code never sees GPU-API types.

**RB-4 (2026-06-03) ‚Äî Shell wired to `Box<dyn RenderBackend>`:**
- `Lumen.renderer: Option<Renderer>` ‚Üí `Option<Box<dyn RenderBackend>>`
- `backend_factory::create_backend(window, font_bytes)` reads `LUMEN_BACKEND` env var and creates the right backend (Phase 1: wgpu only; femtovg/vello log a warning and fallback to wgpu)
- `WgpuBackend` implements the 4 new trait methods by delegating to inner `Renderer`
- Shell no longer imports `Renderer` directly (except in `do_print_to_pdf` local scope for `render_print_pages`)

### vello isolation strategy

ALL vello imports are confined to `backends/vello_backend.rs`. The file has two
internal functions: `translate_to_scene(&[DisplayCommand]) -> vello::Scene` and
`submit_scene(scene, renderer, surface)`. When vello releases a breaking API change,
only these two functions change ‚Äî zero impact on trait, shell, or layout code.

### CompareBackend ‚Äî parallel rendering

`CompareBackend { primary, secondary }` renders the same `DisplayCommand` slice with
both backends, calls `screenshot_rgba()` on each, and computes pixel diff percent.
Used in `lumen-driver` compare tests to validate a new backend against the current
reference before promoting it to default.

```bash
# validate vello against femtovg across all 57 snapshot pages
cargo test -p lumen-driver --features compare-femtovg-vello
```

## Gotchas

- **Blend/filter/backdrop-filter layers in the femtovg backend composite on the CPU with a full-framebuffer `glReadPixels` round-trip per layer per frame (BUG-273).** `composite_blend_layer` / `composite_filter_layer` cost ~50‚Äì80 ms each on a 1024√ó720 window; there is no viewport culling and no cross-frame layer cache, so pages with several such elements scroll at single-digit FPS. Diagnose with `LUMEN_FRAME_LOG=1` (per-frame `[frame] paint/total ‚Ä¶` stderr lines, level 2 adds a top-8 breakdown by DisplayCommand variant ‚Äî see `lumen_paint::frame_log_level`) and `python scripts/scroll_perf.py <page>` (scripted wheel-scroll of a live window + frame-time stats).
- **Radial/repeating gradients render per-pixel in the femtovg backend (BUG-085, D2-1).** femtovg's native gradient paint samples a 256-texel LUT (quantizes repeating-stop boundaries) **and is circle-only** (renders an `ellipse` gradient as a circle). `draw_radial_gradient_cpu` / `draw_linear_gradient_cpu` (for repeating) fill a device-resolution RGBA texture via `sample_fvg_stops` and blit it (`Paint::image`, deleted after flush via `gradient_pending_delete`). Radii come from `DrawRadialGradient.radius_x/radius_y`, computed by `lumen_layout::radial_gradient_radii` in the display-list builder; `rx == ry` ‚áí circle, byte-identical to the old path (BUG-221). Smooth non-repeating *linear* gradients stay on the cheaper femtovg-native path. The wgpu `renderer.rs` radial path is unchanged (its own UV-normalized scheme; not the default backend) and ignores `radius_x/radius_y`.
- **Tests in `lumen-paint::display_list` and `lumen-paint::atlas`** are unit tests. `renderer.rs` is visual, no auto-tests ‚Äî verify via `cargo run`. Display list snapshot tests live in `tests/snapshot_tests.rs`.
- **BUG-057 (FIXED 2026-06-03):** wgpu Vulkan double-panic on first render (Windows). Fix: `Backends::DX12` default in `new_async` + `new_headless_async`. `WGPU_BACKEND` env var overrides.
