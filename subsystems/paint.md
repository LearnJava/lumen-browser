- **Offscreen layer pool тАФ femtovg (BUG-272 slice 1, 2026-07-07; slice 2, 2026-07-08):** every Push{ClipRoundedRect,ClipPath,Opacity,Filter,Mask,BackdropFilter,BlendMode} draws its subtree into a full-framebuffer offscreen texture. These now come from `FemtovgBackend::layer_pool` (`acquire_layer`/`release_layer`): a layer released on Pop is reused by the next Push in the same frame тАФ femtovg executes its command queue strictly in submission order, so overwriting a released layer's pixels is safe where deleting its ImageId mid-frame is not (that is what the `*_pending_delete` after-flush queues are for). Slice 2 added the blend-mode (PA-3) offscreen `src_image_id`: it is only ever used as a render target and read back via `screenshot()`, never GPU-sampled with `Paint::image`, so the pool's `FLIP_Y` sampler flag is a no-op for it тАФ same reasoning as the `acquire_layer()` blur destinations in `composite_filter_layer`. **Invariant: layers acquired from the pool must be cleared (`clear_rect` transparent) before drawing, and one-off images with different creation mechanisms (colour-matrix re-upload via `create_image` from CPU pixels, blend-mode CPU-composited result image, filtered backdrop CPU upload) must NOT be released into the pool.** Before pooling, ~150 layer commands per frame held ~750 MB of GPU memory simultaneously (freed only after flush), which the driver retained forever тАФ lenta.ru sat at 1.37 GB WS. Peak layer memory is now bounded by nesting depth (cap 8; resize retires the pool via the pending-delete queue). Slice-2 measurement (synthetic 120-element `mix-blend-mode:multiply` page): 1230 MB WS / 976 MB private тЖТ 637 MB WS / 447 MB private at first paint. **New observation during slice-2 verification: on this same synthetic page, WS/private keep climbing over ~20s of idle (637тЖТ896 MB WS) even with `ControlFlow::Wait`/`WaitUntil` (no continuous Poll loop) тАФ cause not yet diagnosed. Likely related to BUG-273 (each blend/filter Pop does flushтЖТglReadPixelsтЖТCPU-blendтЖТre-upload; the re-uploaded result image is still a one-off, not pooled) if some periodic timer is triggering redraws on this idle page; not observed on lenta.ru (no mix-blend-mode use). Left as an open follow-up, not blocking this slice.**

## Planned: RenderBackend abstraction (ADR-010, Phase 2тАУ3)

See [ADR-010](../docs/decisions/ADR-010-render-backend-abstraction.md) for full rationale.

### Goal

Replace the single concrete `Renderer` (wgpu) with a `RenderBackend` trait so that
femtovg, vello, wgpu, and the existing cpu_raster are interchangeable at compile time
and testable side-by-side at runtime.

### Stable contract тАФ `RenderBackend` trait

```rust
// crates/engine/paint/src/backend.rs
pub trait RenderBackend: Send {
    fn render(&mut self, content: &[DisplayCommand], overlay: &[DisplayCommand],
              scroll_y: f32, scroll_x: f32) -> Result<(), RenderError>;
    fn resize(&mut self, width: u32, height: u32);
    fn set_scale_factor(&mut self, scale: f64);
    fn register_image(&mut self, src: String, image: &Image) -> Result<(), String>;
    fn clear_images(&mut self);
    fn set_font_provider(&mut self, provider: Option<Arc<dyn FontProvider>>);
    // Extended API (RB-4, Phase 1):
    fn viewport_size(&self) -> Size { Size { width: 1024.0, height: 720.0 } }
    fn scale_factor(&self) -> f64 { 1.0 }
    fn preload_curated_fallbacks(&mut self) {}
    fn on_layer_memory_pressure(&mut self, _level: MemoryPressureLevel) {}

```

## Gotchas

- **Blend/filter/backdrop-filter layers in the femtovg backend composite on the CPU with a full-framebuffer `glReadPixels` round-trip per layer per frame (BUG-273).** `composite_blend_layer` / `composite_filter_layer` cost ~50тАУ80 ms each on a 1024├Ч720 window; there is no viewport culling and no cross-frame layer cache, so pages with several such elements scroll at single-digit FPS. Diagnose with `LUMEN_FRAME_LOG=1` (per-frame `[frame] paint/total тАж` stderr lines, level 2 adds a top-8 breakdown by DisplayCommand variant тАФ see `lumen_paint::frame_log_level`) and `python scripts/scroll_perf.py <page>` (scripted wheel-scroll of a live window + frame-time stats).
- **Session frame-time summary (M0.1, ADR-016).** `lumen_paint::FrameStats` accumulates per-frame ms and computes a `FrameSummary` (count + min/p50/p95/p99/max, nearest-rank percentiles; 5 unit tests in `lib.rs`). The shell owns one `FrameStats`, records each frame only when `LUMEN_FRAME_LOG>=1`, and prints `FRAME_SUMMARY тАж` on the `LUMEN_MEM_REPORT` 10 s cadence and once from `ApplicationHandler::exiting`. These are the baseline numbers each later MT-render stage (M1тАУM4) cites before/after.
- **Viewport culling (M0.2, ADR-016).** `DisplayCommand::cull_rect() -> Option<Rect>` returns the document-space AABB of a **self-contained leaf draw** (fills, text, images, gradients, background/cross-fade, borders, `DrawOutline` grown by `offset+width`, SVG fill/stroke/path bounded from their contours/verts with the stroke inflated by `half_width┬╖miterlimit`, layer snapshot, scrollbar trackтИкthumb, box-model overlay) and `None` for **every structural `Push*`/`Pop*`, sticky/scroll marker and `PageBreak`** тАФ those must always run to keep the stack balanced. Each backend maps that box through the current CTM and skips it when the AABB of the four transformed corners falls fully outside the viewport expanded by a **256 CSS px slop**: femtovg `FemtovgBackend::is_command_culled` guards the top of `render_command` (safe inside offscreen clip/filter/mask groups because those FBOs are full-surface with an unchanged transform); wgpu `renderer.rs::leaf_is_offscreen` guards the top of the `render` command loop, using `Mat4::transform_point_2d` and **disabling culling under any non-2D-affine (3D/perspective) transform** (conservative). AABB-of-transformed-corners is a superset of the true footprint, so a command is only ever culled when entirely off-screen тАФ no visible pixel is dropped. The femtovg `[frame]` log gained a `culled N/M leaf` counter (`cull_stats`, reset per frame). 4 unit tests on `cull_rect`. Supersedes the "no viewport culling" note above for leaf draws (offscreen-layer *composite* passes are still uncelled тАФ that stays a BUG-273 follow-up).
- **Transform-first zoom (M0.3, ADR-016).** `RenderBackend::set_preview_scale(f32)` (default no-op) lets the shell scale the *retained* display list without a relayout. femtovg stores it in `preview_scale` and, in `render()`, applies `canvas.scale(s, s)` **before** the `translate(-scroll)` so a document point maps to `s┬╖(p тИТ scroll)` тАФ a zoom anchored at the viewport top-left. `s == 1.0` skips the call (identity). Culling stays correct: `is_command_culled` maps AABBs through `self.canvas.transform()`, which already includes `s`. `set_preview_scale` clamps non-finite/тЙд0 input to `1.0`. `CompareBackend` forwards to both children; the wgpu backend keeps the no-op default (femtovg is the shell default). The shell drives it via `App::begin_zoom_preview` + a 180 ms debounce that lands a single `relayout()`; every relayout resets the backend to 1:1 and re-syncs `laid_out_zoom_factor`.
- **Render-side page offset тАФ no per-frame display-list clone (M0.4, ADR-016).** `RenderBackend::set_page_offset(x, y)` (default no-op) + `supports_page_offset() -> bool` (default `false`). The shell used to copy the *entire* display list into a `PushTransform(translate(page_offset))`-wrapped buffer every `RedrawRequested` тАФ an O(n) deep clone of every `DisplayCommand` on each momentum-scroll frame тАФ solely to shift the page below the tab bar (`TAB_BAR_HEIGHT`) and right of a left-docked sidebar. femtovg now stores the fixed offset in `page_offset` and applies it as `canvas.translate(offset)` **right after** the `translate(-scroll)` in `render()`, so the CTM stays `scale(preview)┬╖translate(-scroll)┬╖translate(offset)` тАФ identical to the old wrapper, so sticky (uses the true `scroll`) and zoom-preview are unchanged; culling stays correct (`is_command_culled` maps AABBs through the live CTM, which includes the offset). `set_page_offset` clamps non-finite input to `(0,0)`. femtovg overrides `supports_page_offset()` to `true`; the shell then renders the retained display list **by reference** (fast path). Backends that don't support it (wgpu/vello/cpu window path, `CompareBackend` forwards) keep the `PushTransform`-wrapper path, and the wrapper also survives the rare devtools-inspector-overlay frame (it must ride inside the page transform). Trait-default contract test in `backend.rs`.
- **Split frame fingerprint тАФ content hash тЯВ scroll/offset (M0.5, ADR-016).** `lumen_paint::hash_content(content, surface_w, surface_h) -> u64` folds **only** the page-content commands + surface size into the hash (scroll and the fixed page offset excluded), reusing `hash_display_list`'s allocation-free `HashFmt`тЖТ`DefaultHasher` Debug-streaming (same totality guarantee: a new variant/field can never silently collide). `FrameFingerprint { content_hash, scroll: (f32,f32), offset: (f32,f32) }` keeps the two offsets as raw copyable values, and `FrameFingerprint::delta_from(&prev) -> FrameDelta` classifies a frame as `Identical` (nothing moved), `OffsetOnly` (same content, new scroll/page offset тАФ the M3 blit trigger) or `ContentChanged` (a command edit or a surface resize, which the content hash folds in, always wins). **Overlay** commands are deliberately excluded from the hash тАФ the scrollbar thumb is rebuilt from `scroll_y` each frame and would otherwise make every scroll look like a content change. The shell keeps `last_frame_fp` and, **only under `LUMEN_FRAME_LOG`**, logs `[frame] delta {тАж}` after each present so the scroll-vs-content frame mix is measurable; normal runs compute nothing (the content hash is O(n)). No skip/blit is wired yet тАФ that is M3. Split-view bakes scroll into the display list, so the classification is skipped there. 4 unit tests in `display_list.rs`.
- **GL-context handoff to the render thread (M1.2, ADR-016).** `FemtovgBackend` stores its glutin context as `Option<GlContextState>` where `GlContextState` is `Current(PossiblyCurrentContext)` | `NotCurrent(NotCurrentContext)` тАФ a two-state enum because both glutin transitions (`make_current` / `make_not_current`) consume the context by value. Inherent `detach_gl_context()` calls `make_not_current` (idempotent; leaves it movable across threads) and `attach_gl_context()` calls `make_current(&gl_surface)` (idempotent; binds to the calling thread). `swap_buffers`/`resize` fetch the context via `current_ctx() -> Result<&PossiblyCurrentContext, RenderError>`, which errors if the context is detached (i.e. used off the owning thread before `attach`). This is the femtovg side of ADR-016's create-on-main тЖТ detach тЖТ move тЖТ attach-on-render-thread handoff; the shell drives it from `backend_factory::create_threaded_femtovg`. Verified on Windows with `LUMEN_RENDER_THREAD=1` (present off-main, no `make_current` errors). The manual `unsafe impl Send for FemtovgBackend` (guarded by `femtovg_backend_is_send`) is what lets the concrete backend move to the render thread.
- **Render-side momentum trait hooks (M1.3, ADR-016).** `RenderBackend` gains two no-op-default methods тАФ `start_render_momentum(vel_y, vel_x, max_scroll_y, max_scroll_x)` and `stop_render_momentum()`. Single-threaded backends (femtovg/wgpu/vello/cpu) ignore them (the shell keeps ticking momentum itself), so nothing changes when the render thread is off. Only `ThreadedRenderBackend` (in `lumen-shell`) implements them, letting the render thread continue kinetic scroll while the UI thread stalls тАФ see `subsystems/shell.md` (M1.3). Physics lives in `lumen-shell::momentum_anim` (stateless `velocity_at` / `displacement_since`), not in `lumen-paint`.
- **Cross-thread frame-log tags (M1.4, ADR-016).** A third no-op-default `RenderBackend` method `set_frame_commit_id(commit_id, self_tick)` lets the render thread annotate the next present. The femtovg backend stores it in `frame_commit_id: Option<(u64, bool)>` (consumed each `render()`, reset to `None` after) and appends `[thr <name>[ commit <id>[ self-tick]]]` to its `[frame] paint тАж` line, where `<name>` is `std::thread::current().name()` (`lumen-render` off the UI thread, else `main`). Self-tick presents тАФ momentum re-drawn by the render thread *while the UI thread is stalled* (M1.3) тАФ carry the retained frame's commit id + a `self-tick` marker, so the log now proves presentation continued during a stall (the M1 acceptance evidence). The single-threaded path never calls the setter тЖТ `None` тЖТ only the `[thr main]` tag is added. No new deps, no `unsafe`. The shell side (who sets it) is in `subsystems/shell.md` (M1.4).
- **Labeled percentile summary (M2.0, ADR-016).** `FrameSummary::display_with(label) -> LabeledSummary` prints the same `count/min/p50/p95/p99/max` tail under an arbitrary prefix instead of the hard-coded `FRAME_SUMMARY`; the default `Display for FrameSummary` now delegates to `display_with("FRAME_SUMMARY")` so the format string lives in one place. This lets the shell print `ENGINE_SUMMARY тАж` (UI-thread relayout cost, M2.0) from the same tested percentile math without duplicating it тАФ the before/after baseline M2's engine-thread move improves. 1 unit test (`display_with_uses_custom_label_same_numbers`). The shell side (what fills the engine histogram) is in `subsystems/shell.md` (M2.0).
- **Blit-vs-repaint scroll decision brain (M3.0, ADR-016).** `scroll_cache.rs` тАФ `ScrollCache` + `ScrollFramePlan { Blit { src: (f32,f32) }, Repaint { origin, size } }`, the pure decision logic the M3 compositor-scroll fast path builds on. A `ScrollCache` tracks which document-space band the retained content surface holds (`origin` + `covered` size) and the `content_hash` it was rastered for; `plan(content_hash, scroll, viewport)` returns `Blit { src = scroll тИТ origin }` **only** when the hash matches *and* the visible viewport rect is fully inside the cached band (`covers()` containment test), else `Repaint` with a band re-centered on the viewport plus `DEFAULT_OVERSCAN` (512 CSS px) each side, origin clamped at `(0,0)`. `record_repaint()` re-seats the band after the backend actually rasters; `invalidate()` drops it on navigation/resize. The containment **range** check (never an equality/quantized-bucket test on the scroll value) is the deliberate fix for the 2026-07-10 experiment failure (a 20 px-bucketed full-surface cache missed on every 120 px wheel step тЖТ two full renders/frame, 30├Ч slower): inside the overscan band any sub-pixel scroll is a blit. Pure тАФ no GPU state, no rasterization; the femtovg content surface (`layer_pool`/`acquire_layer`) lands in M3.2. 12 unit tests. No new deps, no `unsafe`.
  - **M3.1 тАФ incremental band exposure.** `plan()` gains a third variant `BlitAndExpose { origin, size, retained: Rect, prev_origin, expose: [Option<Rect>; 4] }`. When the viewport leaves the band but the same content's re-centered band still overlaps the old one, it returns the overlap (`retained`, blit from the previous surface) plus up to four `expose` strips (top/bottom/left/right) that the old surface did not cover тАФ `retained` + `expose` tile the new band exactly, so the backend re-rasters only the newly revealed pixels instead of the whole band. A stale hash or a disjoint jump still returns a full `Repaint`. Still pure (the plan carries `prev_origin`/`origin` so the backend can place the blit; GL wiring is M3.2). 6 new/updated tests incl. a tiling invariant (`retained + expose == band`).
  - **M3.2.0 тАФ shell wiring (measurement, no GL).** The shell owns a `scroll_cache: ScrollCache` (`default_overscan`), invalidated in `reset_to_blank_tab` on navigation. New `ScrollFramePlan::label() -> &'static str` (`"blit"`/`"blit+expose"`/`"repaint"`) + one unit test. Dead by default тАФ driven only under `LUMEN_FRAME_LOG`, so normal runs are byte-identical.
