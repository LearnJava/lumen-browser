# Lumen Perf Audit — аудит производительности на реальных сайтах

Цель: не разовые ручные находки, а воспроизводимый цикл
**прогон → статистика → сравнение → журнал → баги**.

**Перед любым cargo** (Windows + Git Bash):
```bash
export PATH="/c/Users/konstantin/.cargo/bin:$PATH"
```

## Шаг 0. Ветка

Работай в worktree `p2-perf-audit-<YYYYMMDD>` (изменения: `docs/perf/*`,
`BUGS.md`, `bugs/BUG-*.md`). Движок НЕ трогать — аудит только наблюдает.

## Шаг 1. Сборка

```bash
cargo build -p lumen-shell --profile dev-release     # НЕ --release (в 2-3 раза дольше)
```
Бинарь: `target/dev-release/lumen.exe` (скрипт найдёт сам; иначе `$LUMEN_EXE`).

## Шаг 2. Прогон (в фоне; на экране появится живое окно)

```bash
python scripts/perf_audit.py > .tmp/perf-audit-run.log 2>&1
```
Режим по умолчанию — **живой**: одно GUI-окно на весь корпус, каждый сайт
открывается в НОВОЙ вкладке (MCP `new_tab`), после загрузки — пауза `--dwell`
(по умолчанию 3 с; пользователь смотрит рендер) и прокрутка вниз/вверх.
Метрики: время до document_ready, кумулятивная RAM процесса (тек/пик) по мере
накопления вкладок, JS-ошибки (console + stderr-дельта по сайту), CPU-скриншот.
Пользователь наблюдает окно — предупреди его перед запуском и не перекрывай
окно другими действиями. Запускай в фоне и опрашивай лог (`[N/14] slug ...`).
Быстрая проверка: `--only example`. Результаты: `.tmp/perf-audit/<stamp>/`.

`--phases` — headless-режим разложения по фазам (три замера на сайт:
source/layout/screenshot) — используй для точечного разбора медленного сайта:
`python scripts/perf_audit.py --phases --only lenta`.

Если есть прошлый прогон — добавь `--compare docs/perf/runs/<последний>.json`
(колонка Δ%, ⚠ при замедлении >20%; сравнивай live с live, phases с phases).

## Шаг 3. Анализ

По summary.md + results.json для каждого сайта ответь:

1. **Статус**: OK / TIMEOUT / FAIL. Известные ограничения ≠ находки:
   403-антибот (stackoverflow, crates, ria — TLS-фингерпринт rustls), 500 docs.rs.
   Сверься с журналом — новое падение = находка.
2. **Живой режим**: `ready_s` (время до document_ready), рост `cur_mb` от
   вкладки к вкладке (аномальный скачок >300 МБ на сайт — находка), `restarted`
   (окно умерло/зависло — всегда находка), `own_tab=False` (new_tab не сработал).
   Доминирующую фазу медленного сайта уточняй через `--phases --only <slug>`:
   paint → CPU-растеризация (известный класс, BUG-272 — не заводи дубликат,
   дополни цифрами); style_layout → `profile_top` (LUMEN_PROFILE_TREE);
   net_parse → сеть/парсер.
3. **Регрессии** vs прошлый прогон: Δ > +20% на той же машине = находка;
   меньше — шум сети/машины, только зафиксируй в журнале.
4. **error_lines** — недекодированные форматы, паники, DNS-ошибки,
   отсутствующие JS API («X is not defined / is not a function» — валит
   site-скрипты целиком и маскирует остальное).
5. **RAM/CPU** (`peak_mb`/`cpu_s`, Windows): пик рабочей памяти процесса
   на стадии screenshot >500 МБ — находка (контекст: BUG-272 давал 1371 МБ
   на lenta.ru до среза 1); cpu_s ≈ wall_s стадии — чистый CPU-bound,
   cpu_s << wall_s — ожидание сети/IO.
6. Для подозрительных сайтов копай точечно: `LUMEN_PROFILE_TREE=1` /
   `LUMEN_FRAME_LOG=2` / `LUMEN_MEM_REPORT=1` вручную (docs/automation.md).

### Что аудит НЕ покрывает (не выдумывай цифры)

- **Числовая плавность скролла** — живой прогон скроллит каждую вкладку
  (визуальная проверка), но FPS/frame-time не меряет; цифры — отдельно
  `python scripts/scroll_perf.py` (p95/p99 — задача PERF-3).
- **GPU-метрики** — окно рендерит wgpu-дефолтом (визуально это видно), но
  GPU-время/память не собираются (PERF-3/PERF-5); скриншоты в отчёте — CPU-путь
  (BUG-221), могут отличаться от того, что в окне.
- **Долгая сессия/утечки** — LUMEN_MEM_REPORT=1 вручную (PERF-5); кумулятивная
  RAM по 14 вкладкам из живого прогона — грубый прокси, не замена.

## Шаг 4. Журнал (коммитится)

1. Скопируй results.json → `docs/perf/runs/<YYYY-MM-DD>.json`.
2. Допиши секцию в `docs/perf/journal.md` (шаблон в шапке файла): дата, коммит
   движка, машина, сводная таблица, сравнение, находки, заведённые баги.
   Скриншоты НЕ коммитить (правило graphic_tests действует и здесь).

## Шаг 5. Баги

На каждую **новую** находку с внятной уликой — баг по протоколу BUGS.md:
один баг = одна первопричина (не «сайт X медленный», а «фаза Y стоит Z секунд
из-за W»), сначала `grep -i <компонент/сайт> BUGS.md` против дубликатов;
номер = максимальный существующий BUG-NNN + 1. Файл `bugs/BUG-NNN-OPEN.md`:
симптом, цифры из прогона, путь к stderr-логу, гипотеза первопричины,
ссылка на журнал. Плюс строка в таблицу BUGS.md. Перф-улучшения крупнее
бага (новая оптимизация) — не баг, а предложение строки в дорожку PERF
(ROADMAP.md) в отчёте пользователю.

## Шаг 6. Завершение

Коммит (docs/perf/* + BUGS.md + bugs/*) → merge `--no-ff` в main → удалить
ветку и worktree (7 шагов, docs/git-workflow.md). В отчёте пользователю:
таблица итогов, топ-3 находки, список заведённых багов, динамика vs прошлый
прогон.

## Ограничения

- Цифры зависят от машины и сети: сравнивай только прогоны одной машины;
  колебания сети до ±20% — норма.
- Живой режим: RAM кумулятивная (вкладки не закрываются — это намеренно,
  сессионное поведение), поэтому per-site вклад = дельта к предыдущей строке.
- `--phases`: скриншот-стадия — CPU-путь рендера (не GPU-окно, BUG-221);
  три стадии повторяют работу друг друга — производные фазы приближённые
  (сеть могла закэшироваться между стадиями).
