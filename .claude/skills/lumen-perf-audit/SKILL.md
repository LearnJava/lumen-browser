# Lumen Perf Audit — аудит производительности на реальных сайтах

Цель: не разовые ручные находки, а воспроизводимый цикл
**прогон → статистика → сравнение → журнал → баги**.

**Перед любым cargo** (Windows + Git Bash):
```bash
export PATH="/c/Users/konstantin/.cargo/bin:$PATH"
```

## Шаг 0. Ветка

Работай в worktree `p2-perf-audit-<YYYYMMDD>` (изменения: `docs/perf/*`,
`BUGS.md`, `bugs/BUG-*.md`). Движок НЕ трогать — аудит только наблюдает.

## Шаг 1. Сборка

```bash
cargo build -p lumen-shell --profile dev-release     # НЕ --release (в 2-3 раза дольше)
```
Бинарь: `target/dev-release/lumen.exe` (скрипт найдёт сам; иначе `$LUMEN_EXE`).

## Шаг 2. Прогон (в фоне, 15–40 минут)

```bash
python scripts/perf_audit.py > .tmp/perf-audit-run.log 2>&1
```
Запускай в фоне и опрашивай лог — в нём по строке на сайт
(`[N/14] slug ... OK total=…s dominant=…`). Быстрая проверка харнесса перед
полным прогоном: `--only example`. Результаты: `.tmp/perf-audit/<stamp>/`
(results.json, summary.md, скриншоты, stderr-логи по стадиям).

Если есть прошлый прогон — добавь `--compare docs/perf/runs/<последний>.json`
(колонка Δ%, ⚠ при замедлении >20%).

## Шаг 3. Анализ

По summary.md + results.json для каждого сайта ответь:

1. **Статус**: OK / TIMEOUT / FAIL. Известные ограничения ≠ находки:
   403-антибот (stackoverflow, crates, ria — TLS-фингерпринт rustls), 500 docs.rs.
   Сверься с журналом — новое падение = находка.
2. **Доминирующая фаза** (`dominant`): paint → CPU-растеризация (известный
   класс, см. BUG-272/BUGS.md — не заводи дубликат, дополни существующий баг
   цифрами); style_layout → смотри `profile_top` (дерево LUMEN_PROFILE_TREE);
   net_parse → сеть/парсер.
3. **Регрессии** vs прошлый прогон: Δ > +20% на той же машине = находка;
   меньше — шум сети/машины, только зафиксируй в журнале.
4. **error_lines** — недекодированные форматы, паники, DNS-ошибки.
5. Для подозрительных сайтов копай точечно: `LUMEN_PROFILE_TREE=1` /
   `LUMEN_FRAME_LOG=2` / `LUMEN_MEM_REPORT=1` вручную (docs/automation.md).

## Шаг 4. Журнал (коммитится)

1. Скопируй results.json → `docs/perf/runs/<YYYY-MM-DD>.json`.
2. Допиши секцию в `docs/perf/journal.md` (шаблон в шапке файла): дата, коммит
   движка, машина, сводная таблица, сравнение, находки, заведённые баги.
   Скриншоты НЕ коммитить (правило graphic_tests действует и здесь).

## Шаг 5. Баги

На каждую **новую** находку с внятной уликой — баг по протоколу BUGS.md:
один баг = одна первопричина (не «сайт X медленный», а «фаза Y стоит Z секунд
из-за W»), сначала `grep -i <компонент/сайт> BUGS.md` против дубликатов;
номер = максимальный существующий BUG-NNN + 1. Файл `bugs/BUG-NNN-OPEN.md`:
симптом, цифры из прогона, путь к stderr-логу, гипотеза первопричины,
ссылка на журнал. Плюс строка в таблицу BUGS.md. Перф-улучшения крупнее
бага (новая оптимизация) — не баг, а предложение строки в дорожку PERF
(ROADMAP.md) в отчёте пользователю.

## Шаг 6. Завершение

Коммит (docs/perf/* + BUGS.md + bugs/*) → merge `--no-ff` в main → удалить
ветку и worktree (7 шагов, docs/git-workflow.md). В отчёте пользователю:
таблица итогов, топ-3 находки, список заведённых багов, динамика vs прошлый
прогон.

## Ограничения

- Цифры зависят от машины и сети: сравнивай только прогоны одной машины;
  колебания сети до ±20% — норма.
- `--screenshot` — CPU-путь рендера (не GPU-окно, BUG-221): paint-цифры
  характеризуют CPU-растеризацию. Это намеренно: детерминированнее и
  чувствительнее к тяжёлым display list.
- Три стадии повторяют работу друг друга — производные фазы приближённые
  (сеть могла закэшироваться между стадиями).
