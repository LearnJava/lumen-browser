# Lumen Health Check — ревизия здоровья кодовой базы (роль P5)

$ARGUMENTS — таргет свипа. Допустимо: `full` (по умолчанию), `clippy`, `stubs`,
`branches`, `docs`, `deps`, `dupes`. Если не передан — выполни `full`.

**Перед любым cargo** (Windows + Git Bash):
```bash
export PATH="/c/Users/konstantin/.cargo/bin:$PATH"
```

Принцип роли: **аудит + безопасная механическая чистка + заведение задач.**
Крупный рефакторинг, меняющий API/поведение, НЕ делать — оформить пунктом в
`STATUS-P1.md`/`STATUS-P2.md`/`STATUS-P4.md` или строкой `OPEN` в `BUGS.md`.

---

## target: `clippy`

Единственное санкционированное использование `--workspace` (P5-исключение из
правила «всегда -p»). Полный свип ловит конфликты, которые per-crate проверки
скрывают после мержа нескольких сессий.

```bash
cargo clippy --workspace --all-targets -- -D warnings
```

Отчёт: только `error[...]` блоки (message + file:line + help). `warning[...]`
блоки пропустить. Тривиальные фиксы (неиспользованный импорт, лишний `&`) в
**своём** крейте можно сделать сразу; всё остальное — задача владельцу крейта.

## target: `stubs`

Аудит заглушек и висящих хэндофов.

```bash
grep -rn "todo!()\|unimplemented!()\|unreachable!()" crates/ --include=*.rs
grep -rn "// CSS:" crates/ --include=*.rs
grep "OPEN" BUGS.md
```

- `todo!()`/`unimplemented!()` в production-коде — потенциальная паника. Каждую
  оценить: ожидаемая заглушка (стаб для P4) или забытый хвост → задача владельцу.
- `// CSS:` — хэндоф для P4. Сверь с «Needs wiring» в `STATUS-P4.md`: есть ли там
  соответствующий пункт. Висящий комментарий без пункта → добавить пункт в P4.
- `OPEN` в BUGS.md — напомнить, что висит на P3.

---

## target: `branches`

Чистка веток и worktree. **Только безопасные удаления.**

```bash
git branch --merged main          # влиты в main — безопасно удалять
git branch -vv | grep ': gone]'   # upstream удалён
git worktree list                 # осиротевшие / зомби-worktree
```

Безопасная чистка (делать сразу, в своём worktree/коммите):
```bash
git worktree prune                              # убрать записи о пропавших worktree
git branch -d <влитая-ветка>                    # ТОЛЬКО для --merged
git worktree remove .claude/worktrees/<name>    # для пустых/завершённых
```

**Запрещено:** `git branch -D` (force) для невлитых веток, удаление чужих
активных worktree. Если worktree чужой и непустой — оставить, сообщить пользователю.
Зомби-worktree (path не совпадает с веткой) — см. `.claude/docs/zombie-worktree.md`,
не форсить.

## target: `docs`

Дрейф автогенерируемых доков и статус-маркеров.

```bash
python scripts/gen_symbols.py
git diff --stat SYMBOLS.md
```

- Если `SYMBOLS.md` изменился — значит он был устаревшим. Закоммить регенерацию
  отдельным коммитом `chore: регенерировать SYMBOLS.md`.
- Сверь статус-маркеры в `lumen-plan.md` (✅🟡⬜) с реальностью кода для недавно
  закрытых задач (по `git log --oneline -20`). Расхождение → поправить маркер.
- Проверь, что `STATUS-PN.md` «Recent» не противоречит `git log`.

---

## target: `deps`

Аудит зависимостей (политика «No new dep without justification», CLAUDE.md).

```bash
cargo tree -d                     # дублирующиеся версии одной крейт-зависимости
```

- Дубли версий — кандидат на унификацию (задача, не правка P5).
- Сверь свежие `[dependencies]` в `git log -p -- '**/Cargo.toml'` (последние
  коммиты): есть ли в теле коммита «Why this dependency: …». Нет — отметить как
  нарушение политики, сообщить пользователю.

---

## target: `dupes`

Поиск дублирования от параллельных сессий (P1/P2/P4 трогают пересекающиеся файлы).
Дорогой свип — в `full` не входит, только по запросу.

- `ComputedStyle` (lumen-layout/src/style.rs): поля с похожим смыслом, добавленные
  разными сессиями. Грепни родственные имена:
  ```bash
  grep -n "pub " crates/engine/layout/src/style.rs | grep -iE "<тема>"
  ```
- Похожие функции в `box_tree.rs` / `display_list.rs`, реализованные дважды.
- Находки оформи пунктом «рефакторинг: объединить X и Y» в `STATUS-P1.md`/`P2.md`.
  P5 не сливает поля сам — это меняет поведение, делает владелец.

---

## target: `full` (по умолчанию)

Запусти по очереди: `clippy` → `stubs` → `branches` → `docs` → `deps`.
`dupes` НЕ включать (дорогой, по отдельному запросу).

Выдай **единый сводный отчёт**:

```
## Health check <YYYY-MM-DD>

### clippy        <OK | N ошибок>
### stubs         <N todo!, M висящих // CSS!, K OPEN-багов>
### branches      <N влитых к удалению, M зомби-worktree>
### docs          <SYMBOLS.md актуален? | дрейф маркеров: …>
### deps          <дубли версий: … | недокументированные deps: …>

### Сделано безопасно
- <удалённые ветки, регенерация SYMBOLS.md, тривиальные clippy-фиксы>

### Заведено задач
- BUGS.md: BUG-NNN …
- STATUS-PN.md «Next»: …
- STATUS-P4.md «Needs wiring»: …
```

После отчёта спроси пользователя, какие из заведённых задач оформлять/чинить
дальше — сам крупный рефакторинг не начинай.

---

## Границы (важно)

- **НЕ** пиши фичи (P1/P2), **НЕ** чини визуальные баги (P3), **НЕ** реализуй
  CSS-свойства (P4). P5 = гигиена.
- Код правь только в своём крейте и только тривиальный clippy-фикс. Всё, что
  меняет API/поведение — задача владельцу.
- Удаляй только `--merged` ветки. Никакого force.
- Коммиты — в worktree `.claude/worktrees/p5-*`, ветка `p5-health-<дата>`.
