# ADR-013: WASM Plugin Sandbox — архитектура плагинной системы

## Status

Accepted

## Date

2026-06-18

## Context

Lumen — собственный движок без унаследованной extension API. Это возможность спроектировать плагинную систему с нуля, избежав проблем WebExtension (JS-only, медленно, широкий DOM-доступ) и NPAPI (полный доступ к процессу, нет изоляции).

Требования, установленные в ходе проектирования:

1. Сторонний плагин не должен иметь возможности навредить ОС, браузеру или другим плагинам.
2. Плагин работает только в sandbox — данные текут внутрь, не наружу.
3. OS-данные (clipboard, файловая система, IP, шрифты, env) недоступны плагинам.
4. Архитектура должна допускать постепенное расширение capability в будущем без переработки sandbox.

## Decision

**Плагины компилируются в `.wasm` и выполняются через `wasmtime` в полностью изолированной среде.**

### Модель безопасности: inbound-only

```
ОС  ←──────────────────────────  плагин не видит ОС
 │
 │ (браузер использует ОС для своей работы)
 ▼
Браузер core
 │
 │ Plugin API: только синтезированные веб-данные
 ▼
Плагин (WASM sandbox)
 │
 │ команды браузеру (suggest_*)
 ▼
Браузер валидирует → применяет
```

Плагин никогда не действует напрямую. Он запрашивает данные у браузера и предлагает браузеру выполнить действие. Браузер — единственный исполнитель.

### Три уровня контроля

**Уровень 1 — Lumen определяет потолок.** Плагин может вызвать только функции из `host imports`. Всё что не задекларировано в imports — физически не существует в WASM-среде плагина. Никакие capability пользователя не могут поднять плагин выше этого потолка.

**Уровень 2 — Классы доверия.**

| Класс | Условие | Доступные capability |
|---|---|---|
| Built-in | Скомпилирован в Lumen | Не применимо — часть браузера |
| Verified | Подписан командой Lumen | Любые из Plugin API |
| Community | Подписан автором (не Lumen) | Безопасные: `dom.write`, `storage.rw`, `ui.panel` |
| Unknown | Без подписи / локальный | Только `dom.write` на текущей вкладке |

**Уровень 3 — Пользователь одобряет в рамках класса.** Пользователь видит список запрошенных capability при установке и подтверждает каждую. Capability вне класса плагина недоступны даже при явном одобрении.

### Manifest плагина

```toml
[plugin]
name    = "Dark Reader"
version = "1.0.0"
author  = "ed25519:abc123..."

[capabilities]
dom.write = true
# net.fetch не указан → физически недоступен
```

### Host imports (полный список API)

Плагин может вызвать только:

```
request_page_text()           → текст страницы
request_current_url()         → URL (без query-параметров с токенами)
request_element(id)           → содержимое DOM-элемента
request_fetch(url, headers)   → HTTP-запрос через браузер, ответ в плагин
read_own_storage(key)         → изолированный storage плагина
suggest_set_text(id, text)    → браузер валидирует id, применяет
suggest_inject_css(rules)     → браузер валидирует CSS, применяет
suggest_show_panel(html)      → браузер рисует панель в своей UI-зоне
write_own_storage(key, value) → пишет только в изолированный storage плагина
```

### OS-данные заблокированы на Барьере 2

Следующие данные доступны браузеру для его работы, но **не передаются в Plugin API ни при каких условиях** (в текущей версии):

- IP-адрес, сетевые интерфейсы
- Пути к файлам, filesystem
- Clipboard
- Список системных шрифтов
- Переменные среды, username, hostname
- Разрешение/размер экрана (fingerprinting)
- Системные HTTP-заголовки

### Лимиты sandbox

- Memory: 64 MB per plugin instance
- CPU: watchdog 100 ms per host-call; зависший плагин убивается, браузер продолжает работу
- Изоляция storage: каждый плагин имеет отдельный SQLite namespace, недоступный другим плагинам

### Процессная изоляция для Community/Unknown

Плагины классов Community и Unknown запускаются в отдельном дочернем процессе. Взаимодействие только через IPC. Даже 0-day в `wasmtime` не даёт доступа к основному процессу браузера.

### Реализация

- Runtime: `wasmtime` (Bytecode Alliance, production-ready)
- Manifest: TOML + Ed25519 подпись
- Plugin directory: `<exe_dir>/plugins/*.wasm`
- Загрузка: при старте браузера + hot-reload при добавлении файла

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| WebExtension API (JS) | JS-only, медленно, широкий DOM-доступ без изоляции, унаследованные проблемы Chrome/Firefox |
| Native plugins (.dll/.so) | Полный доступ к процессу, нет sandbox, платформозависимо, опасно (NPAPI legacy) |
| Lua/Python scripting | Интерпретируемые, медленнее WASM, нет готовой capability-модели, ещё одна зависимость |
| Встроенные возможности без плагинов | Не масштабируется: каждая фича = перекомпиляция браузера, нет пользовательских расширений |

## Consequences

- **Positive:** физическая изоляция без политик; WASM компилируется нативно (~90% скорости); плагины можно писать на любом языке (Rust, C, Go → WASM); архитектура уникальна среди desktop-браузеров
- **Negative / trade-offs:** `wasmtime` добавляет ~5–8 MB к бинарю; первая загрузка `.wasm` = JIT-компиляция (секунды, кэшируется); Plugin API нужно проектировать стабильно — ломать обратную совместимость дорого
- **Future:** когда OS-данные понадобятся плагинам (например, clipboard для менеджера паролей) — добавить соответствующий capability с явным фильтром и классом доверия не ниже Verified; менять sandbox не потребуется
