# ADR-006: Automation API as a first-class engine surface

## Status

Accepted

## Date

2026-05-27

## Context

Browser automation today (Selenium, Playwright, Puppeteer, Cypress) is built **around** browsers as an afterthought, via debug-protocol pretzels (CDP / WebDriver Classic / WebDriver BiDi) and synthetic JS events. Industry evidence (May 2026):

- **Selenium**: 70% of testing budget goes to maintenance; XPath/CSS locators rot on every DOM refactor; HTTP round-trip per action is slow; `StaleElementReferenceException` is industry-folklore.
- **CDP**: not designed for automation, no API stability across Chrome versions, Chromium-only, concurrency conflicts with DevTools.
- **WebDriver Classic**: dying — HTTP request-response model superseded by BiDi.
- **WebDriver BiDi (W3C, in progress 2026)**: cross-browser, WebSocket bidirectional, but still has gaps (no response body, no locale/timezone emulation, expensive interception, no per-context UA, no viewport-before-popup, no preload-script per context, no cookie change events).
- **AI agents** (Claude Computer Use, OpenAI Operator/CUA, Browser Use, Manus) are the fastest-growing consumer of browser APIs. They don't need DOM selectors — they need: screenshot, native input (mouse/keyboard at OS level, not synthetic events), accessibility tree, deterministic rendering. Per a16z, 25-35% of corporate web traffic will be agent-generated by end of 2026.
- **MCP** (Model Context Protocol, Anthropic, Nov 2024) and **WebMCP** (Google, Chrome Canary, Feb 2026) are the emerging standard for AI ↔ tool communication.
- **Accessibility tree** as automation locator surface (Playwright's `getByRole()`) survives DOM refactors and doubles as a11y testing. Recent research (arXiv 2603.20358, March 2026) demonstrates 100% pass-rate self-healing test framework on a11y-tree extraction without LLM API costs.
- **Lightpanda** (open-source Zig+Rust headless, May 2025) proved that a purpose-built browser delivers 11× speed and 8× memory reduction vs headless Chrome for automation — but it has no rendering, no a11y tree, no visual verification. That is a slot Lumen can occupy.

Lumen has a unique architectural opportunity: existing browsers cannot retrofit automation as first-class because their automation is wired through 20-year-old debug protocols. Lumen has no legacy. The engine itself can be the automation surface.

## Decision

Treat automation as a **first-class engine surface**, not a debug-protocol add-on. Concretely:

1. **Single internal `BrowserSession` trait** in `lumen-driver` crate. All automation consumers (in-process tests, MCP server, BiDi server, embed users) call the same trait. The trait surface is engine-native, not protocol-shaped.

2. **In-process Rust API as the primary surface.** `lumen-shell` becomes the first client of the trait; tests, agents, and embed-users are equals, not second-class.

3. **Accessibility tree as primary locator surface.** DOM selectors (CSS / XPath) are supported as fallback. Role+name queries (`get_by_role("button", name = "Submit")`) are recommended. The a11y tree is built by the layout engine, not by a plugin.

4. **Native input injection at the shell layer**, not synthetic JS events. Mouse and keyboard events enter the event loop the same way OS events do, indistinguishable from a user (and indistinguishable from each other for testing purposes).

5. **Auto-wait inside the engine, not in the client.** `session.click(...)` blocks (or returns a future) until the target is actionable (visible, stable layout, not disabled). Implemented on layout/network/JS-idle tick, not as a retry-loop in SDK.

6. **Per-context isolation by default.** Each `BrowserSession` is an isolated context (cookies, storage, cache, viewport, user-agent, fingerprint). Tab is a detail of context, not the unit of isolation.

7. **Deterministic mode is built-in.** Frozen clock, frozen RNG seed, frozen fingerprint profile, bundled-only fonts — configurable on session creation. Repeatable tests across machines and OSes without external setup.

8. **Three transport adapters layered over the trait:**
   - **`lumen-driver`** (in-process Rust crate) — primary, for own `graphic_tests/` and embed users.
   - **`lumen-mcp-server`** — Model Context Protocol over stdio/socket, for Claude/GPT/Browser Use/local LLM agents. Resources: `screenshot`, `a11y_tree`, `layout`, `console`, `network`. Tools: `click`, `type`, `scroll`, `navigate`, `wait`, `eval`.
   - **`lumen-bidi-server`** — WebDriver BiDi subset over WebSocket, for Playwright/Selenium/Cypress.

9. **Ship the BiDi gaps as built-in.** Lumen's BiDi adapter exposes from day one what the W3C spec is still drafting: full response body access, locale/timezone/offline emulation, per-context user-agent, viewport before navigation, preload script per context, full download lifecycle, cookie change events, per-origin storage clear. These are cheap because the engine controls its own stack.

10. **Drop legacy protocols as goals:**
    - **WebDriver Classic** is not implemented.
    - **CDP compatibility is optional and last.** A `lumen-cdp-shim` adapter may be added later if there is real demand from legacy Puppeteer users. It is not the primary path.

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| Implement CDP as the primary protocol (Lightpanda approach) | CDP is unstable, Chromium-version-coupled, never designed for automation. Carrying that legacy contradicts Lumen's no-legacy advantage. |
| Implement WebDriver Classic | Dying protocol; HTTP request-response architecture is slow; superseded by BiDi industry-wide by 2026. |
| Only MCP, no BiDi | Cuts off the entire Playwright/Selenium/Cypress ecosystem. BiDi is the cross-browser standard with vendor commitment. |
| Only BiDi, no MCP | Cuts off the fastest-growing user segment (AI agents). MCP is the lingua franca of AI tooling. |
| Synthetic JS events (dispatchEvent) for input | Distinguishable from real input by anti-bot systems and by sites relying on trusted-event flags. Causes flaky tests. Native injection is structurally cleaner. |
| Make automation a separate crate that calls public engine APIs | Forces engine to expose every internal as `pub`. Trait inside the engine keeps the surface narrow and the implementation flexible. |
| DOM-selector-first locators (Selenium-style) | Locator brittleness is the #1 maintenance cost in industry; a11y-first locators survive DOM refactors and double as a11y testing. |
| Auto-wait in the client SDK (Playwright-style) | Engine knows what's actionable; client retry-loop is a workaround for not-knowing. Lumen can know. |

## Consequences

- **Positive:**
  - In-process tests for `graphic_tests/` become millisecond-fast and numerical (assert on `layout_box.width`, not pixel diff). Locality of failure improves dramatically — broken margin parsing reports as "expected 16.0, got 0.0", not "18% pixel diff somewhere".
  - Snapshot tests run without ffmpeg/gdigrab/Edge, removing Windows-only test infrastructure and making CI work on Linux/macOS uniformly. Calibration `TEST-00` (magenta offset detection) becomes unnecessary because no title bar exists in in-process screenshots.
  - AI agents are first-class clients via MCP without per-agent custom integrations.
  - Anti-bot artifacts (`navigator.webdriver`, CDP side-channels) do not exist by construction — there is no debug protocol bolted onto the JS runtime.
  - The engine can be embedded as a library: `cargo add lumen-driver` and any Rust app drives a real renderer.
  - A11y tree becomes a tested, exercised structure (used by tests, not only by AT bridges), which means the engine's accessibility behaves correctly rather than passes spec-only.

- **Negative / trade-offs:**
  - Three transport adapters (in-process / MCP / BiDi) to maintain. Mitigated by all of them being thin shims over the single trait.
  - Native input injection requires shell-layer plumbing on each OS (Windows/Linux/macOS). Manageable through winit, which already abstracts input.
  - Cross-OS pixel-identical snapshots require a software rasterizer for tests (`tiny-skia` or own CPU backend) under `cfg(test)`. Otherwise GPU drivers give ±1 LSB differences. Tiny-skia is pure Rust, ~30 KLoC, acceptable as a test-only dep.
  - Skipping CDP closes off the existing Puppeteer ecosystem until/unless a `lumen-cdp-shim` is built. BiDi adoption by Playwright/Puppeteer (in progress 2026) softens this over time.

- **Future:**
  - **Graduation triggers for CDP shim:** real, named demand from at least one major Puppeteer-using project that cannot move to BiDi. Until then, no CDP code in Lumen.
  - **MCP spec versioning:** the protocol is young (Nov 2024). Lumen pins one MCP version per Lumen release; adapter is one file.
  - **BiDi spec is a moving target (W3C Working Draft, May 2026).** Lumen tracks the spec; gaps are documented in `subsystems/lumen-bidi-server.md`.
  - **Android automation maps onto the same trait** without surface changes — transports differ (TCP socket via `adb forward`, Android Intent, JNI bridge), but `BrowserSession` is the same. This makes the trait-first approach a structural prerequisite for future mobile support, even though mobile is not in current scope.

## Performance gate (binding)

Automation API code (`lumen-driver`, `lumen-mcp-server`, `lumen-bidi-server`, `lumen-cdp-shim` when it appears) **must not regress `lumen-bench` median by more than 5%** vs the recorded baseline in a **default build without `--mcp` / `--bidi-port` / `--cdp-port` flags**. Default build means: end-user browser, no automation transport activated.

If a PR causes regression:

1. Re-architect to move cost behind a runtime flag (transport not started → zero cost path), or
2. Move the offending code behind a `cargo` feature flag with `default = false`, or
3. Document the regression in the PR body with explicit architectural justification and get reviewer sign-off.

Baseline numbers (as of plan §16 Phase 1): cold start ≤ 300 ms, RAM ≤ 100 MB on `samples/page.html`. CI gate (task 9G.3 in Roadmap) runs `cargo run -p lumen-bench --release`, parses min/median/p95 across **both time and RAM axes** (RAM axis added by ADR-008 task 9G.5), compares against committed `bench/baseline.json`, fails the PR on >5% regression of either axis.

This gate exists because automation is a cross-cutting concern that can quietly seep into hot paths. The trait is meant to be free when nobody calls it — the gate verifies that intent in CI, not in code review.
