# Журнал user-centric перф-метрик Lumen (PERF-2)

Хронология прогонов [`scripts/perf_metrics.py`](../../scripts/perf_metrics.py) по
корпусу реальных сайтов ([corpus.txt](corpus.txt)). Харнесс извлекает метрики
«по образцу» FCP / LCP / TTI + Total Blocking Time из таймлайна одной навигации,
который собирает `lumen --trace-nav` (PERF-1, Chrome Trace Event Format) —
никакого нового кода в движке, только раскладка готовых спанов в метрики.

Модель — [docs/build-speed.md](../build-speed.md): фиксированный протокол замера
+ журнал «метрика ↔ коммит». Отличие от [journal.md](journal.md) (perf_audit.py):
там пофазный аудит целиком (сеть/стиль/paint + RAM/зависания на живом окне),
здесь — узкий набор пользовательских метрик с регрессионным сравнением по каждой.

## Метрики (все в мс от старта навигации)

| Метрика | Из чего | Смысл |
|---|---|---|
| `ttfb_ms` | конец спана `fetch-document` | документ получен целиком (прокси TTFB — сеть не отдаёт под-байтовый тайминг) |
| `html_parse_ms` | длительность `parse-html` | парсинг HTML |
| `script_ms` | длительность `run-scripts` | исполнение JS на UI-потоке |
| `layout_ms` | длительность `layout` | каскад + бокс-дерево |
| `paint_ms` | длительность `paint` | CPU-растеризация (известный тормоз тяжёлых страниц) |
| `fcp_ms` | инстант `first-paint` | прокси First Contentful Paint |
| `lcp_ms` | конец спана `paint` | прокси Largest Contentful Paint |
| `tti_ms` | max(fcp, конец `run-scripts`) | прокси Time To Interactive |
| `tbt_ms` | Σ max(0, task − 50) по script-задачам | прокси Total Blocking Time (порог 50 мс как в Lighthouse) |
| `nav_ms` | длительность корневого `navigation` | полная загрузка |
| `doc_bytes` / `total_bytes` / `resources` | arg `size` net-спанов | детерминированные счётчики (сигнал регрессии, не зависит от сети) |

**Ограничение headless-пути.** `--trace-nav` красит страницу ОДИН раз в конце
(тот же CPU-путь, что `--screenshot`), поэтому `fcp`/`lcp`/`tti` схлопываются к
моменту завершения единственного paint — распределить их как в живом
инкрементальном рендере headless-путь не может (то же, что зафиксировано в
[journal.md](journal.md): headless paint ≠ живое окно). Реально различимые и
действенные числа этого пути — `ttfb` / `html_parse` / `script`(`tbt`) /
`layout` / `paint` и байтовые счётчики; `fcp`/`lcp`/`tti` даются «по образцу» для
полноты и как якорь общего времени. Живые FCP/LCP/TTI появятся, когда
PerformanceObserver paint-timing будет дописан в движке (JS-каркас частично
готов: `performance.now`, long-animation-frames, soft-navigation в `crates/js`).

## Протокол

```bash
# базлайн/регрессионный прогон (медиана из 3 устойчивее к сетевому шуму)
python scripts/perf_metrics.py --repeat 3 --timeout 60
# сравнение с прошлым коммитом (та же машина!)
python scripts/perf_metrics.py --compare docs/perf/metrics-runs/<дата>.json
# проверка извлечения метрик без сети (входит в ворота)
python scripts/perf_metrics.py --selftest
```

Сравнивать числа можно только между прогонами **одной машины**; ±20% на сетевых
фазах — шум, >20% по `nav_ms` на той же машине — находка. Сырые прогоны —
`docs/perf/metrics-runs/<дата>.json` (коммитятся; промежуточные трейсы остаются
в `.tmp/perf-metrics/`, не коммитятся).

Шаблон секции:

```markdown
## YYYY-MM-DD — <коммит> — <машина>

<таблица из summary.md>

**Сравнение с прошлым:** <дельты или «первый прогон»>
**Находки:** <список>
```
