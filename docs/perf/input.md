# Журнал латентности ввода и плавности кадра Lumen (PERF-3)

Хронология прогонов [`scripts/input_perf.py`](../../scripts/input_perf.py) —
две вещи, которые пользователь чувствует напрямую: **латентность ввода**
(клик/клавиша/скролл → видимый кадр) и **плавность кадра как распределение**
(p50/p95/p99/max + счётчик дропнутых кадров), а не среднее.

Тулинг поверх уже существующих поверхностей — `LUMEN_FRAME_LOG=1` + MCP
`scroll`/`click`/`type` в живом окне; **нового кода в движке нет** (дорожка PERF:
тулинг — P2, движковые правки по находкам — P1). Отличие от
[`scroll_perf.py`](../../scripts/scroll_perf.py): тот покрывает только скролл и
печатает avg/median/max; здесь — три вида ввода, перцентили, счётчик дропов и
per-event латентность. Отличие от [metrics.md](metrics.md) (PERF-2): там метрики
загрузки страницы из headless-трейса, здесь — интерактивность живого окна.

## Метрики (все в мс)

| Семейство | Из чего | Смысл |
|---|---|---|
| `frame_{p50,p95,p99,max}_ms` | число в строке `[frame] total XX.XXms` (UI-тик: layout+paint+swap) | плавность; пользователь замечает хвост (p95/p99), не среднее |
| `frame_dropped` / `frame_dropped_pct` | кадры длиннее бюджета (`--budget-ms`, по умолчанию 16.67 = 60 Гц) | сколько кадров дали рывок |
| `ack_{p50,p95,p99,max}_ms` | round-trip MCP-вызова | стоимость диспатча события на UI-потоке; плотный сигнал (одно значение на событие), есть всегда |
| `latency_{p50,p95,p99,max}_ms` + `latency_n` | от отправки события до первой кадровой строки после него | input→repaint конец-в-конце; плотно для scroll, разрежённо (`latency_n` мал) для click/type |

**Почему число из строки, а не интервал прибытия.** `[frame] total 8.84ms` уже
содержит время кадра, измеренное движком. Интервал между прибытием двух строк в
stderr задан паузой между событиями (`--gap-ms`) и как «время кадра»
бессмыслен — харнесс парсит именно зарепортенное число, а настенную метку
использует только для сопоставления события с его кадром (латентность).

**Ограничение чисто-тулингового пути.** `latency_*` меряется до ПЕРВОЙ кадровой
строки после события; включает задержку flush stderr-пайпа (мал, `eprintln`
флашит по `\n`) и **не** разбивает hit-test / dispatch / paint по отдельности —
для этого нужны спаны `input-dispatch`/`hit-test` в движке (дело P1). Дискретный
ввод (click/type) в живом окне **коалесит перерисовки**: 50 кликов дают ~2 кадра,
поэтому `latency_n` для них мал, а надёжный сигнал латентности для них — `ack_*`.
Скролл перерисовывается на каждый щелчок, поэтому `latency_*`/`frame_*` для него
плотные и репрезентативные.

## Протокол

```bash
# базлайн (fixture, все три workload); запись коммитируемого прогона
python scripts/input_perf.py --json docs/perf/input-runs/<дата>.json
# только скролл на тяжёлой реальной странице
python scripts/input_perf.py --page graphic_tests/1000000-final.html --workloads scroll
# сравнение с прошлым прогоном (та же машина!) — ⚠ при росте хвоста > 20%
python scripts/input_perf.py --compare docs/perf/input-runs/<дата>.json
# проверка статистики без браузера (входит в ворота)
python scripts/input_perf.py --selftest
```

Сравнивать можно только прогоны **одной машины**; ±20% на хвостовых метриках —
шум, >20% по `frame_p95`/`frame_p99`/`ack_p95`/`latency_p95` — находка. Сырые
прогоны — `docs/perf/input-runs/<дата>.json` (коммитятся); покадровый stderr
остаётся в `.tmp/input-perf/`, не коммитится. Требуется живое окно (winit) —
в headless-средах без дисплея используйте `--selftest`.

**Сравнение с прошлым:** первый прогон журнала (базлайн интерактивности).

**Находки:**
- **Скролл плавный на fixture**: frame p95 = 3.0 мс, p99 = 7.7 мс, max = 10.4 мс,
  0 дропнутых кадров при бюджете 16.67 мс; input→repaint p95 = 4.3 мс. GPU-путь
  скролла (off-main-thread) держит хвост под бюджетом. Тяжёлые реальные страницы
  (`--page graphic_tests/1000000-final.html`) — отдельный прогон, здесь fixture.
- **Диспатч ввода дёшев**: ack p95 = 1.9–3.7 мс на все три вида — сам путь
  hit-test + dispatch на UI-потоке не узкое место.
- **[Находка для P1] дискретный ввод коалесит перерисовки**: 50 кликов/нажатий
  дают лишь ~2 кадра `[frame] total` (dl 110→111) — программно-диспатченные
  click/type в живом MCP-окне почти не вызывают repaint. Значит: (а) `latency_*`
  для click/type даётся по 1–2 событиям (`latency_n` мал), надёжный сигнал их
  латентности — `ack_*`; (б) настоящую end-to-end input→repaint латентность
  дискретного ввода можно будет мерить, когда движок начнёт перерисовывать на
  каждое событие или отдавать спаны `input-dispatch` в трейсе. Кандидат в
  P1-задачу дорожки PERF; как самостоятельный баг пока не заведён (нужна
  доп-диагностика: событие не помечает shell dirty vs. перерисовки дедуплятся).
