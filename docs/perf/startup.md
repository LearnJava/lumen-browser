# Журнал холодного/тёплого старта Lumen (PERF-4)

Хронология прогонов бенча запуска — `cargo run -p lumen-bench --release -- --startup`
([`crates/bench/src/startup.rs`](../../crates/bench/src/startup.rs)). Меряет то,
что пользователь чувствует при **каждом** запуске браузера: время от старта
процесса `lumen` до готовности первой страницы. Дёшево мерить, легко регрессирует
незаметно (растёт число инициализаций на старте).

## Что и как меряется

Бенч порождает уже собранный shell-бинарь в headless-режиме
`--screenshot <tmp> <url>` — тот же путь, что доводит страницу до первого
отрендеренного кадра (полный старт движка: загрузка бинаря/DLL загрузчиком ОС,
инициализация V8-платформы и изолята, установка DOM-шима, парсинг+layout+CPU-paint
первой страницы) — и меряет настенное время каждого спавна.

В отличие от остальных бенчей крейта (они гоняют pipeline-фазы in-process), здесь
замер **процессный**: только так учитывается стоимость загрузчика ОС для ~70-МБ
бинаря (V8 статически слинкован) — заметная доля холодного старта, невидимая
in-process.

**Cold vs warm.** Первый спавн прогона — «холодный» (страничный кэш ОС для бинаря
и DLL ещё холодный), последующие — «тёплые». Настоящий холодный старт (сброс кэша
ОС) непортабелен, поэтому:

| Метрика | Из чего | Смысл |
|---|---|---|
| `cold_ms` | первый спавн прогона | старт с холодным кэшем загрузчика |
| `warm_{min,median,p95,max}_ms` | остальные спавны | стационарная стоимость старта |
| `cold_minus_warm_median_ms` | `cold − warm_median` | стоимость прогрева загрузчика/init |

Страница по умолчанию — маленькая локальная `samples/page.html`: изолирует старт
движка от сети и тяжёлого layout (сеть — забота PERF-2). `--url` меняет страницу,
`--iters N` — число спавнов, `--exe`/`$LUMEN_EXE` — путь к бинарю (иначе ищется
`target/{dev-release,release,debug}/lumen[.exe]`), `--json <файл>` — сырой прогон.

**Ограничение.** «Окно на экране» из формулировки задачи (GUI-вариант старта) не
меряется детерминированно в headless/CI. Процессный старт до первого кадра
(headless CPU) — измеримый прокси, и именно он несёт основную стоимость, которая
регрессирует (число инициализаций до первого кадра). GUI-разница (создание окна,
swapchain) — отдельный незамеряемый здесь довесок.

## Запуск

```bash
cargo build -p lumen-shell --profile dev-release      # бинарь, который будем стартовать
cargo run -p lumen-bench --release -- --startup       # 12 спавнов samples/page.html
cargo run -p lumen-bench --release -- --startup --iters 20 --url samples/heavy.html
cargo run -p lumen-bench --release -- --startup --json docs/perf/startup-runs/<date>.json
```

Сырые прогоны — `docs/perf/startup-runs/*.json` (коммитятся, как metrics-runs/input-runs).

## Как читать регрессию

Рост `warm_median_ms` при неизменной странице = подорожал старт движка (новая
инициализация на пути до первого кадра). Рост `cold_minus_warm_median_ms` =
разбух бинарь/зависимости (дольше грузит загрузчик ОС). Порог для гейта — задача
PERF-7 (ночной перф-гейт поверх метрик PERF-2/3/4/5, механизм в
[`crates/bench/src/ci_gate.rs`](../../crates/bench/src/ci_gate.rs)).
