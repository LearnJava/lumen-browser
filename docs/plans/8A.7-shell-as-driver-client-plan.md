## Key decisions

### Architecture

WinitSession delegates to InProcessSession for pipeline (parse/CSS/layout), 
adding layer for:
- URL loading (NetworkTransport)
- Window management (winit) — **Phase 2 onwards**
- GPU rendering (wgpu) — **Phase 2 onwards**
- Event handling — **Phase 2 onwards**

### Why this approach

1. **Code reuse:** InProcessSession already handles parse/CSS/layout pipeline
2. **Incremental:** Can implement one method at a time
3. **Safe:** Lumen remains functional during transition
4. **Testable:** WinitSession works headless for graphic_tests before winit integration

### Alternative: full rewrite

- Rewrite parse_and_layout from shell into lumen-driver module
- Create complete Lumen pipeline in lumen-driver
- Migrate shell to be pure UI layer on top of WinitSession

**Tradeoff:** More work upfront, cleaner separation of concerns. Recommended for Phase 2 onwards.

## Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| Large refactor breaks shell functionality | Test each phase independently; keep Lumen functional during Phase 1-3 |
| URL loading duplicates HttpClient setup | Extract HttpClient creation to lumen-driver helper; both Lumen and WinitSession use it |
| Screenshot requires headless renderer unavailable | Implement tinyskia CPU renderer (task 8A.5) in parallel |
| Event loop integration complex | Start with blocking sync API; async/await comes later in 8B |
