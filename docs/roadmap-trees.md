# Roadmap-дерево — как пользоваться

Визуализация роадмапа всего проекта в виде интерактивного дерева: фазы → задачи + баги
с указанием, какой баг какую задачу блокирует. Автоматическая синхронизация статусов
багов из `BUGS.md`.

## Что открыть

Три представления одних и тех же данных (переключение — навигация сверху каждой страницы):

| Файл | Вид |
|---|---|
| [`docs/roadmap-B-twotrees.html`](roadmap-B-twotrees.html) | **HTML-список.** Сворачиваемые `<details>`-деревья: слева фазы→задачи, справа баги. Наведение на задачу/баг подсвечивает связанную пару. Лучше всего для большого объёма (сотни узлов). |
| [`docs/roadmap-svg-columns.html`](roadmap-svg-columns.html) | **SVG два столбца.** Задачи-кружки ● слева, баги-ромбы ◆ справа, красные пунктирные стрелки «блокирует» между ними. |
| [`docs/roadmap-svg-gitgraph.html`](roadmap-svg-gitgraph.html) | **SVG git-граф.** Единое дерево с изогнутыми коннекторами как `git log --graph`: фаза → задачи → баги по дорожкам-уровням. |

Просто открой любой `.html` в браузере (двойной клик / `file://`). Никаких сборок и сервера —
данные вшиты в страницу, внешних зависимостей нет. SVG-виды удобнее с фильтром фазы (по умолчанию
`P2`); на «все» + показ FIXED они становятся очень длинными.

## Управление в странице

- **Поиск** — поле сверху, фильтрует по id и тексту (например `BUG-171`, `scroll`, `U-6`).
- **Скрыть FIXED-баги** — включено по умолчанию (иначе 161 закрытый баг зашумляет дерево).
- **Скрыть готовые задачи** — прячет поддеревья со статусом `done`.
- **Фаза** — селектор, по умолчанию `P2` (текущая фаза). «все» показывает целиком + секцию «Прочие баги».
- **Сворачивание** — клики по треугольнику ▸ (нативные `<details>`).
- **Легенда** — расшифровка цветов статусов задач и багов сверху.

## Как это устроено

```
ROADMAP.md   ─┐  (плоская курируемая структура: фазы, задачи, связи баг→задача — одна задача = одна строка)
              ├─►  scripts/gen_roadmap.py  ─►  вшивает данные в  docs/roadmap-*.html
BUGS.md      ─┘  (живой статус/заголовок/компонент каждого бага — парсится автоматически)
```

- **`ROADMAP.md`** (в корне репозитория) — единственное место, где курируется структура: две
  markdown-таблицы «Фазы» и «Задачи». Одна задача = одна строка → грепается как `BUGS.md`:
  `grep "| U-6 " ROADMAP.md`. Иерархия задаётся колонками `phase`/`parent` (плоско, без
  вложенности). Связи баг→задача — в колонке `bugs` (`BUG-104,BUG-126`). Статусы багов здесь
  НЕ хранятся — только их id.
- **`BUGS.md`** — источник живого статуса багов. Генератор читает таблицу и сам подставляет
  `OPEN/FIXED/...`, заголовок и компонент.
- **`scripts/gen_roadmap.py`** — сшивает оба источника, собирает дерево из плоских строк и
  вшивает итоговый JSON в блок `<script id="roadmap-data">` каждого HTML. Баги без ручной
  привязки автоматически попадают в группу «Прочие баги (по компоненту)».

## Как поддерживать в актуальном состоянии

После любого из изменений запусти из корня репозитория:

```bash
python scripts/gen_roadmap.py
```

| Что изменилось | Нужно ли править `ROADMAP.md` | Команда |
|---|---|---|
| Статус бага в `BUGS.md` (OPEN→FIXED и т.п.) | нет | `python scripts/gen_roadmap.py` |
| Добавлен новый баг в `BUGS.md` | нет (появится в «Прочие по компоненту») | `python scripts/gen_roadmap.py` |
| Хочешь привязать баг к задаче | да — впиши id в колонку `bugs` нужной строки | затем запусти скрипт |
| Новая задача / фаза / смена статуса задачи | да | затем запусти скрипт |

Скрипт печатает сводку и предупреждает, если в `ROADMAP.md` есть связь на баг, которого
нет в `BUGS.md`.

## Формат `ROADMAP.md`

Две markdown-таблицы. Шапки фиксированы (по ним скрипт находит таблицы), пустые ячейки допустимы.

**Таблица «Фазы»** — `id | status | date | title`:

```
| P2 | done | app v0.5.0 | Фаза 2 — v0.5 «Interactive» (завершена) |
```

**Таблица «Задачи»** — `id | phase | parent | status | size | bugs | note | title`:

```
| U-6 | P2 | P2-usability | ready |   | BUG-104,BUG-126 |   | U-6: рендер-паритет high-deviation |
```

- `phase` — id фазы из первой таблицы.
- `parent` — пусто → задача висит прямо под фазой; `<id>` → вкладывается в эту задачу (рекурсивно).
- `status` — производный: скрипт пересчитает его из багов/подзадач (см. ниже). Колонка нужна как
  запасной вариант для фич БЕЗ багов и БЕЗ подзадач.
- `size` — опц.: XS/S/M/L/XL. `bugs` — опц.: id через запятую. `note` — опц.: короткая пометка.

**Статусы задач:** `done` (готово) · `active` (в работе) · `blocker` (блокер) ·
`wait` (ждёт блокера) · `ready` (готово к старту) · `queued` (в очереди) ·
`planned` (план) · `opt` (опционально).

**Статусы багов** берутся из `BUGS.md` и не задаются здесь: `open` · `inprogress` ·
`fixed` · `wontfix`.

**Производный статус.** Скрипт пересчитывает `status` задачи: все связанные баги FIXED/DEBTOR →
`done`; есть баг IN PROGRESS → `active`; все подзадачи `done` → `done`. Ручную колонку править
нужно только у чисто-фичевых задач без багов и подзадач.

## Частые задачи

**Привязать баг к задаче** — впиши id в колонку `bugs` строки задачи в `ROADMAP.md`:

```
| U-5 | P2 | P2-usability | ready | S | BUG-167 |   | U-5: Fullscreen … |
```

**Добавить новую задачу в текущую фазу** — допиши строку в таблицу «Задачи» (`phase = P2`,
`parent` пуст или id родителя), запусти скрипт.

**Отметить задачу выполненной** — поставь `done` в колонке `status`, запусти скрипт
(в дереве она скроется при включённом фильтре «скрыть готовые задачи»).
