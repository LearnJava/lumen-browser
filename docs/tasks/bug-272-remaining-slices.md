## Срез 10 — `elem_image_id` (backdrop-filter): bbox-crop вместо full-framebuffer

Последний нетронутый кусок backdrop-filter из пункта 3(b) остатка: контент самого элемента
(`elem_image_id`, `apply_backdrop_filters`/`composite_backdrop_filter_layer`,
`crates/engine/paint/src/backends/femtovg_backend.rs`) всё ещё идёт через общий `layer_pool` —
full-framebuffer, в отличие от уже обрезанного в срезе 5 `filtered_backdrop_id`
(`backdrop_bbox_pool`). `layer_pool` сегодня несёт фиксированную full-frame + `FLIP_Y`-конвенцию
для render-target'ов; здесь нужен bbox-сайзинг для image, который реально GPU-сэмплится
(`Paint::image`), а не только рендер-таргет.

**Сделать:** расширить (или завести рядом) пул под bbox-сайзинг GPU-сэмплируемых образов элемента;
`elem_image_id` аллоцируется по `bounds`-прямоугольнику вместо `(fb_w, fb_h)`, композит мапится на
тот же прямоугольник (тот же паттерн device-px origin, что `filtered_backdrop_x/y` в срезе 5).

**Проверка:** `cargo test -p lumen-paint --features backend-femtovg`; графический тест 103
(backdrop-filter) — без регрессии; A/B gdigrab branch-vs-main при недоступности TEST-00-калибровки
(см. методику среза 5).

## Срез 11 — `PushOpacity`: bbox-сайзинг видимого слоя

Пункт 3(c) остатка, первая часть. Срез 7 куллит off-viewport opacity-группы целиком, но
**видимый** (on-viewport) слой всё ещё аллоцируется full-framebuffer через `layer_pool`. Ввести
bbox-aware вариант `acquire_layer` (или новый метод рядом), который аллоцирует слой размером с
`bounds`, если оно `Some`, и full-frame — если `None` (полностраничный view-transition fade без
bbox, см. срез 7). Это базовый механизм, который срезы 12–14 переиспользуют без изменений.

**Проверка:** `cargo test -p lumen-paint --lib`; графические тесты с opacity-группами — без
регрессии; синтетика (страница с несколькими вложенными `opacity<1` блоками) — снижение пикового
GPU по аналогии с методикой срезов 1/5.

## Срез 15 — `PushMaskLayer` (SVG `<mask>` content): viewport-cull

Отдельная семантика — `PushMaskLayer` применяется не к своей собственной скобке, а к
**родительскому** слою (в отличие от S9's `PushMask{Image,Gradient}`, которые самодостаточны).
Механизм куллинга срезов 7–9 напрямую не переносится — нужно сперва понять, как
`overlay_partition::layer_delta` балансирует эту скобку и что значит «off-viewport» для контента,
влияющего на родителя. Считать исследовательским срезом: если по итогу анализа безопасный
cull невозможен без более глубокой переработки — задокументировать вывод в `bugs/BUG-272-OPEN.md`
и явно оставить пункт открытым, это не блокирует переход к S16.

## Срез 16 — GPU-счётчик по категориям + расследование glyph atlas (~285 МБ)

Пункт 1 остатка. lenta.ru держит ~509 МБ GPU против ~224 МБ baseline пустого окна — разница
~285 МБ считается страничной, glyph atlas — не подтверждённое предположение. Нужен инструмент:
расширить `LUMEN_MEM_REPORT`/`debug_mem_report()` GPU-счётчиком по категориям текстур (glyph atlas,
framebuffer/swapchain, layer_pool, cpu_upload_pool, decoded images) — не просто суммарный GPU-байт
процесса, как сегодня. Итог среза — не обязательно фикс, а подтверждённая атрибуция (либо
опровержение) glyph-atlas-гипотезы; также зафиксировать (без обязательного фикса) находку по
базовому 224 МБ пустого окна (framebuffer/шрифтовой атлас/драйвер).

**Проверка:** новый счётчик показывает разумные числа на `samples/page.html` (известный маленький
baseline) и на синтетике с большим количеством текста/шрифтов.

## Срез 17 — Multi-copy image cache: `raw_images` deep-copy (femtovg)

пикселей помимо `DecodedImageCache` — дедуплицировать через `Arc`, тем же приёмом, что срез 6

**Проверка:** `cargo test -p lumen-paint`; визуальная приёмка на странице с несколькими `<img>` —
без регрессии рендера.

## Срез 18 — Multi-copy image cache: `@WxH`-варианты

Пункт 4 остатка, часть 2. Ресайз-варианты декодированного изображения (разные CSS-размеры одного
исходника) сегодня хранятся как независимые полные копии — оценить дедупликацию (кэш по
`(source, target_w, target_h)` с переиспользованием исходных пикселей там, где это не требует
повторного ресемплинга) либо LRU-эвикцию, если дедупликация невозможна без потери качества.

**Проверка:** `cargo test -p lumen-paint`; страница с одним исходным изображением в нескольких
размерах — пиксельно без регрессии.

## Срез 19 — Multi-copy image cache: GIF все кадры

Пункт 4 остатка, часть 3. Анимированный GIF сегодня держит в памяти все декодированные кадры
одновременно вне зависимости от того, сколько реально нужно для плавного воспроизведения — оценить
ленивое декодирование окна кадров вместо полного eager-декода при загрузке.

**Проверка:** `cargo test -p lumen-paint`; графический тест на анимированный GIF (если есть в
`graphic_tests/`, иначе — ручная синтетика) — без регрессии воспроизведения.

## Срез 20 — Multi-copy image cache: canvas2d thread-local

Пункт 4 остатка, часть 4, последний. Canvas2D-путь держит thread-local буферы отдельно от общих
кэшей изображений — оценить, дублируется ли с `DecodedImageCache`/`raw_images`, и свести к общему
хранилищу там, где это безопасно (canvas-контент мутабелен кадр-к-кадру, поэтому дедупликация
скорее всего ограничена *snapshot*-путём, не live-буфером — проверить перед изменением).

**Проверка:** `cargo test -p lumen-paint`, `cargo test -p lumen-js -- canvas`; графические тесты на
Canvas 2D (см. `project_canvas2d_phases` в памяти / `graphic_tests/COVERAGE.md`) — без регрессии.
