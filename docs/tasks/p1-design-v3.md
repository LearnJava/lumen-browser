# P1-DS: Внедрение дизайн-системы v3.3 в хром Lumen

**Владелец: P1.** Дорожка в ROADMAP.md: `DS` (подзадачи `DS-1`…`DS-19`).
**Эталон:** [docs/design/lumen-v3_3.html](../design/lumen-v3_3.html) — интерактивный HTML-прототип дизайн-системы. Это **единственный источник визуальной истины** (токены, состояния, поведение). Код прототипа в продукт **не переносится ни строчкой** — реальный Lumen рисует хром display-list'ом через wgpu/femtovg без нативных виджетов.

## Аудит разрыва (2026-07-22, сверка эталона с кодом)

Уже есть и переиспользуется (НЕ переписывать): кастомный рендер хрома (display-list → wgpu, femtovg fallback), таб-бар со всеми состояниями (группы, контейнеры, tree-tabs, вертикальный layout, гибернация, спиннер), омнибокс с `@`-префиксами, ~30 панелей (история/закладки/read-later/настройки/загрузки/Ctrl+K-палитра/find/сертификат/разрешения/фокус-помодоро/AI/web-сайдбар/DevTools), темы light/dark/system + `Palette` с ролевыми токенами, SQLite-бэкенды почти под всё (включая `profiles.rs` + vault), AX-дерево веб-страницы + реальный Windows-MSAA-мост.

Отсутствует (это и есть дорожка DS): токены как система + генератор из эталона, шрифты JetBrains Mono / Golos Text, омоглиф-защита омнибокса, постоянный тулбар с инлайн-омнибоксом, 8 настраиваемых плиток newtab, прокраска иерархии контекстов (workspace-цвет, вертикальная контейнер-полоса), UI профилей + визуальные сигнатуры (красная обводка Anonymous, монохром Guest) + эфемерный режим, a11y собственного хрома, статистика щитов, загрузки-поповер.

## Правила для каждого среза (обязательны)

1. Один срез = одна сессия = одна ветка `p1-ds-<N>-<slug>` в своём worktree `.claude/worktrees/<task>/`. Завершение — `/lumen-task-finish` (7 шагов, merge --no-ff, push).
2. Перед стартом: `git pull origin main`, проверить `git branch` на уже занятые `p1-ds-*`.
3. Цвета/радиусы/отступы — **только** из `theme_tokens.rs` (после DS-1). Радиусы строго 2/4/6 px, сетка 4 px, **никаких пилюль** (радиус >6 в хроме запрещён).
4. **Два текст-движка**: живое окно рисует текст через femtovg, CPU/wgpu-headless — через lumen-font. Любая правка шрифтов/текста хрома проверяется в обоих путях (см. память проекта и `docs/automation.md`).
5. Doc-sync в том же коммите: `CAPABILITIES.md` (секция lumen-shell), `subsystems/shell.md`, при закрытии среза — вычеркнуть строку-указатель из `STATUS-P1.md`.
6. Cargo: только `-p lumen-shell` / `-p lumen-core` и т.д., clippy `-D warnings` перед коммитом, `///`-доки на все pub-элементы.
7. Если срез меняет пиксели **веб-контента** (не хрома) — регенерировать CPU-снапшоты. Срезы DS меняют только хром, снапшоты затрагивает лишь DS-9 (сдвиг вьюпорта) — там отдельный чек-лист.
8. Эталон открывать в обычном браузере для сверки; в коде ссылаться на номера строк эталона (они стабильны — файл заморожен, правки дизайна = новая версия файла).

## Этап B — безопасность и честность UI

### DS-5 (S): Детектор омоглифов/mixed-script в `lumen-core::idn`

**Цель:** чистая логика (без UI): решить, безопасно ли показывать Unicode-хост, или показывать punycode.

**Файлы:** [crates/core/src/idn.rs](../../crates/core/src/idn.rs) — сейчас только ToASCII (`domain_to_ascii`), шапка файла честно говорит «без UTS-46/bidi/confusable».

**Шаги:** (1) `pub enum HostDisplay { Unicode(String), Punycode { ascii: String, reason: SpoofReason } }` + `pub fn display_host(host: &str) -> HostDisplay`; (2) правила (детерминированные, Chromium-lite): (а) метка смешивает латиницу с кириллицей/греческим → spoof; (б) метка целиком из кириллических символов, каждый из которых имеет латинский омоглиф (таблица: а е о р с х у і ѕ ј ԁ ɡ и т.п. — завести const-таблицу с комментом-источником), и TLD — латинский (`.com`/`.net`…) → spoof; (в) чистая кириллица с кириллическим TLD (`.рф`) — НЕ spoof; (3) punycode-кодирование уже есть (`domain_to_ascii`) — переиспользовать; (4) юнит-тесты: `аpple.com` (кир. «а») → Punycode `xn--pple-43d.com`; `google.com` → Unicode; `яндекс.рф` → Unicode; `раураl.com` → Punycode; пустые/ASCII-метки → Unicode as-is.

**DoD:** `cargo test -p lumen-core` зелёный; никаких новых зависимостей; `///`-доки с описанием правил.

### DS-15 (S): Визуальные сигнатуры Anonymous и Guest

**Цель:** правило вложенных рам, уровень 0: Anonymous → тонкая красная обводка окна; Guest → полностью монохромный UI. Эталон: строки 165–167, 184.

**Шаги:** (1) Anonymous: в конец display-list кадра — 4 `FillRect` по периметру контент-области, 2 px, `profile::ANONYMOUS` (поверх всего, аналог inset box-shadow); (2) Guest: `fn desaturate(c: Color) -> Color` (luma-формула) применяется ко ВСЕМ полям `Palette` при построении, accent := `profile::GUEST`; (3) не трогать рендер веб-контента — сигнатуры только на хроме (обводка — по краю окна, поверх контента, как в эталоне).

### DS-16 (M): ADR «Профили как контексты безопасности» + эфемерный срез Anonymous

**Цель:** зафиксировать архитектуру изоляции профилей и сделать первый реальный срез эфемерности.

**Файлы:** новый `docs/decisions/ADR-0NN-profile-security-contexts.md` (номер — следующий свободный, по TEMPLATE.md, обновить индекс README.md); `main.rs` (запись истории при навигации — найти вызов записи в History store); cookie jar (crates/storage — у stores есть `open_in_memory`-паттерн, см. `ProfileRegistry::open_in_memory` :66); [crates/shell/src/panels/history_panel.rs](../../crates/shell/src/panels/history_panel.rs).

**Шаги:** (1) ADR: уровни изоляции (0=профиль: отдельные data-каталоги/vault; эфемерный Anonymous: всё in-memory; Guest: in-memory + монохром), связь с ADR-012 (нет DB-менеджера — полная пер-профильная изоляция ~36 store'ов поэтапно), что делает ЭТОТ срез, что отложено; (2) срез: при активном Anonymous — история НЕ пишется, cookie jar открывается in-memory, downloads-история не пишется; (3) баннер в history_panel: «История не сохраняется — Анонимный профиль» (эталон `.hist-banner` :415, 1108); (4) переключение НА Anonymous и обратно не должно терять обычный профиль (in-memory экземпляры живут только пока профиль активен).

**DoD:** сценарий вручную: Anonymous → походить по страницам → переключиться на Личный → истории этих страниц нет, куки не сохранились; ADR в индексе; тесты зелёные.

## Что сознательно НЕ входит в дорожку (не делать без отдельного решения)

- Миграция панелей на `SurfaceManager` (ADR-009) — отдельная инфраструктурная дорожка; DS-срезы работают в текущей ad-hoc-модели.
- P2P/WebDAV-синхронизация (секция Sync эталона) — отдельная будущая дорожка, требует своего ADR.
- Полная пер-профильная изоляция всех ~36 SQLite-store'ов — этапы в ADR из DS-16.
- UIA-провайдер Windows / macOS / Linux a11y-мосты — за пределами DS-17.
- Ленты новостей на newtab, облачная синхронизация, пилюли-кнопки — **запрещены дизайн-системой**, не добавлять никогда.
