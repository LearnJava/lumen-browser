# Каскад CSS ~1с на тяжёлых сайтах — три задачи, план для следующей сессии

> **Точка входа.** Читать этот файл ПЕРВЫМ, до кода, если задача — «скролл
> подвисает» / «каскад медленный» / «BUG-284/BUG-286 backlog». Написан
> 2026-07-15 по итогам расследования [BUG-284](../../bugs/BUG-284-FIXED.md) +
> [BUG-286](../../bugs/BUG-286-OPEN.md). Перепроверяй факты против кода перед
> стартом — не факт, что за это время что-то не поменялось.

## TL;DR — что уже точно известно

- На тяжёлых реальных сайтах (ria.ru: 2490 DOM-узлов, 3093 CSS-правила)
  пересчёт оформления всей страницы (`compute_style` для каждого узла по
  всем CSS-правилам) занимает **~1.0–1.1с** (было ~2.4с до BUG-284).
- Этот пересчёт синхронно блокирует UI-поток на дефолтной сборке. Триггерится:
  сразу после загрузки страницы, и повторно — когда при скролле в зону
  видимости впервые попадает блок с `content-visibility:auto` (BUG-286).
- **Три независимых способа этого не чувствовать**, ниже по приоритету
  (дешевизна/риск → полезность). Не делай их «на автомате» одну за другой —
  каждая требует замера ДО и ПОСЛЕ, и №2/№3 — это отдельные архитектурные
  решения, не мелкие правки.
- **Профилировщик уже подключён и работает — не нужно ничего доустанавливать
  для старта.** `LUMEN_PROFILE_TREE=1` (без сборки с фичами, без внешних
  программ) уже даёт разбивку по фазам `layout_measured_hyp` →
  `precompute_counters` / `build_box` / `lay_out` / `post_layout_passes`
  (`crates/engine/layout/src/box_tree.rs:2585` и рядом, `lumen_core::profile`,
  `crates/core/src/profile.rs`). `tracy` (визуальный таймлайн) тоже подключён
  (`cargo run -p lumen-shell --features tracy`), но требует отдельно скачать
  и запустить GUI-приложение Tracy — используй только если `LUMEN_PROFILE_TREE`
  не даёт достаточной детализации.

## Задача 1 (делать первой) — найти и ускорить настоящий узкий участок `compute_style`

**Статус: ВЛИТА 2026-07-15** (кэш `active_media`/`active_supports` в
`CascadeIndex` + ленивый `ComputedStyle::clone()` для CSS revert,
`crates/engine/layout/src/style.rs`). Замер: `layout_measured_hyp` 982.47мс →
~720-730мс (~26%), выше порога приёмки 15-20%. `graphic_tests/run.py` на
Windows подтвердил отсутствие НОВЫХ визуальных регрессий от этой ветки (все
расхождения — уже существующий на main дрейф, см. [BUG-287](../../bugs/BUG-287-OPEN.md)).

**Зачем первой:** дешевле и безопаснее двух других — меняется только код
внутри пересчёта стилей, ничего не переключает архитектурно, риск регрессий
локален. Даёт информацию, нужны ли вообще задачи 2/3.

**Статус на 2026-07-15:** одна гипотеза уже проверена и ОПРОВЕРГНУТА —
замена `BTreeSet` на `Vec`+`sort_unstable`+`dedup` в `RuleIndex::candidates`
не дала измеримого ускорения (три замера в пределах шума среды), правка
отброшена, не влита. Причина неудачи: искали вслепую (`eprintln!`-таймеры
по крупным фазам), а не через сэмплирующий профайлер по строчкам/функциям.

**Как делать:**

1. `LUMEN_PROFILE_TREE=1 cargo run --release -p lumen-shell -- https://ria.ru`
   (или любой другой тяжёлый реальный сайт с адаптивной вёрсткой на
   `@media`) — снять дерево времён по существующим 5 фазам. Подтвердить,
   что `precompute_counters`/`build_box` всё ещё доминируют (ожидание по
   BUG-284: да, это единственные проходы, реально считающие каскад).
2. Добавить **более мелкие** `lumen_core::profile::scope(...)` внутри
   `compute_style` (`crates/engine/layout/src/style.rs`) — как минимум вокруг:
   `matches_complex` (сопоставление селектора с узлом), сборки/поиска
   кандидатов через `RuleIndex`/`CascadeIndex`, применения деклараций
   (`apply_declaration` — там же, где парсится каждое CSS-значение),
   наследования (`inherited`-цепочка). Прогнать снова на том же сайте —
   `LUMEN_PROFILE_TREE` сам напечатает вложенное дерево, дополнительных
   инструментов не нужно.
3. Если разбивка по scope недостаточно точна (например, всё «размазано» и
   не видно явного лидера) — подключить `tracy` (`cargo run -p lumen-shell
   --features tracy -- https://ria.ru`, СНАЧАЛА скачать и запустить
   [Tracy GUI](https://github.com/wolfpld/tracy) — клиент сам находит
   запущенный GUI). Даёт таймлайн по кадрам, а не только суммарные числа —
   видно, разброс это или стабильно дорогой один вызов.
4. Точечно ускорить то, что реально доминирует. Кандидаты (не проверены,
   не гарантированы — именно это и должен показать профиль):
   аллокации `Vec`/`HashMap` на узел внутри `compute_style` (не в
   `RuleIndex::candidates` — там уже проверили и не сработало), повторные
   вычисления `node_classes`/атрибутов селектора, клонирование
   `ComputedStyle` там, где хватило бы ссылки.
5. **Замер ДО/ПОСЛЕ обязателен, тем же способом (`LUMEN_PROFILE_TREE`),
   в один заход** — фон (конкурирующие cargo-сборки на машине) двигает
   числа на десятки процентов, см. `docs/tasks/linux-scroll-perf.md`
   «Железное правило замера».

**Приёмка:** `precompute_counters`+`build_box` заметно (не в пределах шума
— минимум 15-20%) короче на том же сайте/срезе; `cargo test -p lumen-layout`
зелёный; `graphic_tests/run.py --continue-on-fail` — «Изменений нет» относительно
эталонного прогона (визуальных регрессий нет).

## Задача 2 (отдельное решение, не мелкая правка) — `LUMEN_ENGINE_THREAD=1` по умолчанию

**Зачем:** пока каскад сам по себе не станет быстрым (задача 1), эта задача
не убирает секундную задержку — она убирает **мёртвую заморозку окна**: с
флагом пересчёт стилей уходит на отдельный движковый поток
(`crates/shell/src/engine_thread.rs`), UI-поток не блокируется, картинка
досчитывается на один из следующих кадров вместо повисания курсора/скролла.

**Почему не мелкая правка:** флаг `LUMEN_ENGINE_THREAD` сейчас проверяется
в десятках мест по всему `shell` (`main.rs`, ADR-016 M1–M4) — каждое
написано с инвариантом «поведение байт-в-байт одинаковое, флаг включён или
нет». Перевод в дефолт требует перепроверить это утверждение для ВСЕГО
браузера, не только скролла: формы, JS-таймеры, DevTools, навигацию,
find-in-page — везде, где есть relayout. Это отдельная задача с своим
приёмочным прогоном (весь `graphic_tests/run.py`, весь набор unit-тестов
затронутых крейтов), не довесок к задаче 1.

**С чего начать, если решили делать:** прочитать все упоминания ADR-016 в
`docs/decisions/` и `crates/shell/src/main.rs` (`grep -n "ADR-016"`) — там
уже описаны все M1–M4 срезы и что каждый гарантирует. Прогнать
`graphic_tests/run.py --continue-on-fail` дважды (флаг off/on) и сравнить
дельту, не только на скролл-страницах.

## Задача 3 (самая рискованная, делать только если 1 не хватит) — частичный пересчёт стилей

**Зачем:** сейчас при любом relayout (в том числе локальном — один блок
раскрылся по `content-visibility:auto`) пересчитывается оформление ВСЕГО
документа. Частичный пересчёт (только для нового поддерева) сократил бы
именно этот сценарий (BUG-286) до долей секунды.

**Почему рискованно:** CSS-каскад не всегда локален. Наследование
(`inherited`-свойства идут от родителя), счётчики (`content: counter(...)`
элемента зависит от `counter-reset`/`counter-increment` предыдущих узлов в
document order — из-за этого сейчас `precompute_counters` и `build_box` два
отдельных прохода, см. BUG-284 «Остаток»), некоторые селекторы вроде
`:nth-child`/сиблинг-комбинаторов зависят от соседних узлов. Наивный
«пересчитать только новое поддерево» может дать неверный результат в этих
случаях — нужен явный анализ, какие зависимости пересекают границу
поддерева, и это может оказаться архитектурной переделкой кэша каскада, не
локальной правкой.

**С чего начать, если решили делать:** НЕ начинать без задачи 1 (профиль
может показать, что после ускорения самого `compute_style` частичный
пересчёт вообще не нужен — секунда на весь документ и секунда на маленькое
поддерево это разные заказы величины экономии, посчитать заранее). Если всё
же нужно — читать `layout_mutation_incremental`
(`crates/engine/layout/src/incremental.rs`, уже частично пересчитывает
геометрию, не стили — отправная точка) и `precompute_counters`/`build_box`
контракт в BUG-284-FIXED.md перед любой правкой.

## Железное правило (как в linux-scroll-perf.md)

Ни одной строчки оптимизации без замера ДО тем же способом, которым будет
замер ПОСЛЕ, в один заход (фон машины двигает числа на десятки процентов).
Задача 1 (BTreeSet→Vec) уже один раз провалилась именно потому, что
угадывали узкое место, а не смотрели профиль по строчкам.
