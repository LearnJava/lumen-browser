# Программист 1 (P1) — инструкция для изолированной сессии

Самодостаточная памятка по роли **P1** для сессии, которая не видит контекст Claude Code.
Проект — **Lumen**, браузер на Rust с собственным движком. Корень: `D:\RustProjects\lumen-browser`.

> Остальные роли: P2 — резерв, P3 — баг-фиксы, P4 — CSS-свойства, P5 — здоровье кода.
> Каждая сессия работает в своём `git worktree`.

---

## Кто такой P1

P1 — **разработчик фич**. Берёт задачи из roadmap (любая подсистема: source → layout → paint → shell).

**Ограничения роли:**
- P1 **не чинит баги** (это P3) и **не реализует CSS-свойства** (это P4).
- Нашёл баг по ходу → добавь строку `OPEN` в `BUGS.md` со следующим номером `BUG-NNN` и **продолжай свою задачу**, не переключайся.
- Нужен новый layout/render-примитив с привязкой к CSS → P1 пишет алгоритм и ставит коммент `// CSS: <property>` в точке вызова + запись в `STATUS-P4.md` «Needs wiring». Само свойство в `ComputedStyle`/`apply_declaration()` добавляет P4.
- Писать код только внутри папки браузера и её worktree-копий в `.claude/worktrees/*`. Ничего снаружи не трогать.

## Шаг 1. Старт сессии

1. Прочитать `STATUS-P1.md` — голые строки-указатели `<источник>:NN` (для P1 источник = `ROADMAP.md`), одна на открытую задачу, приоритет сверху вниз.
2. `git branch` — убедиться, что ты на `main`.
3. Если в `STATUS-P1.md` есть «In progress» — продолжай эту задачу. Иначе бери **первый** указатель из списка.
4. Указатель `ROADMAP.md:NN` → открой строку (`sed -n 'NNp' ROADMAP.md`), по `<id>` найди детальный бриф `docs/tasks/<id>.md`.
5. Очередь P1 пуста → спроси пользователя, какую задачу брать.
6. Нужен контекст:
   - архитектура → `docs/plan/architecture.md` (§1 принципы, §3 граф зависимостей);
   - зависимости → `docs/plan/tech-stack.md` (§5 dependency policy);
   - «что браузер уже умеет» → **только** `CAPABILITIES.md`.

## Шаг 2. Резервирование задачи (worktree + ветка)

```bash
git worktree add .claude/worktrees/<task-name> -b p1-<task-name>
git push origin p1-<task-name>
```

- Имя ветки **обязательно** с префиксом `p1-` (идентификация владельца при краше).
- Worktree **только** внутри `.claude/worktrees/` — не `../lumen-<task>/`.
- В **первом коммите** на ветке передвинь указатель в `STATUS-P1.md` из «Next» в «In progress» с именем ветки и первым шагом.
- В worktree редактируй файлы **по пути worktree**, не по абсолютному пути main-root — иначе правка молча уйдёт в чужое дерево.

## Шаг 3. Работа над задачей

Поиск без чтения целых файлов:

```bash
grep "SymbolName" SYMBOLS.md      # индекс pub fn/struct/enum → file:line
grep "OPEN" BUGS.md               # открытые баги
```

Dump-режимы вместо чтения исходников:

```bash
cargo run -p lumen-shell -- --dump-layout samples/page.html
cargo run -p lumen-shell -- --dump-display-list samples/page.html
```

Быстрая проверка во время работы:

```bash
cargo check -p <crate>            # 1-2 сек, без линковки
```

**Правила кода:**
- `///` doc-комменты обязательны на всех pub struct/field/fn (что значение представляет, система координат / box-model / единицы).
- Никаких `panic!`/`unwrap()` в production-коде (только в тестах). Ошибки: `Result<T,E>` для API, `Option` для «не найдено».
- `unsafe` только на FFI-границах, с `// SAFETY:`.
- Тесты-первыми для парсеров и алгоритмов; интеграционные тесты на реальных данных обязательны.
- Перед длинной паузой — коммить wip (защита от краша).

**Коммиты:**
- Один логический шаг = один коммит. `git add path1 path2` (не `-A`).
- Сообщение **на русском**: короткий subject (<80 симв.), пустая строка, тело объясняет *почему*.
- Трейлер в конце:
  ```
  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  ```
- Прямые коммиты в `main` запрещены. `git push` — только по явной просьбе пользователя.

## Шаг 5. Завершение (7 обязательных шагов)

Проще всего вызвать скилл `/lumen-task-finish` — он сам прогоняет clippy+тесты и делает merge. Вручную:

```bash
# 1. Готовность кода
cargo clippy -p <crate> --all-targets -- -D warnings
cargo test -p <crate>

# 2. Сначала влить main в свою ветку, разрешить конфликты локально, потом:
git checkout main
git merge --no-ff p1-<task-name> -m "Merge p1-<task-name>: описание"

# 3. Удалить ветку
git branch -d p1-<task-name>

# 4. Обновить STATUS-P1.md (убрать указатель), выставить done в ROADMAP.md → gen_roadmap.py
git add STATUS-P1.md ROADMAP.md
git commit -m "P1: задача <id> завершена"

# 5. Push (по согласованию с пользователем)
git push origin main

# 6. Удалить worktree (КРИТИЧНО — иначе блокирует другие сессии)
git worktree remove .claude/worktrees/<task-name>
```

- Merge лучше делать в отдельном временном worktree, а не в общем корне (его может занять чужая ветка).
- Не оставлять worktree на `main` с незакоммиченными изменениями — это блокирует все остальные сессии.
